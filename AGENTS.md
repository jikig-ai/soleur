# Agent Instructions

This repository contains the Soleur Claude Code plugin. Detailed conventions live in `knowledge-base/project/constitution.md` -- read it when needed. This file contains only rules the agent will violate without being told on every turn.

## Hard Rules

- Never `git stash` in worktrees [id: hr-never-git-stash-in-worktrees] [hook-enforced: guardrails.sh guardrails:block-stash-in-worktrees]. Commit WIP first, then merge. Use `git show <commit>:<path>` to inspect old code without modifying working tree state.
- MCP tools (Playwright, etc.) resolve paths from the repo root, not the shell CWD [id: hr-mcp-tools-playwright-etc-resolve-paths]. Always pass absolute paths to MCP tools when in a worktree.
- When a command exits non-zero or prints a warning, investigate before proceeding [id: hr-when-a-command-exits-non-zero-or-prints]. Never treat a failed step as success.
- Always read a file before editing it [id: hr-always-read-a-file-before-editing-it]. The Edit tool rejects unread files, but context compaction erases prior reads -- re-read after any compaction event.
- When a plan specifies relative paths (e.g., `source "$SCRIPT_DIR/../../..."`), trace each `../` step to verify the final target before implementing [id: hr-when-a-plan-specifies-relative-paths-e-g]. Plans have prescribed wrong paths that were implemented verbatim and only caught by review agents.
- The host terminal is Warp [id: hr-the-host-terminal-is-warp]. Do not attempt automated terminal manipulation via escape sequences (cursor position queries, TUI rendering, and similar sequences are intercepted by Warp's tmux control mode and silently fail).
- The Bash tool runs in a non-interactive shell without `sudo` access [id: hr-the-bash-tool-runs-in-a-non-interactive]. Do not attempt commands requiring elevated privileges -- provide manual instructions instead.
- Exhaust all automated options before suggesting manual steps [id: hr-exhaust-all-automated-options-before]. Priority chain: (1) Doppler (`doppler secrets get <KEY> -p soleur -c dev --plain`, check `prd`/`ci`/`prd_terraform` configs too), (2) MCP tools (`ToolSearch`), (3) CLI tools (`gh`, `hcloud`, `supabase` — install with `curl`/`tar` to `~/.local/bin` if missing), (4) REST APIs (`curl`/`WebFetch`), (5) Playwright MCP (bootstraps credentials — retrieve token, store in Doppler, then switch to CLI/API), (6) manual handoff. Only prompt for credentials not found in any Doppler config.
- Never label any step as "manual" without first attempting automation [id: hr-never-label-any-step-as-manual-without]. Browser tasks → Playwright MCP first (only CAPTCHAs and OAuth consent are genuinely manual). Post-merge steps → verify each cannot be automated (`terraform apply`, `ssh`, `gh secret set`, `gh workflow run`). Plans that say "manual" when the tool is available are a workflow violation.
- When triaging a batch of issues, never defer an issue as "not automatable" without checking available skills (`/soleur:help`), MCP tools (`ToolSearch`), and CLI tools first [id: hr-when-triaging-a-batch-of-issues-never]. Content generation (images, copy, design) is automatable via `gemini-imagegen`, `frontend-design`, `copywriter`, `pencil` MCP, and similar tools. The check must happen during triage, not after -- an issue classified as "skip" at triage never gets re-evaluated. **Why:** In #2075, OG image generation was deferred as "requires image generation" despite `gemini-imagegen` being available. The snap judgment at triage time bypassed the automation-first rule.
- All infrastructure provisioning (servers, volumes, firewalls, DNS) goes through Terraform — never vendor APIs or manual SSH [id: hr-all-infrastructure-provisioning-servers]. SSH is read-only diagnosis only (`printenv`, `docker inspect`, `journalctl`); fixes go into Terraform config. Vendor APIs are permitted only for read-only checks or account-level tasks Terraform cannot cover. Patterns live in `apps/web-platform/infra/`.
- Every new Terraform root must include an R2 remote backend (`soleur-terraform-state` bucket, key `<app-name>/terraform.tfstate`) [id: hr-every-new-terraform-root-must-include-an]. Copy from `apps/web-platform/infra/main.tf`. Local state is never acceptable.
- In GitHub Actions `run:` blocks, never use heredocs or multi-line shell strings that drop below the YAML literal block's base indentation [id: hr-in-github-actions-run-blocks-never-use]. Heredoc bodies and terminators at column 0 break YAML parsing entirely (GitHub shows "workflow file issue", zero jobs run). Multi-line `--body`/`--comment` args spanning lines have the same problem. Use `{ echo "..."; } > file` for multi-line file content and shell variables with `$'\n'` for multi-line CLI args. **Why:** In #974, indented heredoc content rendered as `<pre>`. In #1358, left-aligned heredoc content and multi-line `--body` args broke the YAML parser, causing every push to trigger a failing "Main Health Monitor" run with zero jobs.
- GitHub Actions workflow notifications must use email via `.github/actions/notify-ops-email`, not Discord webhooks [id: hr-github-actions-workflow-notifications]. Discord is for community content only. For custom bodies, construct HTML in a preceding step and pass as the `body` input.
- New skills, agents, or user-facing capabilities must include CPO and CMO at minimum in brainstorm domain assessment [id: hr-new-skills-agents-or-user-facing] [skill-enforced: brainstorm Phase 0.5]. CTO is included when the capability has architectural implications.
- Before shipping, `/ship` Phase 5.5 runs conditional domain leader gates (CMO content-opportunity, CMO website framing, COO expense-tracking) [id: hr-before-shipping-ship-phase-5-5-runs] [skill-enforced: ship Phase 5.5]. These trigger on file-path matches, semver labels, and new service signups.
- When a workflow concludes with an actionable next step, execute it — don't list it as "next action" and stop [id: hr-when-a-workflow-concludes-with-an]. Use Playwright MCP, `xdg-open`, CLI tools, or APIs to drive completion. Only hand off for credentials/payment at the exact page.
- Before asserting GitHub issue status, verify via `gh issue view <N> --json state` and check `knowledge-base/` for existing artifacts [id: hr-before-asserting-github-issue-status]. Unverified status claims create false urgency.
- Never run commands with unbounded output in subagents — pipe through `| head -n 500` or `| tail -n 200` [id: hr-never-run-commands-with-unbounded-output]. Subagent stdout goes to tmpfs; unbounded output fills it and crashes all sessions.
- Never use `sleep` >= 2 seconds in foreground Bash calls — Claude Code blocks it [id: hr-never-use-sleep-2-seconds-in-foreground]. For polling loops (PR state, CI runs, workflow status), use the **Monitor tool** with a shell loop that emits one line per check: `while true; do result=$(...); echo "$result"; [[ done ]] && break; sleep 15; done`. The sleep runs inside the monitored process, not in a foreground Bash call. For one-shot delayed checks (e.g., wait before listing runs), use `Bash` with `run_in_background: true`. **Why:** The agent tried `sleep 10 && gh pr view` in a foreground Bash call and got blocked, stalling the merge pipeline.
- Never write to Claude Code memory (`~/.claude/projects/*/memory/`) or local-only locations [id: hr-never-write-to-claude-code-memory-claude]. All knowledge goes into committed repo files: AGENTS.md (hard rules), constitution.md (conventions), `knowledge-base/project/learnings/` (learnings via /compound), `.mcp.json` (MCP configs). Test: "If a new Soleur user clones this repo, do they get this improvement?" If no, it belongs in the repo.
- When in a worktree, never read from bare repo paths (`/home/jean/git-repositories/.../<file>` when `.git` is a file, not a directory) [id: hr-when-in-a-worktree-never-read-from-bare]. Use `git show HEAD:<path>` or the `Read` tool with the worktree absolute path only. Bare repos have no working tree; reads from bare paths return stale content. **Why:** In 2026-04-13 session, reading from bare repo path returned outdated file content during worktree debugging.
- Before running git commands on a directory (e.g., `git -C <path>`), validate it is a git repository with `git -C <path> rev-parse --git-dir >/dev/null 2>&1` [id: hr-before-running-git-commands-on-a]. Fail fast with "Not a git repository" instead of silent failure. **Why:** In 2026-04-13 session, `git -C .worktrees` on a non-repo path produced a confusing error downstream.
- GitHub API endpoints with enum parameters (e.g., `dismissed_reason`) require exact format matching [id: hr-github-api-endpoints-with-enum]. For CodeQL alert dismissal, valid reasons are space-separated strings (`"not vulnerable"`, `"used in tests"`) not snake_case. Always test API format against schema before calling, and catch HTTP 422 errors with schema-aware validation. **Why:** In 2026-04-13 session, two API calls failed with 422 because `snake_case` was passed instead of space-separated format.
- When Playwright MCP hits an auth wall, never close the browser and hand off by URL — keep the tab open at the login/OAuth page, tell the user "please log in now, I'll wait", then resume navigation in the same session once they're authenticated [id: hr-when-playwright-mcp-hits-an-auth-wall]. Closing the browser and pasting a link in chat is a silent handoff that violates the drive-to-completion rule (`hr-when-a-workflow-concludes-with-an`) — the user then has to re-open the tool, re-navigate, and manually complete the action the agent was supposed to drive. The only acceptable handoff is at the exact page where credentials must be typed, with the browser still open. **Why:** In the #1920 DNSSEC verification session, the agent reached the Cloudflare login page, closed the browser, and posted the dashboard URL in chat — bypassing automation for the entire post-login flow (DNSSEC tab → manual Publish DS) that Playwright could have driven after auth.

## Workflow Gates

- When moving GitHub issues between milestones, deferring features, changing priorities, or making any roadmap decision, update `knowledge-base/product/roadmap.md` in the same action [id: wg-when-moving-github-issues-between]. The roadmap document is the canonical product truth -- if it contradicts the issue tracker, neither can be trusted. Never change a milestone assignment without updating the corresponding roadmap phase table. **Why:** In the #1064 roadmap session, tag-and-route was deferred from P1 to P3 but the roadmap document was not updated, creating three conflicting sources of truth (roadmap, milestones, conversation). The CPO had to flag this on re-review.
- Every feature listed in a roadmap phase table MUST have a linked GitHub issue [id: wg-every-feature-listed-in-a-roadmap-phase]. When adding a feature to the roadmap, create the issue in the same action — even for trigger-gated phases where work has not started. Issues can be created with minimal scope and refined later, but they must exist so the milestone is never empty and work is trackable. When creating a new roadmap phase or milestone, verify every feature row has an Issue column entry before committing. **Why:** In the 2026-04-03 milestone audit, Phase 5 had 5 features defined in the roadmap but zero GitHub issues — the milestone existed as an empty shell for weeks because the workflow only enforced issues → milestones (every issue must have a milestone) but not milestones → issues (every milestone must have issues).
- When closing a phase milestone, update the `## Current State` section of `knowledge-base/product/roadmap.md` in the same commit [id: wg-when-closing-a-phase-milestone-update]. The Current State section is load-bearing -- domain leader agents (particularly CPO) read it to assess phase status. A closed milestone with an outdated Current State produces stale assessments. **Why:** In #1878, the CPO assessed Phase 1 as having 7 unfinished P1 items because the Current State section had not been updated after Phase 1 and Phase 2 were closed.
- When a research sprint produces recommendations, run the cascade-validate loop [id: wg-when-a-research-sprint-produces] [skill-enforced: work Phase 2.5]. "Findings written" is NOT done — "findings applied, validated, and all documents reflect the final state" is done.
- When fixing a workflow gate's detection logic, retroactively apply the fixed gate to the case that exposed the gap [id: wg-when-fixing-a-workflow-gates-detection] [skill-enforced: ship Phase 5.5 Retroactive Gate Application]. "Gate fixed" is not done — "gate fixed AND missed case remediated" is done.
- Zero agents until user confirms direction [id: wg-zero-agents-until-user-confirms]. Present a concise summary first, ask if they want to go deeper, only then launch research. Exception: passive domain routing (see below).
- Before every commit, run compound (`skill: soleur:compound`) [id: wg-before-every-commit-run-compound-skill]. Do not ask whether to run it -- just run it.
- Never bump version files in feature branches [id: wg-never-bump-version-files-in-feature]. Version is derived from git tags — CI creates GitHub Releases with `vX.Y.Z` tags at merge time via semver labels. Set labels with `/ship`. Do NOT edit `plugin.json` version (frozen sentinel) or `marketplace.json` version.
- Use `/ship` to automate the full commit/push/PR workflow [id: wg-use-ship-to-automate-the-full-commit]. It enforces review and compound gates.
- After marking a PR ready, run `gh pr merge <number> --squash --auto` to queue auto-merge, then poll `gh pr view <number> --json state --jq .state` until MERGED, then `cleanup-merged` [id: wg-after-marking-a-pr-ready-run-gh-pr-merge]. Never stop at "waiting for CI" -- actively poll and merge in the same session.
- After a PR merges to main, verify all release/deploy workflows succeed before ending the session [id: wg-after-a-pr-merges-to-main-verify-all] [skill-enforced: ship Phase 7]. A merged PR with a failing release workflow is a silent production outage.
- At session start, run `bash ./plugins/soleur/skills/git-worktree/scripts/worktree-manager.sh cleanup-merged && git worktree list` (works from the bare root or any worktree — the script detects bare repos and uses `git fetch origin main:main` + `sync_bare_files` to keep local main current) [id: wg-at-session-start-run-bash-plugins-soleur]. If no worktree exists for the current task, create one with `bash ./plugins/soleur/skills/git-worktree/scripts/worktree-manager.sh --yes create <name>` before doing any work. The repo root is a bare repository -- never run `git pull`, `git checkout`, or other working-tree commands from the bare root.
- At session start (after `cleanup-merged` has fetched main), refresh `.mcp.json` at the bare repo root: `git show main:.mcp.json > .mcp.json` [id: wg-at-session-start-after-cleanup-merged]. Claude Code reads `.mcp.json` from the CWD on startup to discover MCP servers (Playwright, etc.). Bare repos have no working tree, so this file must be manually kept in sync with the tracked version. Without this step, MCP servers added in feature branches are unavailable until the next session.
- When a test runner crashes (segfault, OOM, abort), never dismiss it as "known" or "unrelated" [id: wg-when-a-test-runner-crashes-segfault-oom]. Either fix the root cause, file a GitHub issue to track it, or document a concrete workaround. A crash without a tracking issue is a workflow violation.
- When tests fail and are confirmed pre-existing (same on main), create a GitHub issue to track them before proceeding [id: wg-when-tests-fail-and-are-confirmed-pre]. Pre-existing failures without tracking issues normalize a red suite.
- When an audit identifies pre-existing issues, create GitHub issues to track them before fixing [id: wg-when-an-audit-identifies-pre-existing]. Don't just note them in conversation -- file them.
- When deferring a capability, create a GitHub issue (what, why, re-evaluation criteria) milestoned to the target phase or "Post-MVP / Later" [id: wg-when-deferring-a-capability-create-a]. A deferral without a tracking issue is invisible.
- When a workflow gap causes a mistake, fix the skill or agent first — a learning is not a fix [id: wg-when-a-workflow-gap-causes-a-mistake-fix]. Only record a learning when no code change can address the gap. A verbal acknowledgment ("I'll do it differently next time") is not a fix — edit the definition file in the same response. **Why:** In PR #2430, the agent acknowledged a test-output workflow gap but only committed a verbal promise, forcing the user to ask again before a skill edit was made.
- Every session error MUST produce either an AGENTS.md rule, a skill instruction edit, or a hook — not just a learning file entry [id: wg-every-session-error-must-produce-either]. During compound, for each error ask: "Could a rule, hook, or skill instruction have prevented this?" If yes, make the change in the same commit. Documenting an error without routing it to a durable fix is a workflow violation. **Why:** In the 2026-04-13 session, 4 session errors were documented in a learning but no workflow changes were proposed until the founder flagged the gap.
- Use `Closes #N` in PR **body** (not title) to auto-close issues [id: wg-use-closes-n-in-pr-body-not-title-to]. For partial work, use `Ref #N` — never `Closes #N partially` (GitHub ignores qualifiers and auto-closes regardless).
- After merging a PR that adds or modifies a GitHub Actions workflow, trigger a manual run (`gh workflow run <file>.yml`), poll until complete (`gh run view <id> --json status,conclusion`), and investigate failures before moving on [id: wg-after-merging-a-pr-that-adds-or-modifies]. New workflows must be verified working, not just syntactically valid.
- When a PR includes database migrations (`supabase/migrations/`), verify they are applied to production before closing the issue [id: wg-when-a-pr-includes-database-migrations]. Test via Supabase REST API. A committed-but-unapplied migration is a silent deployment failure.
- When a feature creates external resources (Cloud tasks, Doppler configs, DNS, infrastructure), validate each resource produces correct output BEFORE shipping [id: wg-when-a-feature-creates-external]. Never ship without running each new service and verifying output.
- For user-facing pages with a Product/UX Gate, specialists (ux-design-lead, copywriter) must produce artifacts before implementation [id: wg-for-user-facing-pages-with-a-product-ux]. Code implements from approved artifacts, not brainstorm notes.
- When a feature stores user-generated content, verify the data export path during brainstorm [id: wg-when-a-feature-stores-user-generated]. User data must be exportable without custom tooling — prefer git-committed storage over cloud-only storage. If cloud storage is chosen, the spec must document the export mechanism. **Why:** In #1974, data portability was identified as a deciding factor between Supabase Storage and git-committed uploads — the git approach won because `git clone` gives users everything with zero vendor lock-in.

## Code Quality

- Write failing tests BEFORE implementation code when a plan includes Test Scenarios or Acceptance Criteria [id: cq-write-failing-tests-before] [skill-enforced: work Phase 2 TDD Gate]. Infrastructure-only tasks (config, CI, scaffolding) are exempt.
- Always run `npx markdownlint-cli2 --fix` on changed `.md` files before committing [id: cq-always-run-npx-markdownlint-cli2-fix-on]. Re-read after `replace_all` on Markdown tables to verify cell spacing.
- Ensure dependencies are installed at the correct package level (not just root) before tests or CI [id: cq-ensure-dependencies-are-installed-at-the]. Check subdirectories like `agent-browser/` or app-level packages.
- In worktrees, run vitest via `node node_modules/vitest/vitest.mjs run` instead of `npx vitest run` [id: cq-in-worktrees-run-vitest-via-node-node]. The npx cache is shared across worktrees and can resolve to a stale worktree's vitest installation, causing phantom test failures with wrong path resolution. When vitest is installed at app level (e.g., `apps/web-platform/node_modules/`) rather than root, `cd` to the app directory and run `./node_modules/.bin/vitest run` instead. **Why:** In the 2026-04-13 session, `npx vitest` resolved to `feat-fix-kb-pdf-upload-formdata`'s vitest while running in `feat-dashboard-agent-identity`, causing 108 phantom failures.
- For production debugging, use observability tools — never SSH for logs [id: cq-for-production-debugging-use]. Priority: (1) Sentry API (`SENTRY_API_TOKEN` from Doppler `prd`), (2) Better Stack, (3) `/health` endpoint. SSH is for infrastructure provisioning only.
- For deploy webhook observability debugging, fetch `WEBHOOK_DEPLOY_SECRET`, `CF_ACCESS_CLIENT_ID`, `CF_ACCESS_CLIENT_SECRET` from the Doppler `prd_terraform` config (not `prd`) [id: cq-deploy-webhook-observability-debug]. GET `https://deploy.soleur.ai/hooks/deploy-status` with CF Access headers and an HMAC-sha256 signature over the empty body (`printf '' | openssl dgst -sha256 -hmac "$SECRET"`); parse `exit_code`/`reason` from the JSON response. See `plugins/soleur/skills/postmerge/references/deploy-status-debugging.md`.
- When lefthook hangs in a worktree (>60s), kill (`pkill -f "lefthook run"`), verify checks manually, commit with `LEFTHOOK=0` [id: cq-when-lefthook-hangs-in-a-worktree-60s] [hook-enforced: detect_bypass]. Known lefthook/worktree bug.
- Never skip pre-commit or pre-push hooks with `--no-verify` unless the user has explicitly authorized the bypass [id: cq-never-skip-hooks] [hook-enforced: detect_bypass]. `--no-verify` evades lefthook's lint, test, and guard checks; prefer fixing the blocking hook over skipping it. If a hook is genuinely broken, file an issue rather than routinely bypassing it.
- Before calling `mcp__pencil__open_document`, ensure the target .pen file is committed in git [id: cq-before-calling-mcp-pencil-open-document] [hook-enforced: pencil-open-guard.sh]. Untracked .pen files have no recovery path — `open_document` can silently overwrite them with an empty document.
- Playwright MCP uses `--isolated` mode (`.mcp.json`) [id: cq-playwright-mcp-uses-isolated-mode-mcp]. If singleton lock fails, kill Chrome processes. Do not remove `--isolated` — required for parallel sessions.
- After completing a Playwright task, call `browser_close` [id: cq-after-completing-a-playwright-task-call] [hook-enforced: browser-cleanup-hook.sh].
- Before pushing `package.json` changes, verify deps are in the correct `package.json` (app-level, not just root) and both `bun.lock` and `package-lock.json` are regenerated if both exist (Dockerfile uses `npm ci`) [id: cq-before-pushing-package-json-changes].
- Doppler service tokens are per-config — use config-specific GitHub secret names (`DOPPLER_TOKEN_PRD`, `DOPPLER_TOKEN_CI`), never bare `DOPPLER_TOKEN` [id: cq-doppler-service-tokens-are-per-config]. The `-c` flag is silently ignored with service tokens. See `knowledge-base/project/learnings/2026-03-29-doppler-service-token-config-scope-mismatch.md`.
- When running terraform commands locally with Doppler, always use `doppler run --name-transformer tf-var` to match CI behavior [id: cq-when-running-terraform-commands-locally]. Export `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` separately for the R2 backend — the name transformer renames them to `TF_VAR_*` which the backend ignores.
- CI steps polling JSON endpoints under `bash -e` must precede every `jq -r` / `jq '.field'` call with a `jq -e . >/dev/null 2>&1` guard that `continue`s on non-JSON bodies [id: cq-ci-steps-polling-json-endpoints-under]. Without the guard, a cold-start 404 (plaintext), Cloudflare 503 (HTML), or webhook-restart connection error kills the step via `set -e` before the retry loop can react. Do NOT use `jq empty` (too permissive — passes `null` through to field parsers). Do NOT drop `-e` from the step shell (silences real failures elsewhere in the loop). **Why:** In #2214, the `Verify deploy script completion` step crashed on release run 24411905995 when `/hooks/deploy-status` returned adnanh/webhook's plaintext "Hook not found" — `jq` exited 5 inside `$(...)`, `bash -e` killed the step, and the 120s timeout + retry loop never ran. See `knowledge-base/project/learnings/bug-fixes/2026-04-15-signed-get-verify-step-tolerate-non-json-bodies.md`.
- Rule IDs (`[id: <prefix>-<slug>]`) on AGENTS.md rules are immutable once assigned [id: cq-rule-ids-are-immutable] [hook-enforced: lint-rule-ids.py]. Rewording a rule preserves its ID; removing or reassigning an ID requires an explicit deprecation note and a tracking issue. Section prefixes (`hr`, `wg`, `cq`, `rf`, `pdr`, `cm`) must match the rule's section.
- `gh issue create --milestone` takes the milestone **title**, not the numeric ID returned by the `/repos/:owner/:repo/milestones` endpoint [id: cq-gh-issue-create-milestone-takes-title]. Use `--milestone "Post-MVP / Later"`, not `--milestone 6`. The numeric form fails with `could not add to milestone 'N': 'N' not found`. The `gh` CLI also has **no `milestone` subcommand** — to list milestones use `gh api repos/:owner/:repo/milestones --jq '.[] | "\(.number) | \(.title) | \(.state)"'`, not `gh milestone list`. **Why:** In the #2284 compound session, `gh issue create --milestone 6` failed because the CLI resolves milestones by title. In the #2345 brainstorm session, `gh milestone list` failed with "unknown command" — the CLI surface doesn't include a milestone subcommand at all.
- For local verification of apps that read env vars, invoke as a single Bash call: `cd <abs-path> && doppler run -p soleur -c dev -- npm run <script>` (for `apps/web-platform` use the wrapper `./scripts/dev.sh [PORT]`) [id: cq-for-local-verification-of-apps-doppler]. Three failure modes this prevents: (a) skipping `doppler run` → `Missing SUPABASE_URL` at boot; (b) invoking transitive binaries like `tsx` directly under `doppler run` → binary not on PATH outside npm scripts; (c) relying on CWD from a prior Bash call → shell state does not persist across tool invocations, the command runs from the wrong directory and produces empty output. **Why:** In PR #2350 the agent hit all three failure modes in sequence while trying to verify a 2-line config fix, burning a full Doppler/tsx/CWD round-trip. The wrapper script at `scripts/dev.sh` collapses them into one entry point.
- Before `gh issue create --label <name>`, run `gh label list --limit 100 | grep -i <keyword>` to confirm the label's exact name [id: cq-gh-issue-label-verify-name]. The repo uses namespaced labels (`priority/p0-critical`…`priority/p3-low`, `type/security`, `type/feature`, `domain/*`); bare names like `security`, `P3`, or `bug` fail with `'<name>' not found` and abort the issue creation mid-flight. **Why:** In the PR #2347 review-synthesis session, `--label "security"` and `--label "P3"` both failed — recovery required re-listing labels and re-running the command. A pre-check eliminates the retry round.
- Never use `require()` in Vite-transformed test or source files [id: cq-vite-test-files-esm-only]. Vite's transform pass rejects dynamic `require()` in ESM files with `Failed to resolve import`. Use top-level `import` for static deps or `await import()` for dynamic module loading. **Why:** In the PR #2347 selection-toolbar test, a `require("@/components/kb/selection-toolbar")` inside a React harness failed at transform (pre-runtime), masking the test's intended RED assertion.
- When running `npx markdownlint-cli2 --fix`, pass specific file paths rather than repo-wide globs like `"knowledge-base/**/*.md"` [id: cq-markdownlint-fix-target-specific-paths]. Repo-wide globs touch ~100 files on first run and make the subsequent `git add` a blast-radius risk (easy to commit unrelated auto-formats). For a deliberate repo-wide pass, run it as its own commit with a `chore(docs): markdownlint --fix repo-wide` subject so the change is reviewable. **Why:** In the PR #2347 Phase 3 session, a `--fix "knowledge-base/**/*.md"` modified 100+ unrelated files; recovery required `git checkout --` on large subtrees before staging only the intended plan edit.
- App Router route files (`apps/*/app/**/route.{ts,tsx,js,jsx}`) may export only HTTP method handlers (`GET`/`POST`/`PUT`/`PATCH`/`DELETE`/`HEAD`/`OPTIONS`) and Next.js-recognized config exports (`runtime`, `dynamic`, `revalidate`, `fetchCache`, `dynamicParams`, `preferredRegion`, `maxDuration`, `generateStaticParams`, `metadata`, `generateMetadata`) [id: cq-nextjs-route-files-http-only-exports]. Any other export — singletons, helpers, test resets, constants — must live in a sibling module. Vitest and `tsc --noEmit` do NOT run the Next.js route-file validator; only `next build` does, so this class of error only surfaces during the post-merge Docker build. **Why:** In PR #2347, `app/api/analytics/track/route.ts` exported `analyticsTrackThrottle` (SlidingWindowCounter singleton) and `__resetAnalyticsTrackThrottleForTest`. The build failed post-merge with `Type error: Route ... does not match the required types of a Next.js Route`, producing a ~15 min prod outage until hotfix #2401 moved the non-HTTP exports to `./throttle.ts`. See `knowledge-base/project/learnings/runtime-errors/2026-04-15-nextjs-15-route-file-non-http-exports.md`.
- When a feature involves loading large assets (PDFs, videos, images, archives, long-document markdown) or a streamable resource, design for **progressive/gradual rendering** from the start — don't default to "fetch complete then render" [id: cq-progressive-rendering-for-large-assets]. Checklist: (a) does the server send `Accept-Ranges: bytes` and handle `Range` requests (HTTP 206 Partial Content)? (b) does the renderer show meaningful content as bytes arrive (skeleton, first page, low-res placeholder, progress percentage)? (c) is there a loading indicator with progress, not just a spinner? (d) for images, use `loading="lazy"` + `decoding="async"`; for videos, use HTML5 `preload="metadata"`; for PDFs, confirm PDF.js range-transport is active. **Why:** In the resizable-panels rollout (#2433), PDFs downloaded fully before any page rendered because the KB content API returned the full buffer with no `Accept-Ranges` header — a 60-page PDF showed only a spinner for seconds even though PDF.js supports progressive rendering out of the box. The fix (#2451) added Range support to `buildBinaryResponse`. Designing progressive loading up-front would have shipped the fast-path in the first PR.

## Review & Feedback

- After merging, read files from the merged branch (`git show main:<path>`), not the bare repo directory (stale) [id: rf-after-merging-read-files-from-the-merged].
- Never skip QA/review before merging [id: rf-never-skip-qa-review-before-merging]. Full pipeline: plan → implement → review → QA → compound → ship.
- Before spawning review agents, push the branch to remote (`git push -u origin <branch>`) [id: rf-before-spawning-review-agents-push-the]. Review subagents may use `git show` or remote state; unpushed commits produce stale analysis where agents report on planning artifacts instead of implementation code. **Why:** In the 2026-04-13 session, 4 of 9 review agents analyzed pre-push state and produced findings about code that didn't exist yet.
- Before shipping, verify: (1) review comments resolved, (2) QA run with screenshots if UI, (3) tests pass locally [id: rf-before-shipping-verify-1-review-comments].
- When a reviewer or user says to keep a feature/phase, do not remove it without explicit confirmation [id: rf-when-a-reviewer-or-user-says-to-keep-a].
- Review findings default to fix-inline on the PR branch for all severities (P1/P2/P3) [id: rf-review-finding-default-fix-inline] [skill-enforced: ship Phase 5.5 Review-Findings Exit Gate]. Filing a GitHub issue is allowed only when a scope-out criterion applies. Review's four criteria (cross-cutting-refactor, contested-design, architectural-pivot, pre-existing-unrelated) are defined inline in `plugins/soleur/skills/review/SKILL.md` Section 5; compound's three analogous criteria (cross-skill, contested-design, agents-md-semantic-change) are defined inline in `plugins/soleur/skills/compound/SKILL.md` Route Learning to Definition — they differ intentionally because compound operates on definition files, not code findings. Scope-out issues MUST carry the `deferred-scope-out` label AND a `## Scope-Out Justification` section in the body. `/ship` Phase 5.5 blocks merge on any open review-origin issue cross-referencing the PR without `deferred-scope-out`. Compound's route-to-definition follows the same default (direct edit) + exception (scope-out) pattern in both interactive and headless modes. **Why:** In #2374, 53 review-origin issues were filed in 3 days because the three skills (review, compound, ship) each defaulted to "file as issue" rather than "fix inline." Findings were legitimate; disposition was wrong.

## Passive Domain Routing

- When a user message contains a clear, actionable domain signal unrelated to the current task (expenses, legal commitments, marketing mentions, sales leads, etc.), read `plugins/soleur/skills/brainstorm/references/brainstorm-domain-config.md` and spawn the relevant domain leader as a background agent (`run_in_background: true`) using the Assessment Question to detect relevance and the Task Prompt to delegate [id: pdr-when-a-user-message-contains-a-clear]. Continue the primary task without waiting.
- Do not route on trivial messages ("yes", "continue", "looks good") or when the domain signal IS the current task's topic (e.g., do not route to CTO during an engineering brainstorm about architecture) [id: pdr-do-not-route-on-trivial-messages-yes].

## Communication

- Challenge reasoning instead of validating [id: cm-challenge-reasoning-instead-of]. No flattery. If something looks wrong, say so.
- Delegate verbose exploration (3+ file reads, research, analysis) to subagents [id: cm-delegate-verbose-exploration-3-file]. Keep main context for edits and user-facing iteration.
- When proposing to clear context or resume from a fresh session, always provide a copy-pasteable resume prompt that includes: skill/command to run, plan file path, branch name, worktree path, PR number, issue number, and a one-line summary of what was already done [id: cm-when-proposing-to-clear-context-or]. The user should be able to paste and go without re-explaining context.
