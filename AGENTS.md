# Agent Instructions

This repository contains the Soleur Claude Code plugin. Detailed conventions live in `knowledge-base/project/constitution.md` -- read it when needed. This file contains only rules the agent will violate without being told on every turn.

## Hard Rules

- Never commit directly to main [hook-enforced: guardrails.sh Guard 1]. Create a worktree: `git worktree add .worktrees/feat-<name> -b feat/<name>`. If one exists for the task, use it.
- Never `--delete-branch` with `gh pr merge` [hook-enforced: guardrails.sh Guard 3]. Use `gh pr merge <number> --squash --auto`, then poll with `gh pr view <number> --json state --jq .state` until MERGED, then run `cleanup-merged`.
- Never edit files in the main repo when a worktree is active [hook-enforced: worktree-write-guard.sh]. Run `pwd` before every file write or git command to verify you're in `.worktrees/<name>/`.
- Never `git stash` in worktrees. Commit WIP first, then merge.
- Never `rm -rf` on the current directory, a worktree path, or the repo root [hook-enforced: guardrails.sh Guard 2].
- MCP tools (Playwright, etc.) resolve paths from the repo root, not the shell CWD. Always pass absolute paths to MCP tools when in a worktree.
- When a command exits non-zero or prints a warning, investigate before proceeding. Never treat a failed step as success.
- Before merging any PR, merge origin/main into the feature branch [hook-enforced: pre-merge-rebase.sh] (`git fetch origin main && git merge origin/main`).
- Every `gh issue create` must include `--milestone` [hook-enforced: guardrails.sh Guard 5]. Default to "Post-MVP / Later" for operational issues. For feature issues (plan, brainstorm, work), read `knowledge-base/product/roadmap.md` to determine the correct phase.
- Always read a file before editing it. The Edit tool rejects unread files, but context compaction erases prior reads -- re-read after any compaction event.
- When a plan specifies relative paths (e.g., `source "$SCRIPT_DIR/../../..."`), trace each `../` step to verify the final target before implementing. Plans have prescribed wrong paths that were implemented verbatim and only caught by review agents.
- PreToolUse hooks block: commits on main, rm -rf on worktrees, --delete-branch with active worktrees, writes to main repo when worktrees exist, commits with conflict markers in staged content, gh issue create without --milestone, gh pr merge without review evidence [hook-enforced: pre-merge-rebase.sh]. Work with these guards, not around them.
- The host terminal is Warp. Do not attempt automated terminal manipulation via escape sequences (cursor position queries, TUI rendering, and similar sequences are intercepted by Warp's tmux control mode and silently fail).
- The Bash tool runs in a non-interactive shell without `sudo` access. Do not attempt commands requiring elevated privileges -- provide manual instructions instead.
- Before prompting the user for any API key, access token, or service credential, check Doppler first: `doppler secrets get <KEY_NAME> --project soleur --config dev --plain 2>/dev/null`. If the key name is uncertain, list available secrets with `doppler secrets --only-names -p soleur -c dev` and check additional configs (`prd`, `ci`, `prd_terraform`). Only prompt the user if the key is not found in any Doppler config. **Why:** In the feat-repo-connection session (2026-03-29), the agent attempted interactive `pencil login` when `PENCIL_CLI_KEY` was already stored in Doppler `soleur/dev` -- a wasted round-trip that violated the "exhaust all automated options" rule.
- Exhaust all automated options before suggesting manual steps to the user. The founder is a solo operator -- every manual step is a context switch. Priority chain for external service interactions: (1) MCP tools (`ToolSearch` for keywords), (2) CLI tools (`gh`, `doppler`, `hcloud`, `supabase`), (3) REST APIs (`curl`/`WebFetch`), (4) Playwright MCP, (5) manual handoff. Always run `ToolSearch` before opening a browser. When a CLI or API requires an access token that is not stored (e.g., `supabase login`, service dashboard tokens), use Playwright MCP to generate/retrieve the token, store it in Doppler, then switch back to CLI/API for the actual work. Playwright is a bootstrapping tool for credentials, not the primary execution path. **Why:** In #1094, Playwright was used to create a Cloud scheduled task when the `RemoteTrigger` MCP tool was available the entire time — one API call vs 10+ Playwright interactions. In the 2026-03-28 session, the agent used Playwright to run SQL in the Supabase dashboard instead of generating a Supabase access token first and using the CLI.
- When blocked by missing information or access, check if a CLI tool can be installed to retrieve it (`hcloud`, `doctl`, `flyctl`, `wrangler`, `aws`, `gcloud`, etc.) before labeling the step as manual. Install with `curl`/`tar` or `npm` to `~/.local/bin` -- no `sudo` needed. "Exhaust all automated options" includes tool installation, not just tool usage. **Why:** In #748 the agent tried 7 read-only approaches to get a server IP but never considered installing `hcloud`, which was a 10-second `curl | tar` away.
- Never label a browser task as "manual" without first attempting Playwright MCP. Account signups, credential generation, settings configuration, and form submissions are all automatable via Playwright. The only genuinely manual browser steps are CAPTCHA solving and interactive OAuth consent screens — and even those should be driven to the CAPTCHA/consent step via Playwright, then handed to the user for that single interaction. Plans and task lists that say "manual — browser" are a code smell.
- Plans, PRs, and task lists must not contain "manual steps needed" post-merge sections without first verifying each step cannot be automated (`terraform apply`, `ssh`, `gh secret set`, `gh workflow run`, `hcloud`, etc.). A plan that says "Manual: SSH into server" when SSH is available to the agent is a workflow violation — the same as labeling a browser task manual without trying Playwright. **Why:** In #982, five "manual" post-merge steps were all automatable via `doppler run -- terraform apply`, `ssh`, `gh secret set`, and `gh workflow run`. The plan propagated "manual" labels unchallenged through the entire pipeline.
- Always use Terraform for infrastructure provisioning — never vendor-specific APIs (Hetzner API, AWS CLI, etc.) for creating servers, volumes, firewalls, or DNS records. Terraform provides reproducibility and portability across cloud providers. Use vendor APIs only for read-only operations (checking availability, listing resources) or account-level tasks that Terraform doesn't cover (account creation, API token generation). Existing Terraform patterns live in `apps/telegram-bridge/infra/` and `apps/web-platform/infra/`.
- Every new Terraform root must include an R2 remote backend block. Use the `soleur-terraform-state` bucket with a unique key path matching the app name (e.g., `key = "<app-name>/terraform.tfstate"`). Copy the backend block from `apps/telegram-bridge/infra/main.tf` as the template. Local state is never acceptable.
- In GitHub Actions `run:` blocks, never indent heredoc body content to match YAML nesting depth -- the indentation becomes literal content and breaks Markdown rendering (4+ leading spaces = code block). Left-align heredoc content or extract to a separate file. **Why:** In #974, a 10-space-indented issue body rendered as a `<pre>` block instead of formatted Markdown.
- New skills, agents, or user-facing capabilities must include CPO and CMO at minimum in brainstorm domain assessment. New capabilities always have product strategy and marketing implications (content, positioning, announcements). CTO is included when the capability has architectural implications. **Why:** In #675, the CMO was not consulted during brainstorm for a new product-roadmap skill. The domain assessment ran once at the start (when scope was "internal tooling") and was never re-run after the scope pivoted to "product feature for all users."
- Before shipping, `/soleur:ship` Phase 5.5 runs three conditional domain leader gates in parallel: (1) **CMO content-opportunity gate** — triggers on file-path matches (`knowledge-base/product/research/`, `knowledge-base/marketing/`, new workflow patterns) or feature signals (`semver:minor`/`major` label, `feat:` title); assesses content and distribution opportunities. (2) **CMO website framing review gate** — triggers when PRs modify `brand-guide.md` positioning/framing sections; audits website copy for alignment with updated framings and proposes specific copy changes. (3) **COO expense-tracking gate** — triggers when the session involved new service signups, tool provisioning, or API subscriptions; ensures new costs are recorded in `knowledge-base/operations/expenses.md`. **Why:** In #1173, brand guide was updated with new primary framing and memory-first variant but the website still used old copy (CMO website gap). Research produced content-worthy data but no content was planned (CMO content gap). In prior sessions, new subscriptions went unrecorded (COO gap).
- When a workflow concludes with an actionable next step (subscription, account setup, configuration change, deployment), execute it — don't list it as "next action" and stop. Use Playwright MCP, `xdg-open`, CLI tools, or APIs to drive the action to completion. If the step requires user credentials or interaction (login, payment), navigate to the exact page and hand off there. Stopping at "Action item: do X" when X is automatable is the same violation as labeling a browser task manual. **Why:** In #1043, the agent completed an analytics comparison brainstorm but stopped at "Action item: subscribe to Plausible" instead of opening the subscription page — a 5-second `xdg-open` away.
- Before asserting the status of a GitHub issue (open, closed, not started, in progress), verify via `gh issue view <N> --json state`. Before claiming work "has not been done" or "has not started," check `knowledge-base/` for existing artifacts (specs, memos, status docs) and `gh pr list --search "<N>"` for related PRs. Unverified status claims waste the founder's time and create false urgency. **Why:** In the #1056 investigation, both CLO and COO asserted "#670 not started" without checking — #670 had been closed 5 days earlier with all DPAs signed and verified.
- Never run commands with unbounded output in subagents (Agent tool). Subagent stdout is captured to a `.output` file on tmpfs (`/tmp`). A single verbose command (`cat` on a large file, build log without `| tail`, recursive `find` on a deep tree) can produce a multi-GiB file that fills tmpfs and crashes all Claude sessions. Always pipe through `| head -n 500` or `| tail -n 200` when output size is uncertain. For builds, use `2>&1 | tail -n 100`. The `scripts/tmpfs-guard.sh` cron job is a safety net, not a substitute for bounded output. **Why:** On 2026-03-28, a single subagent `.output` file grew to 16 GiB, filled the entire tmpfs, and killed all Claude Code sessions and bash usage on the machine.
- Never write to Claude Code memory (`~/.claude/projects/*/memory/`). CC memory is local to one machine and one user — it does not transfer to other systems, other Soleur users, or other agents. All knowledge must go into committed repo files: AGENTS.md (hard rules), constitution.md (conventions), `knowledge-base/project/learnings/` (session learnings via /compound). When the auto-memory system prompts you to save, ignore it and commit to the repo instead. **Why:** In the 2026-03-28 session, a Supabase CLI feedback was written to CC memory instead of being recognized as a restatement of the existing AGENTS.md priority chain rule. The "knowledge" was trapped on one machine.

## Workflow Gates

- When moving GitHub issues between milestones, deferring features, changing priorities, or making any roadmap decision, update `knowledge-base/product/roadmap.md` in the same action. The roadmap document is the canonical product truth -- if it contradicts the issue tracker, neither can be trusted. Never change a milestone assignment without updating the corresponding roadmap phase table. **Why:** In the #1064 roadmap session, tag-and-route was deferred from P1 to P3 but the roadmap document was not updated, creating three conflicting sources of truth (roadmap, milestones, conversation). The CPO had to flag this on re-review.
- When a research sprint, audit, or analysis produces recommendations targeting specific artifacts, run the full cascade-validate loop: (1) apply recommendations to source artifacts, (2) re-run validation against updated artifacts, (3) if new weak spots emerge, fix and re-validate, (4) update the research brief with final validated results and a Cascade Status section, (5) present a founder summary (findings table + files changed table). "Findings written" is NOT done — "findings applied, validated, and all documents reflect the final state" is done. The `/soleur:work` skill enforces this via Phase 2.5. Max 3 iterations. **Why:** In #1173, a synthetic research sprint required 3 manual prompts from the founder ("was any action taken?", "can you re-run?", "did we cascade the learnings?") before the work was actually complete. Each time the answer was "no." The loop must be automatic.
- When fixing a workflow gate's detection logic (trigger conditions, assessment questions, routing rules), retroactively apply the fixed gate to the case that exposed the gap before shipping the fix. The fix session must not end at "gate is fixed" — it must also run the gate against the original missed case and produce the artifacts (content briefs, expense entries, website audits) that would have been created if the gate had worked. "Gate fixed" is not done — "gate fixed AND missed case remediated" is done. **Why:** In #1265, the CMO content gate was fixed to catch product features like PWA, but the PWA feature itself was never assessed for content — the founder had to ask "did we run this against PWA?"
- Zero agents until user confirms direction. Present a concise summary first, ask if they want to go deeper, only then launch research. Exception: passive domain routing (see below).
- Before every commit, run compound (`skill: soleur:compound`). Do not ask whether to run it -- just run it.
- Never bump version files in feature branches. Version is derived from git tags — CI creates GitHub Releases with `vX.Y.Z` tags at merge time via semver labels. Set labels with `/ship`. Do NOT edit `plugin.json` version (frozen sentinel) or `marketplace.json` version.
- Use `/ship` to automate the full commit/push/PR workflow. It enforces review and compound gates.
- After marking a PR ready, run `gh pr merge <number> --squash --auto` to queue auto-merge, then poll `gh pr view <number> --json state --jq .state` until MERGED, then `cleanup-merged`. Never stop at "waiting for CI" -- actively poll and merge in the same session.
- At session start, from any active worktree (not the bare repo root): run `bash ../../plugins/soleur/skills/git-worktree/scripts/worktree-manager.sh cleanup-merged && git worktree list`. If no worktree exists for the current task, run `git worktree list` from the bare root, then create a worktree with `bash ./plugins/soleur/skills/git-worktree/scripts/worktree-manager.sh --yes create <name>` before doing any work. The repo root is a bare repository -- never run `git pull`, `git checkout`, or other working-tree commands from the bare root.
- When a test runner crashes (segfault, OOM, abort), never dismiss it as "known" or "unrelated". Either fix the root cause, file a GitHub issue to track it, or document a concrete workaround. A crash without a tracking issue is a workflow violation.
- When an audit identifies pre-existing issues, create GitHub issues to track them before fixing. Don't just note them in conversation -- file them.
- When a workflow gap causes a mistake, fix the skill or agent first — a learning is not a fix. Learnings are for context that *cannot* be encoded as a rule (e.g., "this API is flaky on Tuesdays"). If the failure can be prevented by changing a skill, agent, or hook, make that change. Only record a learning when no code change can address the gap. **Why:** In #656, the agent recorded a learning ("read wireframes before implementing") instead of fixing `/work` to enforce it — the same mistake would have recurred in the next session.
- When creating PRs that resolve a GitHub issue, include `Closes #N` in the PR **body** (not just the title). Parenthetical `(#N)` in titles creates a link but does NOT trigger auto-close.
- After merging a PR that adds or modifies a GitHub Actions workflow, trigger a manual run (`gh workflow run <file>.yml`), poll until complete (`gh run view <id> --json status,conclusion`), and investigate failures before moving on. New workflows must be verified working, not just syntactically valid.
- When a PR includes database migrations (files in `supabase/migrations/`), verify they have been applied to the production database BEFORE closing the issue. Use the Supabase REST API to test: query a column added by the migration (`curl "${SUPABASE_URL}/rest/v1/<table>?select=<new_column>&limit=1"`). A migration committed but not applied is a silent deployment failure — the code runs against a schema that doesn't match. **Why:** In the 2026-03-28 session, migration `010_tag_and_route.sql` was committed and deployed but never applied, causing `NOT NULL` constraint failures on every Command Center session start.
- When a feature creates external resources (Cloud scheduled tasks, Doppler configs, Cloud environments, DNS records, infrastructure), validate each resource produces correct output BEFORE shipping. For Cloud tasks: `RemoteTrigger run` + verify the session creates expected artifacts (PRs, issues, files). For Doppler configs: `doppler secrets --only-names` to verify all expected keys exist. Never ship a migration without running each migrated service and comparing output with the previous implementation. **Why:** In #1094, 3 Cloud tasks were created and shipped without test runs. Community-monitor failed silently (Doppler secrets not accessible), discovered only when the user checked the output issue.
- For user-facing pages with a BLOCKING Product/UX Gate, specialists (ux-design-lead, copywriter) must produce artifacts before implementation. Domain leader (CMO) reviews both. Code implements from approved artifacts, not brainstorm notes. **Why:** In #656 v2, the agent went straight from brainstorm to coding HTML/CSS without wireframes or reviewed copy — the user caught it and flagged the entire page as unreviewed.

## Code Quality

- Always run `npx markdownlint-cli2 --fix` on changed `.md` files before committing. Blank lines are required around headings and lists (MD032, MD038). **Why:** Markdown lint errors were the #1 friction source across sessions, blocking commits and requiring multiple fix iterations.
- When editing Markdown tables, verify that adjacent cell spacing and alignment are preserved after any `replace_all` operation. Re-read the file after table edits to confirm formatting. **Why:** `replace_all` operations on Markdown tables broke adjacent cell spacing in multiple sessions, causing lint failures and rework.
- Ensure all dependencies are installed at the correct package level (not just root) before running tests or CI. Check that `bun install` or equivalent has been run in subdirectories like `agent-browser/` or app-level packages. **Why:** CI failures due to missing dependencies (pino, agent SDK) at the app level occurred in multiple sessions, requiring workflow fixes and test rewrites.
- For production debugging, use observability tools (Sentry, Better Stack) — never SSH into servers to read logs. Priority chain: (1) Sentry API (`curl` with `SENTRY_API_TOKEN` from Doppler `prd` config) for errors and exceptions, (2) Better Stack for uptime and structured logs, (3) health endpoint (`/health`) for service status. SSH is for infrastructure provisioning and deployment, not log inspection. **Why:** In the 2026-03-28 session, the agent attempted SSH into the production server to check Docker logs instead of querying Sentry, which was already configured with `captureException` calls in the WebSocket handler and agent runner.
- When working with Playwright browser testing, check for Chrome user-data-dir lock conflicts before launching. If Playwright MCP fails due to singleton lock, kill existing Chrome processes or use a fresh profile directory. **Why:** Playwright MCP failed 3+ times due to Chrome singleton user-data-dir lock, and version mismatches caused additional fallback friction.
- Before pushing any changes that modify `package.json` or add new imports, verify: (1) the dependency is listed in the correct `package.json` (app-level, not just root), (2) the CI workflow installs it, (3) no version mismatches exist across lockfiles. Use a sub-agent for this check when multiple packages are involved. **Why:** CI failures from missing or misplaced dependencies (pino, agent SDK, Playwright) caused friction in 3+ sessions, each requiring workflow fixes and re-runs.
- When adding `doppler run -c <config>` or `doppler secrets get` to a GitHub Actions workflow, the `DOPPLER_TOKEN` GitHub secret must be a service token scoped to that exact config. Use config-specific secret names (`DOPPLER_TOKEN_PRD`, `DOPPLER_TOKEN_PRD_TF`, `DOPPLER_TOKEN_CI`) -- never bare `DOPPLER_TOKEN`. Do not set `DOPPLER_PROJECT`/`DOPPLER_CONFIG` env vars alongside service tokens (they are ignored; the token's built-in scope wins). See `knowledge-base/project/learnings/2026-03-29-doppler-service-token-config-scope-mismatch.md` for the full prevention checklist. **Why:** A CI workflow used `doppler run -c prd` with a service token scoped to a different config. Doppler service tokens are per-config credentials -- the `-c` flag is silently ignored, causing a cryptic 403.

## Review & Feedback

- After merging a PR, always read files from the merged branch (using `git show main:<path>` or checking out the branch) rather than reading from the bare repo directory, which may contain stale files. **Why:** Multiple sessions hit issues with stale bare-repo files being read post-merge, causing incorrect reviews and wasted cycles.
- Never skip QA/review phases before merging, even if the dev server isn't running. If the dev server is needed, start it first. The full pipeline is: plan → implement → review → QA → compound → ship. **Why:** QA and review agents were skipped in multiple sessions due to dev server not running or workflow gaps, leading to bugs reaching production.
- Before starting the ship phase, verify: (1) all review comments are resolved, (2) QA was run with browser screenshots if UI changes were made, (3) all tests pass locally. If any of these are false, stop and complete them first. **Why:** Failures clustered around skipped phases -- QA skipped because dev server wasn't running, post-merge validation missed, review agents reading stale files. These are "last mile" problems that a pre-ship checklist prevents.
- When a reviewer or user says to keep a feature/phase (e.g., "defense-in-depth"), do not remove it. Ask for explicit confirmation before removing any phase or feature that was part of the original plan. **Why:** Claude removed Phase 5.5 based on reviewer recommendations but the user wanted it kept as defense-in-depth, requiring restoration and realignment.

## Passive Domain Routing

- When a user message contains a clear, actionable domain signal unrelated to the current task (expenses, legal commitments, marketing mentions, sales leads, etc.), read `plugins/soleur/skills/brainstorm/references/brainstorm-domain-config.md` and spawn the relevant domain leader as a background agent (`run_in_background: true`) using the Assessment Question to detect relevance and the Task Prompt to delegate. Continue the primary task without waiting.
- Do not route on trivial messages ("yes", "continue", "looks good") or when the domain signal IS the current task's topic (e.g., do not route to CTO during an engineering brainstorm about architecture).

## Communication

- Challenge reasoning instead of validating. No flattery. If something looks wrong, say so.
- Delegate verbose exploration (3+ file reads, research, analysis) to subagents. Keep main context for edits and user-facing iteration.
