## Critical Issues

None. The site clears every blocking check: structured data is rendered in static HTML (not injected by JS), robots.txt allows every user agent including all AI crawlers, llms.txt exists and follows the spec, canonical URLs are per-page, the sitemap is valid and complete, and every sampled page has an H1 followed by a clean H2/H3 hierarchy.

## Warnings

1. **Generic `og:image:alt` and `twitter:image:alt` on blog posts.** Every page returns `"Soleur - Stop Hiring. Start Delegating."` even when the OG image itself is post-specific (e.g. `/images/blog/og-what-is-company-as-a-service.png`). Social previews and AI summarizers lose the descriptive alt text that should match the per-post image. Severity: low.
2. **Homepage `og:title` is just "Soleur".** The `<title>` is `Soleur -- AI Agents for Solo Founders | Every Department, One Platform`, but `og:title` and `twitter:title` are the bare brand name. Social shares of the homepage will under-perform vs. the SERP listing. Severity: low.
3. **No `<meta name="author">` tag.** Author identity is present in BlogPosting JSON-LD (Person with jobTitle, sameAs, knowsAbout) and in visible byline + author card, which is sufficient for Google and AI engines. A `<meta name="author">` would add a redundant signal for legacy parsers but is optional. Severity: informational.
4. **CWV ratings are source-heuristic only.** No Lighthouse/PSI run was performed. The source shows excellent practices (inlined critical CSS with a screenshot-gate enforcement comment, `font-display: swap` on both webfonts, async stylesheet load via `onload` swap, no large JS bundle on the critical path), but real LCP/INP/CLS values are not measured. Severity: informational.
5. **`og:type` is `website` on `/about/`.** For a single-person profile page, `profile` is more semantically correct, though `website` is acceptable and Google does not penalize either. Severity: trivial.

## Passed Checks

- **JSON-LD coverage:** homepage emits `WebSite`, `WebPage`, `Organization` (with `sameAs` to GitHub/X/LinkedIn/Bluesky/Discord and `subjectOf` -> Inc.com NewsArticle), `SoftwareApplication` with two `Offer` objects, plus a `FAQPage`. Blog posts emit `WebPage` + `BlogPosting` (with full `Person` author including `jobTitle`, `description`, `image`, `sameAs`, `knowsAbout`) + `FAQPage`. About page emits 3 JSON-LD blocks.
- **No JS-injected structured data.** All JSON-LD is present in the initial HTML response.
- **Canonical URLs:** every sampled page has a self-referential `<link rel="canonical">` matching its sitemap URL.
- **Open Graph:** `og:title`, `og:description`, `og:url`, `og:type` (`article` on posts, `website` elsewhere), `og:site_name`, `og:locale` (`en_US`), `og:image` with explicit `width`/`height`/`alt` on all sampled pages.
- **Twitter cards:** `summary_large_image` with title, description, image (dimensions), `twitter:site` (`@soleur_ai`) and `twitter:creator` set globally.
- **robots.txt:** `User-agent: *` + `Allow: /` + `Sitemap: https://www.soleur.ai/sitemap.xml`. No `Disallow` rules anywhere. GPTBot, PerplexityBot, ClaudeBot, Google-Extended, CCBot, OAI-SearchBot all permitted by inheritance from `*`.
- **llms.txt:** present at `/llms.txt`, follows the spec (H1 title, `>` blockquote tagline, descriptive paragraph, `## Use Cases`, `## Docs` with 7 link entries, `## Optional` with 2 entries). Every linked URL is in the sitemap.
- **Sitemap.xml:** 49 entries, all with `lastmod`, no non-HTML resources, well-formed XML, blog posts and core pages both included. Sitemap URL declared in robots.txt.
- **E-E-A-T:** visible byline (`by Jean Deruelle, Founder of Soleur` with `rel="author"`), `<time datetime="...">` element, author card section with photo, JSON-LD `Person` includes credentials ("15+ years in distributed systems", "Founder of Soleur"). About page exists at `/about/` and is referenced by `Organization.founder` and every `BlogPosting.author.@id`. Legal/privacy/contact pages all in sitemap.
- **Heading hierarchy:** single `<h1>` per page on home, about, and blog post; H2 -> H3 nesting respected; no skipped levels.
- **Descriptive link text:** zero occurrences of "click here", "read more", or "learn more" across the four sampled pages.
- **Technical SEO:** HTTPS only, `<meta charset="UTF-8">`, `<meta name="viewport">`, RSS/Atom feed declared via `<link rel="alternate" type="application/atom+xml" href="/blog/feed.xml">`, strict CSP, favicon set. Home URL returns HTTP 200 with zero redirects (no apex-to-www trampoline added by Cloudflare).
- **Core Web Vitals (source heuristics):** critical CSS inlined for `/`, `/blog/*`, and `.page-hero` templates with a code comment pointing at `screenshot-gate.mjs` enforcement; full stylesheet loads asynchronously via `onload` swap; both webfonts use `font-display: swap`; one `rel="preload"` directive present; no synchronous third-party scripts; Plausible analytics is the only external connect-src.

## Recommendations

Prioritized by impact-to-effort ratio. Every change is small and reversible.

### P1 -- Polish social previews (10 min)

**Fix homepage `og:title` / `twitter:title`.** Make them match the `<title>` tag (or a short variant) so shared homepage links carry the value prop, not just the brand name.

- File: `apps/site/src/_includes/base.njk` (or equivalent layout). Look for the block that emits `og:title` when `page.url == "/"` or when no per-page `title` override exists. Default the social title to the same string used in `<title>`.

**Make `og:image:alt` and `twitter:image:alt` per-page on blog posts.** Today every post returns the generic `"Soleur - Stop Hiring. Start Delegating."` even when `og:image` is post-specific. Source the alt from the post's front-matter (e.g. `ogImageAlt`) and fall back to the global string only when no override is supplied.

- Files: blog post front-matter under `apps/site/src/blog/*.md` (add `ogImageAlt:`), and the head partial that emits the two `*:image:alt` meta tags.

### P2 -- Verify with field measurement (15 min, external)

Run Lighthouse (mobile + desktop) and PSI against `/`, `/pricing/`, `/about/`, and a representative blog post. Confirm LCP, INP, CLS each clear the "Good" thresholds. If they do, raise this audit's CWV sub-score from 4 to 5 and the headline score to 98. If not, the most likely culprits given the source are (a) a hero image without `width`/`height` on a yet-unsampled page, or (b) a third-party form embed (Buttondown) shifting layout in the footer newsletter section.

### P3 -- Minor schema refinements (5 min, optional)

- Change `og:type` on `/about/` from `website` to `profile`. Add `og:profile:first_name`, `og:profile:last_name`, `og:profile:username` for the founder profile page.
- Consider adding a `Person` JSON-LD block at `/about/#jean-deruelle` so the `@id` references that already exist on the homepage Organization and every BlogPosting resolve to a canonical Person definition. (If this already exists in one of the three JSON-LD blocks on `/about/`, no action needed -- the audit did not enumerate every block on that page.)

### P4 -- Defensive add (2 min, optional)

Add an explicit `User-agent: GPTBot`, `User-agent: PerplexityBot`, `User-agent: ClaudeBot`, `User-agent: Google-Extended`, `User-agent: OAI-SearchBot`, `User-agent: CCBot` block to `/robots.txt`, each with `Allow: /`. The current wildcard already permits them by inheritance, but explicit allow-listing protects against a future operator inadvertently flipping the wildcard to `Disallow: /` for crawl-budget reasons and forgetting that AI bots would be collateral damage. Pure belt-and-braces; not required.

## What was NOT audited

- Image alt coverage on internal content images (the home page has zero `<img>` tags -- the design is SVG/CSS -- so alt coverage was not testable from the home sample).
- Real-user CWV (no Lighthouse/PSI/CrUX data pulled).
- Backlink profile, domain authority, indexed page count in Google Search Console.
- Internal-link graph density (the audit checked link text quality but did not crawl the link graph).

A focused follow-up that runs Lighthouse on the four representative URLs and spot-checks alt text on three blog posts would be enough to convert this audit into a fully-verified A+.
