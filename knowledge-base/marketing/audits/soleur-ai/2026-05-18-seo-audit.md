## Critical Issues

1. **Origin is unreachable — Cloudflare returns HTTP 526 on every HTML and XML route.** Verified against:
   - `GET https://soleur.ai/` → 526
   - `GET https://www.soleur.ai/` → 526
   - `GET https://soleur.ai/about` → 526
   - `GET https://soleur.ai/sitemap.xml` → 526
   - `GET https://www.soleur.ai/sitemap.xml` → 526
   - `GET https://soleur.ai/llms.txt` → 526
   - `GET https://www.soleur.ai/llms.txt` → 526
   - `GET https://soleur.ai/index.html` → 526

   Tested with multiple user agents (default curl, generic Mozilla, Chrome desktop, Googlebot) and HTTP/2 + HTTP/1.1. All requests reach Cloudflare (`server: cloudflare`, `cf-ray` present) but Cloudflare cannot validate the TLS certificate on the origin server. Response body in every case is the literal string `error code: 526`.

   **Impact:** Googlebot, Bingbot, GPTBot, ClaudeBot, and PerplexityBot will all receive 526 for every URL in the sitemap. Repeated 5xx responses trigger soft de-indexing in Google Search; AI crawlers will fail to refresh any cached content. If this state has persisted for more than ~24 hours, expect ranking drops within days.

2. **`robots.txt` Sitemap directive points at a host that is also down.** `https://soleur.ai/robots.txt` (200 OK, served from Cloudflare edge) declares:
   ```
   User-agent: *
   Allow: /
   Sitemap: https://www.soleur.ai/sitemap.xml
   ```
   `https://www.soleur.ai/sitemap.xml` itself returns 526. Crawlers that pick up the robots.txt cannot then reach the sitemap, compounding the discoverability hit.

3. **Cannot verify any on-page SEO/AEO signal.** Because no HTML was retrievable, this audit cannot confirm the current state of: structured data (JSON-LD blocks), canonical URLs, Open Graph and Twitter Card tags, page titles and descriptions, heading hierarchy, author/E-E-A-T markup, llms.txt content, sitemap completeness, or Core Web Vitals indicators. The previous audit (2026-05-04) found all of these in excellent shape (97/100), but that finding cannot be re-confirmed today.

## Warnings

- **Apex/www host ambiguity (carried over from 2026-05-04).** Cloudflare is the public-facing edge for both `soleur.ai` and `www.soleur.ai` and both return 526, suggesting either a shared origin or symmetric misconfiguration. Whoever fixes the 526 should also resolve the apex vs www canonical decision called out in the prior audit, because the fix for one likely touches the same edge configuration as the other.
- **Audit tooling cannot distinguish "origin offline" from "origin TLS expired/misconfigured" remotely.** Both produce 526. Operator must inspect the origin host directly (cert chain, expiry date, Cloudflare origin pull rules, port 443 reachability) — `curl` from outside Cloudflare cannot disambiguate.

## Passed Checks

- **`robots.txt` is reachable and permissive.** `User-agent: *` with `Allow: /` is the correct posture for AI/SEO discoverability. (However, the referenced sitemap host is currently down — see Critical Issue 2.)
- **HTTPS, HSTS, and the Cloudflare edge are healthy.** TLS 1.3 handshake to the edge completes (`TLS_AES_256_GCM_SHA384`); `cf-ray` and `server: cloudflare` indicate the request reached Cloudflare. The failure is strictly origin-side.

## Recommendations

Listed in priority order. Items 1 and 2 are the only items that matter today; items 3+ are unchanged from the 2026-05-04 audit and should be picked up after restoration.

1. **Restore the origin (P0).** Investigate why Cloudflare cannot reach or validate the origin:
   - Confirm the origin server is up and listening on the port Cloudflare expects (commonly 443 with full-strict, or 80 with full).
   - Check the origin TLS certificate: not expired, hostname matches what Cloudflare's origin pull is configured for, full chain present.
   - Verify Cloudflare's SSL/TLS mode (Full vs Full (strict) vs Flexible). A recent switch to "Full (strict)" against an origin with a self-signed or hostname-mismatched cert will produce 526.
   - Check `infra/` Terraform and `seo_page_redirects` ruleset (per recent commit `54ce580`) for any change that may have altered origin pull config or HTTPS upgrade behaviour.
   - If origin is healthy and reachable from the Cloudflare data centre but 526 persists, raise a Cloudflare support ticket with the `cf-ray` IDs captured here (e.g., `9fdadca78aea5d0d-IAD`).
2. **Once HTTP 200 is restored, re-run this audit immediately.** The 2026-05-04 audit fingerprinted 42 sitemap entries at 97/A; this run cannot verify those numbers held. A confirmation pass with the previous audit's URL set will catch any incidental regressions introduced by the 526 fix.
3. **Submit the sitemap to Google Search Console as soon as the origin is back** and check the Coverage report 24–48 hours later for any URLs newly flagged as `Server error (5xx)`. Request re-indexing on the most important pages (`/`, `/blog/`, top-trafficked blog posts) to shorten the recovery window.
4. **Add origin-uptime monitoring with alerting on first 5xx**, if not already in place. A 526 outage is silently catastrophic for SEO — there is no visible UI breakage for an internal user who is whitelisted past Cloudflare, so the failure mode only manifests in crawler logs and rankings days later.
5. **Carry-over from 2026-05-04 (defer until origin is restored):**
   - Pick one canonical host (apex or www) and align canonical / og:url / sitemap / JSON-LD / llms.txt.
   - Pass post-specific OG image into `BlogPosting.image` JSON-LD field.
   - Reshape `Person.knowsAbout` to topical areas, not job titles.
   - Drop non-standard `twitter:image:width` / `twitter:image:height`.
   - Add explicit `User-agent: GPTBot/ClaudeBot/PerplexityBot/Google-Extended` Allow blocks in robots.txt.
   - Confirm `screenshot-gate.mjs` wired into deploy workflow per `cq-eleventy-critical-css-screenshot-gate`.
   - Run Lighthouse/PSI lab pass and save next to audit for CWV trend tracking.
