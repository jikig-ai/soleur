### Critical Issues

None. No blockers to search indexing or AI-engine discoverability were observed on the sampled pages.

### Warnings

- **W1 — Render-blocking stylesheet with redundant preload (CWV / LCP).**
  `base.njk` emits both `<link rel="preload" href="css/style.css" as="style">` and, on the
  next line, `<link rel="stylesheet" href="css/style.css">`. The preload does not help
  LCP because the stylesheet tag that immediately follows already makes the resource
  a render-blocking high-priority discovery. The single CSS bundle is ~43 KB and is the
  only render-blocking resource in the critical path; either inline critical CSS or drop
  the redundant preload. Affects every page.

- **W2 — BlogPosting `image` is the generic site OG image.**
  `_includes/blog-post.njk` hard-codes `"image": site.url + "/images/og-image.png"` for
  every BlogPosting. Google's Article rich-result guidelines recommend a post-specific
  hero image (≥1200 px wide, relevant to the article) for eligibility in Top Stories /
  Discover. Current behavior will not disqualify the schema, but it will not unlock
  image-enhanced rich results either.

- **W3 — `og:title` on the homepage is bare "Soleur".**
  Production HTML on `/` shows `<meta property="og:title" content="Soleur">` because the
  template suppresses the " - {{ site.name }}" suffix when `title == site.name`. This is
  correct logic but produces a weaker social-share headline than the `<title>` element
  ("Soleur - Stop Hiring. Start Delegating."). Consider `seoTitle`-style override for the
  homepage OG title so it matches the page `<title>`.

- **W4 — Apex-to-www 301 with canonical pointing to apex.**
  `curl -I https://soleur.ai/` returns `301 → https://www.soleur.ai/`, but the canonical
  inside the rendered page is `https://soleur.ai/` (apex). Both hosts ultimately serve
  the same content, but the canonical-vs-final-URL mismatch plus an extra hop is
  suboptimal. Decide on one canonical host and align: either redirect www → apex, or
  change canonicals (and `site.url` in `_data/site.json`) to the www host.

- **W5 — E-E-A-T: author expertise signals are thin on the post body itself.**
  The BlogPosting JSON-LD includes `author.jobTitle` ("Founder of Soleur") and the visible
  byline links to `/about/`, but posts do not render an inline author card
  (photo, 1–2-sentence bio, credentials, link to other posts by the author). This is
  the highest-leverage remaining E-E-A-T gap for Google's helpful-content signals.

- **W6 — `<base href="/">` with relative asset paths is brittle.**
  `base.njk` emits `<base href="/">` and then references `css/style.css`, `images/favicon.png`,
  etc. as relative URLs. It works today because `<base>` rewrites them to absolute, but it
  couples every subdirectory page's asset loading to the `<base>` tag and makes any future
  migration to a sub-path deploy (docs.soleur.ai/plugin, staging previews) silently
  reroute all assets. Prefer absolute root-relative paths (`/css/style.css`).

  Organization (with `sameAs` to GitHub/X/LinkedIn/Bluesky/Discord and `subjectOf`

  Spark tiers, `softwareVersion: 3.55.1`, `operatingSystem`, `applicationCategory`).

### Recommendations

Priority order. Each recommendation is non-destructive — no changes have been
applied per the audit directive.

1. **Remove the redundant CSS preload (W1).** In
   `plugins/soleur/docs/_includes/base.njk`, delete the line
   `<link rel="preload" href="css/style.css" as="style">`. The `<link rel="stylesheet">`
   that follows already blocks render on the same URL. This is zero-cost and
   marginally improves LCP. If the goal was to accelerate CSS discovery, the correct
   pattern is to move the preload into the HTTP response headers (Cloudflare Workers
   or `Link:` header) so it arrives before the document body parses.

2. **Add per-post hero images and wire into BlogPosting (W2).** Extend
   `plugins/soleur/docs/_includes/blog-post.njk` to accept an optional `heroImage`
   front-matter value and prefer it over the site OG fallback for the `"image"` JSON-LD
   field. Even populating hero images on the top 5 traffic posts will unlock Article
   rich-result eligibility on those URLs.

3. **Embed an inline author card on blog posts (W5).** Add a small component after
   the post body: author photo, 60–80-word bio pulling from `site.author`, link to
   `/about/`, and a "More posts by this author" link. The data is already in
   `_data/site.json`; this is a template-only change. This is the single highest
   E-E-A-T lever for the 10-post blog corpus.

4. **Align canonical host with redirect target (W4).** Decide whether apex or www
   is canonical. If apex is canonical (current `site.url`), change the edge redirect
   to `www → apex`. If www is canonical, update `_data/site.json` `url` field to
   `https://www.soleur.ai` — all canonicals, sitemap `<loc>`s, JSON-LD `url`
   properties, and llms.txt links flow from that single source.

5. **Override the homepage `og:title` (W3).** Introduce a `seoOgTitle` or reuse
   `seoTitle` when set, and on `/index.njk` set a value like
   `"Soleur — Stop Hiring. Start Delegating."` so social unfurls match the
   `<title>` and the hero pitch. One-line template change in `base.njk`.

6. **Replace `<base href="/">` with absolute root-relative asset paths (W6).**
   Change `href="css/style.css"` → `href="/css/style.css"` (and similarly for
   `images/favicon.png`, the header `href="index.html"`, and the footer logo link).
   Remove `<base href="/">`. This removes a fragile dependency and future-proofs
   sub-path deployments.

7. **Run a live Lighthouse / PageSpeed Insights pass.** Source-level heuristics
   cannot produce LCP/INP/CLS numbers; a real-device run will confirm the W1 impact
   and surface any network-waterfall issues the static audit cannot see.

8. **Consider adding `aggregateRating` to the homepage SoftwareApplication schema**
   once first-party ratings (GitHub stars, G2, Product Hunt) exceed a minimum volume
   and are representative. Current schema is valid without it; this is an upgrade,
   not a fix.
