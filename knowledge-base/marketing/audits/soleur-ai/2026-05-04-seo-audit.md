## Warnings

1. **Apex → www 301 redirect chain conflicts with apex canonical.** Every fetched URL on `https://soleur.ai/...` returns `HTTP/2 301` to `https://www.soleur.ai/...`, but the canonical link, `og:url`, sitemap entries, and JSON-LD `url` properties all point to the apex (`https://soleur.ai/`). When Googlebot follows the redirect and lands on `www`, the served HTML still claims the apex as canonical. Google generally accepts this, but it adds one redirect hop to every crawl, splits internal-link equity, and creates an inconsistency between the served host and the declared canonical host. Decide on one host (apex or www) and align: choose host → emit canonical/og:url/sitemap/JSON-LD with that host → 301 the other host to it.
2. **Critical-CSS path depends on JavaScript for the stylesheet swap.** `_includes/base.njk` uses an inline `<script>` to flip `link[rel="preload"]` to `rel="stylesheet"` after the inlined critical CSS paints. There is a `<noscript>` fallback that loads the full stylesheet synchronously, but for JS-disabled or JS-failed clients the FOUC window is still controlled by `<noscript>` rather than the swap. AGENTS.md rule `cq-eleventy-critical-css-screenshot-gate` exists precisely because this mechanism has shipped FOUC bugs twice. Confirm `screenshot-gate.mjs` runs on every deploy and that all above-the-fold selectors used by the new pages remain in the inlined critical block.
3. **`twitter:image:width` / `twitter:image:height` are non-standard.** The X (Twitter) card spec only defines `twitter:image:alt`. Width/height are ignored. Harmless but noise — the OG width/height fields cover what crawlers need.
4. **Blog post `image` in `BlogPosting` JSON-LD points to the generic OG image, not the post-specific OG image.** On `/blog/best-claude-code-plugins-2026/`, the OG meta correctly references `og-best-claude-code-plugins-2026.png`, but the `BlogPosting.image` field uses the generic `images/og-image.png`. Search results and AI summaries pulling from JSON-LD will get the wrong image. Have the post template pass the same image variable to both the OG meta block and the JSON-LD `image` field.
5. **`Person.knowsAbout` contains role/biography strings, not topical knowledge areas.** The values `"Founder, Soleur"` and `"15+ years in distributed systems"` belong in `jobTitle` / `description`. `knowsAbout` should hold subject areas (e.g., `"Distributed Systems"`, `"Developer Tools"`, `"AI Agents"`, `"Claude Code"`). This is the field LLMs and Knowledge Graph use to disambiguate expertise.

## Passed Checks

- **Structured Data.** Five distinct schema types are present across the sampled pages and all render in static HTML (no JS injection): `WebSite`, `WebPage`, `Organization` (with `sameAs` and `subjectOf` referencing the Inc. article), `SoftwareApplication` (with two `Offer` entries), `FAQPage`, `BlogPosting` (with `author` as `Person`, `publisher`, `datePublished`, `dateModified`, `mainEntityOfPage`), and `ProfilePage` on `/about/`. The `SoftwareApplication` schema is correctly scoped to the homepage only. JSON-LD blocks parse as valid JSON and use canonical schema.org types.
- **Meta Tags.** `<link rel="canonical">`, `og:title`, `og:description`, `og:url`, `og:type` (`website` for static pages, `article` for blog posts), `og:site_name`, `og:locale` (`en_US`), `og:image` with width/height/alt, and the full `twitter:*` set are present on every sampled page. Titles and descriptions are page-specific.
- **AI Discoverability.** `/llms.txt` exists, follows the spec (H1 title, blockquote tagline, intro paragraph, "Use Cases" section, "Docs" section with annotated links, "Optional" section), and is reachable. `/robots.txt` declares `User-agent: *` with `Allow: /` and references the sitemap. GPTBot and ClaudeBot fetch the homepage with HTTP 200. All audited pages are server-rendered HTML — no client-side-only content.
- **E-E-A-T.** Blog posts include a visible author byline (`<span class="blog-post-author">by <a rel="author" href="/about/">Jean Deruelle</a>, Founder of Soleur</span>`), an author card with photo, role, bio, and credentials list, and an `Article` JSON-LD with full `Person` author. `/about/` has a `ProfilePage` schema with `Person` mainEntity and `sameAs` social links. `datePublished` and `dateModified` are present in JSON-LD. Trust signals are complete: privacy policy, terms, GDPR policy, cookie policy, disclaimer, and acceptable-use policy are all in the sitemap. The post under review describes its evaluation methodology.
- **Sitemap.** 42 URLs, every entry has a `lastmod` date, no non-HTML resources, no `eleventyExcludeFromCollections` leakage. Blog posts retain their original publication `lastmod` (e.g., 2026-03-10), while listing/index pages reflect today's build (2026-05-04) — the right pattern.
- **Content Quality.** Each sampled page has exactly one `<h1>`, h2/h3 hierarchy is clean (no skips), no "click here" / "read more" / bare-URL link text was found, all pages declare `lang="en-US"`, the author card section uses `aria-label="About the author"`, and there is a "skip to main content" link.
- **Technical SEO.** HTTPS with `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`, `X-Content-Type-Options: nosniff`, charset and viewport meta present, RSS/Atom feed (`/blog/feed.xml`) auto-discoverable via `<link rel="alternate">`, favicon present, content-type with `charset=utf-8`. CSP is strict (script-src self + Plausible + three SHA-256 hashes; no `unsafe-inline` for scripts).
- **Core Web Vitals (source-level indicators).** Critical CSS is inlined in `<head>` for first paint. The full stylesheet uses preload + post-load rel-swap to avoid render-blocking. Plausible analytics loads with `async`. Fonts use `font-display: swap` with `woff2`. The homepage uses no `<img>` tags above the fold — visuals are CSS/SVG-based, eliminating CLS from missing dimensions. No heavy client-side framework bundle was observed.

## Recommendations

Listed in priority order. None are blockers.

1. **Pick one canonical host (apex or www) and align everything.** Decision is operational, not technical. Recommended: keep apex `soleur.ai` (matches branding and current canonical) and either (a) configure Cloudflare/host so `soleur.ai/...` returns 200 directly without redirecting to `www`, or (b) switch to `www.soleur.ai` everywhere — canonical, og:url, sitemap, JSON-LD `url`, llms.txt link bodies. Option (a) is cheaper.
2. **Pass the post-specific OG image into `BlogPosting.image`.** In the post layout, replace the hardcoded `https://soleur.ai/images/og-image.png` with the same variable that drives `og:image`. This will give Google Discover, AI summaries, and social previews pulling JSON-LD the correct article image.
3. **Reshape `Person.knowsAbout` on the BlogPosting and ProfilePage author schemas.** Move "Founder, Soleur" into `jobTitle` (already set) and biography text into `description`. Populate `knowsAbout` with topical areas — e.g., `["Distributed systems", "Developer tools", "AI agents", "Claude Code", "Solo founders"]`. This is the field that grounds LLM responses about Jean's domain expertise.
4. **Drop `twitter:image:width` and `twitter:image:height`.** They are not part of the X (Twitter) Card spec. Keep `twitter:image:alt`. Cosmetic cleanup.
5. **Add an explicit `User-agent: GPTBot` / `User-agent: ClaudeBot` / `User-agent: PerplexityBot` / `User-agent: Google-Extended` block in robots.txt with `Allow: /`.** Functionally redundant given the wildcard rule (which is why this is a "low" recommendation, not a warning), but explicit allows make the intent unambiguous and survive future copy-paste changes to the wildcard rule.
6. **Confirm `screenshot-gate.mjs` is wired into `deploy-docs.yml` and runs on every deploy.** AGENTS.md rule `cq-eleventy-critical-css-screenshot-gate` documents two prior FOUC incidents on this site. Verify the gate runs against the live preview build after every PR, not only on a periodic schedule.
7. **Run a Lighthouse / PageSpeed Insights pass against `/`, `/blog/`, and one long blog post to confirm field LCP/INP/CLS scores.** Source-level indicators look strong, but only a lab run will tell you the actual numbers. Save results next to this audit for trend tracking.
8. **Consider adding `BreadcrumbList` JSON-LD on the blog post layout.** Posts already nest under `/blog/` and the breadcrumb would surface in SERPs. Skip if you want to keep schemas minimal — site is flat enough that this is optional.
