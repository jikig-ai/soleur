## Critical Issues (P0)

## Warnings (P1)

1. **No resource hints for third-party origins** — `plausible.io` and `buttondown.com` are allowed in CSP and loaded at runtime, but no `<link rel="preconnect">` or `dns-prefetch` is emitted. On first paint this adds DNS+TLS latency (~100-300 ms on cold connections), nudging LCP upward. Location: `<head>` of every page.
2. **CSS loaded render-blocking without `media="print"` swap** — `<link rel="preload" href="css/style.css" as="style">` is followed immediately by a synchronous `<link rel="stylesheet" href="css/style.css">`. The preload is redundant (browser already has the stylesheet in the critical path) and adds a useless `Link:` header without unblocking render. Consider either (a) inlining critical CSS + deferring the rest, or (b) removing the redundant preload.
3. **Blog post OG image falls back to default** on pages that define a post-specific image — `/blog/ai-agents-for-solo-founders/` emits `og:image=https://soleur.ai/images/blog/og-ai-agents-for-solo-founders.png` (good), but `og:image:alt` is the generic "Soleur - Stop Hiring. Start Delegating." rather than a post-specific alt. Minor social-card quality hit.
4. **BlogPosting JSON-LD `image` field uses generic OG image**, not the post-specific image — the meta tag has `og-ai-agents-for-solo-founders.png` but the JSON-LD `image` property is `https://soleur.ai/images/og-image.png`. Google treats this as a mismatch and may downgrade rich-result eligibility.
5. **Homepage `og:title` is just "Soleur"** — truncates the value proposition on LinkedIn/Twitter previews. The `<title>` is the full "Soleur — AI Agents for Solo Founders | Every Department, One Platform"; `og:title` should match or at least include "AI Agents for Solo Founders".
6. **No `<meta name="robots">` directive** emitted. Default indexing behavior is fine, but explicit `index, follow, max-image-preview:large` unlocks Google Discover large-image treatment.
7. **No `hreflang` even though `og:locale=en_US` is declared** — non-blocking for a single-locale site, but if multilingual expansion is planned, wire now.

## Recommendations

Prioritized by impact × effort.

1. **P1 — Align BlogPosting JSON-LD `image` with the post-specific OG image** (5 min per template). File: Eleventy blog post layout. Change the `"image"` property in `BlogPosting` JSON-LD to use the same post-slug-based URL as `og:image`. Unblocks Google rich-result `image` requirement for article cards.
2. **P1 — Expand homepage `og:title`** to include the value proposition, e.g. "Soleur — AI Agents for Solo Founders". File: homepage template head.
3. **P2 — Add `<link rel="preconnect" href="https://plausible.io" crossorigin>`** and `<link rel="dns-prefetch">` for `buttondown.com` in `<head>`. Saves 100-300 ms on LCP for first-time visitors.
4. **P2 — Remove the redundant `<link rel="preload" ... as="style">`** for the synchronously-loaded stylesheet, or convert to the media-swap pattern (`media="print" onload="this.media='all'"`) if the goal is to defer. Currently it only adds a `Link:` header without effect.
5. **P2 — Add explicit `<meta name="robots" content="index, follow, max-image-preview:large, max-snippet:-1">`** on content pages to unlock Google Discover large-image treatment.
6. **P3 — Post-specific `og:image:alt`** per blog post (not the generic site tagline). File: blog layout — derive from post title.
7. **P3 — If any blog post adds images**, enforce `width`, `height`, `loading="lazy"`, and `decoding="async"` via a Nunjucks shortcode to prevent CLS/LCP regression.
8. **P3 — Consider adding `dateModified` to the sitemap** — it currently has `lastmod` (good), but verify the Eleventy filter writes the real build time, not today's date for every URL (all non-blog entries show `2026-04-19`, suggesting it's using build date rather than page-modified date; this dilutes signal to crawlers).

## Notes on Methodology

- Live HTTP fetches with `curl -sL` (follows redirects to defeat Cloudflare Bot Fight Mode). All sampled pages returned HTTP 200 with full content in the initial response (no JS-required rendering).
- Core Web Vitals scored at 3/5 on source-level heuristics only (redundant preload, no preconnect). Recommend running Lighthouse/PageSpeed Insights for lab scores. No evidence of heavy JS bundles or render-blocking scripts — the source-level indicators suggest real-world scores are likely already strong.
- Structured data was not run through `validator.schema.org` or Google's Rich Results Test as part of this audit. Manual inspection found all required properties present and correctly typed.
- Sample size: 3 HTML pages of 36. A full crawl of the remaining 33 (legal, changelog, agents, skills, getting-started, pricing, vision, community) would be needed to rule out per-template regressions, though the shared base template visible in sampled pages suggests consistent SEO scaffolding.
