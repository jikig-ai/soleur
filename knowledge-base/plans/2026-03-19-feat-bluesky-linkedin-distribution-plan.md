# feat: Add Bluesky and LinkedIn Company Page as Automated Distribution Channels

## Overview

Extend the content publishing pipeline to post to Bluesky and LinkedIn company page automatically on scheduled publish dates. Both platforms already have API wrapper scripts (`bsky-community.sh`, `linkedin-community.sh`) and content generation templates (partial). This feature wires them into the automated pipeline end-to-end: content generation, content publishing, CI workflow secrets, and tests.

## Problem Statement / Motivation

The content pipeline currently publishes to Discord and X/Twitter only (`channels: discord, x`). LinkedIn personal and company page sections are generated by `social-distribute` but only published manually. Bluesky has an API wrapper script but no content generation template and no publishing integration. Every manual posting step is a context switch for the solo founder.

- **Bluesky:** Growing developer/indie community. API wrapper exists (`bsky-community.sh`), UTM tracking configured in SKILL.md, but zero integration in the publisher or content generator.
- **LinkedIn company page:** Content section exists in SKILL.md (Phase 5.7), test fixtures have `## LinkedIn Company Page` sections, and `content-publisher.sh` already dispatches `linkedin-company` to `post_linkedin()`. However, the underlying `post_linkedin()` calls `linkedin-community.sh` which posts as a **person** (uses `LINKEDIN_PERSON_URN`). Company page posting needs an **organization URN** author.

## Proposed Solution

### Architecture

Five files change, zero new files created. The existing graceful-skip pattern (skip with warning when credentials are missing, create fallback GitHub issue on failure) is replicated for both platforms.

```

                                  Update Phase 6, 9, 10 for Bluesky

.github/workflows/scheduled-content-publisher.yml
                               -- Pass BSKY_* and LINKEDIN_ORG_ID secrets
test/content-publisher.test.ts -- Add bluesky and linkedin-company tests

```

#### Phase 2: content-publisher.sh -- LinkedIn Company Page (Separate Function)

The existing `post_linkedin()` function (lines 327-349) calls `linkedin-community.sh post-content --text "$content"` which uses `LINKEDIN_PERSON_URN` as the author. For company page posting, the author must be `urn:li:organization:{id}` (set via `LINKEDIN_ORG_ID`).

**Decision point:** Two approaches:

**Option A (Recommended): Add `--author` flag to `linkedin-community.sh`**

Add an optional `--author` parameter to `cmd_post_content()` in `linkedin-community.sh` that overrides the default `LINKEDIN_PERSON_URN` author. The content-publisher calls:
- Personal: `bash "$LINKEDIN_SCRIPT" post-content --text "$content"` (existing, uses LINKEDIN_PERSON_URN)
- Company: `bash "$LINKEDIN_SCRIPT" post-content --text "$content" --author "urn:li:organization:${LINKEDIN_ORG_ID}"` (new)

This is minimal, backward-compatible, and lets the same script handle both cases.

**Option B: Separate `post_linkedin_company()` function that calls linkedin-community.sh differently**

Duplicate the function with a different author. More code, same outcome.

**Implementation (Option A):**

**2.1 Update `linkedin-community.sh` `cmd_post_content()`** to accept `--author` flag

--author)
  author_override="${2:-}"

**2.2 Add `post_linkedin_company()` to `content-publisher.sh`** (~after `post_linkedin`, line 349)

```bash
post_linkedin_company() {
  local file="$1"

  if [[ -z "${LINKEDIN_ACCESS_TOKEN:-}" ]]; then
    echo "Warning: LINKEDIN_ACCESS_TOKEN not set. Skipping LinkedIn Company Page posting." >&2
    return 0
  fi

  if [[ -z "${LINKEDIN_ORG_ID:-}" ]]; then
    echo "Warning: LINKEDIN_ORG_ID not set. Skipping LinkedIn Company Page posting." >&2
    return 0
  fi

  local content
  content=$(extract_section "$file" "LinkedIn Company Page")
  if [[ -z "$content" ]]; then
    echo "Warning: No LinkedIn Company Page content found in $(basename "$file"). Skipping." >&2
    return 0
  fi

  bash "$LINKEDIN_SCRIPT" post-content --text "$content" --author "urn:li:organization:${LINKEDIN_ORG_ID}" || {
    echo "Error: LinkedIn Company Page posting failed. Creating fallback issue." >&2
    create_linkedin_fallback_issue "$file" "LinkedIn Company Page"
    return 1
  }
  echo "[ok] LinkedIn Company Page post published."
}
```

## Test Scenarios

### Bluesky

- Given `BSKY_HANDLE` and `BSKY_APP_PASSWORD` are unset, when content-publisher runs with `channels: bluesky`, then it prints a warning and exits 0 (graceful skip)
- Given valid Bluesky credentials, when content-publisher runs with `channels: bluesky` and the content file has a `## Bluesky` section, then it calls `bsky-community.sh post` with the section content
- Given Bluesky posting fails, when content-publisher runs, then a fallback GitHub issue is created with the content for manual posting
- Given `bsky-community.sh` does not exist, when `BSKY_HANDLE` is set, then content-publisher exits 1 with error

### LinkedIn Company Page

- Given `LINKEDIN_ACCESS_TOKEN` is unset, when content-publisher runs with `channels: linkedin-company`, then it prints a warning and exits 0
- Given `LINKEDIN_ORG_ID` is unset (but `LINKEDIN_ACCESS_TOKEN` is set), when content-publisher runs with `channels: linkedin-company`, then it prints a warning and exits 0
- Given valid LinkedIn credentials and org ID, when content-publisher runs with `channels: linkedin-company`, then it calls `linkedin-community.sh post-content --text "$content" --author "urn:li:organization:${LINKEDIN_ORG_ID}"`
- Given LinkedIn company posting fails, when content-publisher runs, then a fallback issue is created

### Backward Compatibility

- Given existing content files with `channels: discord, x`, when content-publisher runs, then Discord and X work exactly as before
- Given `linkedin-personal` channel, when content-publisher runs, then personal posting still uses `LINKEDIN_PERSON_URN` (no regression from adding `--author` flag)

## Dependencies & Risks

### Dependencies

- **Bluesky app password:** Must be created in Bluesky account settings and added as GitHub secret
- **LinkedIn org ID:** Must be obtained from LinkedIn company page admin settings and added as GitHub secret
- **LinkedIn API permissions:** Company page posting may require Marketing API access or Community Management API access with the `w_organization_social` permission. The existing `linkedin-community.sh` uses the basic Posts API (`/rest/posts`) which supports organization authors IF the token has the right scopes. Verify the current OAuth token has `w_organization_social` scope.

### Risks

| Risk | Mitigation |
|------|-----------|
| LinkedIn token lacks org posting scope | Graceful skip pattern; fallback issue created. Document the required scope. |
| Bluesky rate limits (1667 posts/day, 11.5/minute for creates) | Single post per content file; rate limit buffer (sleep 5) between files already exists. |
| Existing content files only have `channels: discord, x` | New channels are additive; old files won't post to Bluesky/LinkedIn until their frontmatter is updated. Only newly generated content gets the expanded channels. |
| LinkedIn API version changes (current: 202602) | Versioned in `linkedin-community.sh` header; same risk as existing personal posting. |
