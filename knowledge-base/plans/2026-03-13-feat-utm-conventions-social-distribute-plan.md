# feat: UTM conventions and social-distribute integration

Add platform-specific UTM parameters to all article URLs generated by the social-distribute skill. Plausible Analytics reads UTM query strings natively — no analytics-side changes needed. This enables per-channel traffic attribution in the Sources > Campaigns report.

## Acceptance Criteria

- [ ] social-distribute SKILL.md adds UTM mapping table (Phase 3) and per-platform URL instructions (Phase 5)
- [ ] Each platform section gets its own `utm_source` and `utm_medium` values per the convention table
- [ ] Reddit gets `utm_source=reddit` only (no medium or campaign) to avoid spam filter risk
- [ ] Slug derived from blog post URL path (e.g., `/blog/caas-pillar/` → `caas-pillar`)
- [ ] 4 scheduled distribution-content files (02-05) updated with UTM-tagged URLs
- [ ] UTM conventions documented in `knowledge-base/marketing/content-strategy.md`
- [ ] Regeneration is idempotent — UTMs derived from source path, never from existing content file URLs
- [ ] Existing content-publisher.sh tests pass

## Test Scenarios

- Given a blog post at `/blog/case-study-operations-management/`, when social-distribute generates Discord content, then the URL includes `?utm_source=discord&utm_medium=community&utm_campaign=case-study-operations-management`
- Given a blog post, when social-distribute generates Reddit content, then the URL includes only `?utm_source=reddit` (no medium or campaign anywhere in the Reddit section)
- Given a blog post, when social-distribute generates X/Twitter content, then the URL includes `?utm_source=x&utm_medium=social&utm_campaign=<slug>`
- Given a blog post, when social-distribute generates LinkedIn content, then the URL includes `?utm_source=linkedin&utm_medium=social&utm_campaign=<slug>`
- Given content-publisher.sh publishes a UTM-tagged content file, then URLs are posted verbatim (no double-encoding or stripping)

### Key Architecture

social-distribute is a **prompt-driven skill** (SKILL.md only, no scripts). Phase 3 constructs a single base URL from `site.url` + blog path. Phase 5 generates per-platform content variants. The UTM mapping table goes in Phase 3 (reference) and per-platform URL construction happens in Phase 5.

content-publisher.sh extracts content sections **verbatim** and posts them. No changes needed.

## MVP

### social-distribute SKILL.md changes (pseudo)

**Phase 3 addition** — after constructing the base article URL:

```markdown
**UTM Tracking:** Derive the campaign slug from the article URL path (strip `/blog/`
prefix and trailing `/`). Example: `/blog/caas-pillar/` → `caas-pillar`.

Use this mapping to construct platform-specific tracked URLs:

| Platform | Tracked URL |
|----------|------------|
| Discord | `<base-url>?utm_source=discord&utm_medium=community&utm_campaign=<slug>` |
| X/Twitter | `<base-url>?utm_source=x&utm_medium=social&utm_campaign=<slug>` |
| IndieHackers | `<base-url>?utm_source=indiehackers&utm_medium=community&utm_campaign=<slug>` |
| Reddit | `<base-url>?utm_source=reddit` |
| Hacker News | `<base-url>?utm_source=hackernews&utm_medium=community&utm_campaign=<slug>` |
| LinkedIn | `<base-url>?utm_source=linkedin&utm_medium=social&utm_campaign=<slug>` |
| Bluesky | `<base-url>?utm_source=bluesky&utm_medium=social&utm_campaign=<slug>` |

If the derived slug contains characters other than a-z, 0-9, hyphens, or underscores,
replace them with hyphens.
```

**Phase 5 update** — each platform variant section uses its platform-specific tracked URL instead of the bare base URL. Every occurrence of the article URL within a section gets that platform's UTM parameters.

**Discord note:** UTM-tagged URLs are ~60-80 chars longer. Generate content with the tracked URL so character count is accurate within the 2000-char limit.
