# fix(observability): refresh model.c4 cron-monitor count 49 → 50 (and 6 → 7 CI check-ins)

🐛 **Bug** — a stale factual count in the `github -> sentry` C4 edge description reds the
`deploy-script-tests` CI job, disarming every later-registered infra guard in that `bash -e`
sequential job.

## Overview

`apps/web-platform/infra/sentry/cron-monitors.tf` now declares **50** `sentry_cron_monitor`
resources (verified: `grep -c '^resource "sentry_cron_monitor"' … = 50`). The 50th, added by
commit `2726466dd` (#6549 item 2, merged via #6632), is
`sentry_cron_monitor.scheduled_heartbeat_reconcile` — a GHA-workflow-fired heartbeat that
checks in from CI ("here"), **not** from webapp.

The `github -> sentry` edge description in `knowledge-base/engineering/architecture/diagrams/model.c4`
(line 451) still reads **"Of 49 cron monitors, 6 check in from here and 43 from webapp"**. Two
numbers are stale:

- `Of 49 cron monitors` → `Of 50 cron monitors`
- `6 check in from here` → `7 check in from here`
- `43 from webapp` — **UNCHANGED** (the new monitor is CI-fired, not webapp-fired). 7 + 43 = 50 ✓

This staleness makes the cross-file assertion in
`apps/web-platform/infra/supabase-advisor/scan-workflow.test.sh` (lines 519–523) FAIL
(reproduced: `c4_count=49` vs `tf_count=50`). That `*.test.sh` runs sequentially under `bash -e`
in the `deploy-script-tests` job (`.github/workflows/infra-validation.yml`), so the `exit 1`
halts the whole job and disarms every later infra guard.
