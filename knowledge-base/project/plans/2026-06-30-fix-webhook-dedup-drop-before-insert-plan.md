## Overview

`app/api/webhooks/github/route.ts` writes one `INSERT INTO processed_github_events(delivery_id)`
for **every** delivery at "Step 3" (`route.ts:154-156`), BEFORE the body is parsed and BEFORE all
drop-filters. Most deliveries produce no side effect — the dominant case is `workflow_run` events
whose `conclusion !== 'failure'` — yet each still writes (and migration 094 later auto-deletes) a
useless dedup row. Per `pg_stat_statements.wal_bytes` on the production project
(`ifsccnjhymdmidffkzhl`), this single INSERT is **63% of the database's total WAL** — the dominant
Disk-IO-budget consumer, the warning that prompted this work.

This change **reorders** the dedup INSERT so it fires ONLY for deliveries that will actually
dispatch to Inngest. The INSERT moves from "first thing after header checks" to "immediately before
each `inngest.send` / push-reconcile dispatch," after every drop-filter. All drop paths return their
existing status (200 `{received:true}`, or the existing 4xx for malformed/missing headers) WITHOUT
writing a dedup row.

This is the **source-side** complement to the 2026-06-02 retention work
(`knowledge-base/project/plans/2026-06-02-fix-supabase-disk-io-recurrence-and-sentry-monitor-plan.md`,
migration `094_dedup_tables_retention.sql`): retention bounds the table's *row count* by pruning old
rows; it does nothing about the *WAL* generated by writing every row in the first place. WAL is
per-write, not per-retained-row, so the only lever on the 63% is to stop writing the rows that never
mattered. **Do not change migration 094 or the GitHub App manifest subscriptions** — all 6 subscribed
events stay handled; the win is purely the drop-before-dedup reorder.

### Why this is correct (not a wrong-layer repeat)

Per `hr-no-dashboard-eyeball-pull-data-yourself` and the recurring-symptom Sharp Edge: the WAL
attribution is from `pg_stat_statements.wal_bytes` on the live prod project — the INSERT statement is
the measured #1 WAL writer, so the fix layer (the route's INSERT) is the verified-correct layer, not
a guess. The prior plan (094) measured the same table as the "#3 Disk-IO write consumer" at 65,240
live rows; bounding row count did not bound WAL, which is why the symptom recurred at the budget
level. Reducing INSERT *frequency* is the untried lever.

## User-Brand Impact

**If this lands broken, the user experiences:** a dropped or double-processed GitHub webhook. The
webhook is the sole ingress for two per-user signals — CI-failure drafts (`engineering.ci_failed`)
and workspace-reconcile-on-push. A reorder bug that drops the dedup INSERT on a dispatching event
would let a GitHub redelivery double-dispatch (duplicate CI-failure draft / duplicate reconcile run);
a reorder bug that writes the row but fails to release on dispatch failure would silently drop the
redelivery (the user never gets their CI-failure signal).

**If this leaks, the user's data is exposed via:** no new exposure surface. The dedup row holds only
`delivery_id` (opaque GitHub UUID, no PII). rawBody redaction (`redactGithubSourcedText`) on the
Inngest dispatch path is unchanged.

**Brand-survival threshold:** single-user incident — a single user's dropped/double-processed event
is a per-user data-integrity incident. (CPO sign-off required at plan time; `user-impact-reviewer`
runs at review time.)

**2a. Move the dedup INSERT out of "Step 3" into a reusable closure** defined after

```ts
// Claim the delivery_id dedup row immediately before dispatch. Plain
// .insert() — NO ON CONFLICT (supabase-js returns data:null not [] on no-op,
// so the affected-row gate is unreliable; catch PG_UNIQUE_VIOLATION instead,
// mirroring the Stripe webhook). Returns a Response to return early
// (200 replay / 500 db-error), or null to proceed to dispatch.
async function claimDedupRow(): Promise<NextResponse | null> {
  const { error: dedupErr } = await supabase
    .from("processed_github_events")
    .insert({ delivery_id: deliveryId });
  if (!dedupErr) return null;
  if (dedupErr.code === PG_UNIQUE_VIOLATION) {
    logger.info({ deliveryId, githubEvent }, "GitHub webhook replay — delivery_id already processed, skipping");
    return NextResponse.json({ received: true });
  }
  logger.error({ err: dedupErr, deliveryId }, "GitHub webhook dedup insert failed — returning 500");
  Sentry.captureException(dedupErr, {
    tags: { feature: "github-webhook", op: "dedup-insert" },
    extra: { deliveryId, githubEvent },
  });
  return NextResponse.json({ error: "DB error" }, { status: 500 });
}
```

subscribed event types that the handler *acts on* — every Stripe event type is handled, none is a
high-frequency guaranteed no-op — and at very low volume (the 094 retention test notes
`processed_stripe_events` holds ~1 row today). The GitHub route receives a high-volume stream whose
*dominant* case (`workflow_run` `conclusion !== 'failure'`) produces no side effect — that no-op
INSERT is the measured 63% of prod WAL. Stripe has no equivalent no-op case, so dedup-first costs
Stripe nothing; for GitHub it is the entire budget problem. (A reader who notices Stripe FR5's Inngest
dispatch would otherwise see an apparent contradiction — the PR body + ADR-036 amendment must cite the
volume/no-op-ratio reason explicitly.) The reorder keeps the two Stripe idioms that protect correctness
(unique-violation replay + release-on-dispatch-failure) and drops only the property that has no value
on a no-side-effect path (deduping an event that will never be dispatched). The route's header comment
(`:1-31`) must be updated so the "mirrors Stripe" claim is scoped to those two idioms, not the ordering
(Phase 2j).

## Risks & Mitigations

### Precedent-diff — Stripe webhook dedup ordering (Phase 4.4)

The route's header (`route.ts:3-5`) and Phase 2 say it "mirrors the canonical Stripe webhook." Verified
against the sibling precedent:

| Aspect | Stripe (`webhooks/stripe/route.ts`) | GitHub (after this change) |
| --- | --- | --- |
| Dedup INSERT position | dedup-first — `:118-120`, before the `switch` (`:162`) | drop-before-dedup — immediately before each `inngest.send` |
| `PG_UNIQUE_VIOLATION`→200 replay idiom | `:122-129` | preserved (`claimDedupRow`) — **parity kept** |
| `releaseDedupRow()` on handler 5xx | `:145-160`, called on every post-INSERT error path | preserved on both dispatch-failure paths — **parity kept** |
| Why the ordering can diverge | every Stripe event is actioned by the `switch`; volume is ~1 row/day (094 test) → no no-op WAL waste | GitHub has a dominant no-op high-volume case (`workflow_run` non-failure) = 63% of prod WAL |

**Conclusion:** the divergence is deliberate and scoped — the load-bearing parity (the unique-violation
replay idiom + release-on-dispatch-failure + **dedup strictly before dispatch**) is fully preserved; only
the dedup-before-*drop-filters* ordering changes, and drops have no side effect to dedup against. Phase 2j
MUST rewrite the `:3-5` header comment so it no longer claims ordering parity (scope the parity claim to
the two idioms). A future maintainer reading "mirrors Stripe ordering" against the new code would
otherwise see a false contradiction.

### Risks

- **Risk:** the dedup INSERT slips AFTER `inngest.send` during the move, opening a double-dispatch window on concurrent redeliveries. **Mitigation:** AC4 (PG_UNIQUE_VIOLATION → 200, both paths) + the `claimDedupRow()` closure is invoked on the line immediately before each dispatch; reviewer verifies INSERT precedes `sendInngestWithRetry` in both branches.
- **Risk:** a drop path is missed and still inserts (no WAL reduction) — or a dispatch path is missed and never inserts (double-dispatch). **Mitigation:** Phase 1 adds explicit `mockInsert` (not-)called assertions to every drop and every dispatch branch across all three suites.
- **Risk:** removing `releaseDedupRow()` from a path that DID write a row. **Mitigation:** after the reorder, the only INSERT sites are the two `claimDedupRow()` calls; the only paths reachable after them are the two dispatch try/catches (which keep `releaseDedupRow()`). Every removed `releaseDedupRow()` is provably on a pre-INSERT path. Inverted test assertions (`:189/:207/:263/:357/:361/:407`) lock this in.
- **Risk:** stale comments mislead a future maintainer (the file's load-bearing ordering doc). **Mitigation:** Phase 2j + 2d + 2e update the header block and inline comments.
- **Noted behavioral change (not a defect; data-integrity-guardian sub-P1):** because the early dedup is gone, a redelivery of an already-dispatched non-push event now re-runs founder resolution. If the founder became `ambiguous` (standing-state drift) between the original delivery and the redelivery, the redelivery hits the `founder-ambiguous` Sentry PAGE + 404 instead of the old `23505`→200 short-circuit. This is added paging noise on a rare drift state, NOT a double-dispatch (the original row still blocks any new INSERT, and the ambiguous branch returns before `claimDedupRow`). No code change; recorded so the on-call reader is not surprised.
- **Crash-window note (data-integrity-guardian sub-P1):** a process death between INSERT-success and `inngest.send` still strands a redelivery (row written, not dispatched). This window pre-existed the change and the reorder *shrinks* it (INSERT now adjacent to dispatch). Not a regression; no action.
