# Plan: frontend-anti-slop skill (Hallmark adaptation for React/Next.js)

Closes #4264.

**Plan-review note:** v1 scope was cut by the 3-reviewer panel (DHH + Kieran + code-simplicity) — Tier 2 LLM-judgment reviewer agent + macrostructure of pre-wired review-pipeline integration are **deferred to v1.5**. v1 ships a deterministic Tier 1 scanner only, invoked manually or via a minimal `soleur:review` hook that runs the scanner and surfaces findings inline. The Tier 2 design (gate split, reviewer agent shape) is documented in `## v1.5 Roadmap (Documented, Not Built)` so v2 can pick it up without re-design.

## Overview

Build a new Soleur-native skill `frontend-anti-slop` that adapts the React-applicable subset of [Nutlope/hallmark](https://github.com/Nutlope/hallmark) (MIT, by Together AI) into a deterministic audit pass over `apps/web-platform/{app,components}/**/*.{tsx,jsx,css}`. v1 ships a Bun-runnable scanner emitting structured findings; the existing `ux-audit` `FINDING_CATEGORIES` pipeline is extended with one `anti-slop` category to flow scanner findings through the established dedup + caps + GitHub-issue infra. `soleur:review` gets a minimal conditional block that runs the scanner inline on PRs touching frontend code (no separate review agent in v1).

Ships v1 opt-in + `dry_run=true` default; auto-filing gated on calibration signal (≤ 10% operator-confirmed false-positive rate on ≥ 20 findings over ≥ 2 weeks). MIT attribution lands at the canonical root sites (`plugins/soleur/NOTICE` + `LICENSES/hallmark.MIT.txt`), not in a per-skill NOTICE file. The "Powered by Together AI" footer is stripped from the skill body per spec TR5.

## Risks

- **False-positive cliff on Tier 1 rules.** Generic-font detection can flag legitimate Inter use (some brands require it). Mitigation: per-rule disable comment `<!-- anti-slop:disable RULE_ID reason="..." -->` in source files; scanner respects.
- **Selector regex relaxation may admit malformed selectors elsewhere.** The regex `^[A-Za-z0-9_./#:\-]*$` is wider than the CSS-selector-only form. Mitigation: existing 5 categories never emit findings with `#` or `:` in selectors (CSS selectors don't need them), so the relaxation is backward-compatible. Validation: positive fixture asserts a CSS-selector finding still passes; negative fixture asserts a malformed selector (e.g., `<script>`) still rejects.
- **`soleur:review` description budget for new stanza.** Stanza adds words to the body, not `description:` — should not affect skill description budget. Verify at Phase 5 second-pass via the same `bun test components.test.ts`.
- **Calibration signal is operator-subjective in v1.** ≤ 10% FP rate requires manual operator review without a labeling instrument. Mitigation: v1.5 adds `--label` flag; v1 captures findings in a follow-up issue thread for ad-hoc review.
- **Hash collision risk under selector overload.** Same `<file>#<rule_id>` from two different routes hashes to two different entries (route is part of the hash) — no collision. Inverse: same `<file>#<rule_id>` on the same route (re-fired finding) dedups correctly. Verified by the existing 3-tuple hash.
