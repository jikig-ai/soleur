### New considerations discovered

- **DSAR solo-subject correctness:** a DSAR is per-subject (`expectedUserId`); a member's *personal*
  data lives in their solo workspace, NOT a shared workspace they belong to (shared-workspace files
  are the owner's data, in the owner's DSAR). So `workspacePathForWorkspaceId(expectedUserId)`
  (N2 solo) is the correct resolver choice — NOT `resolveActiveWorkspacePath` (which would resolve a
  member's *active* shared workspace and over-export the owner's files into the member's DSAR — a
  cross-tenant leak). This is the single most important resolver-selection call in the plan.

## Overview

PR #5003 fixed the **read** side of the KB-share path after the ADR-044 `users → workspaces`
relocation: shared documents 404'd because the read path resolved a KB root from the owner's
stale/empty `users.workspace_path` column instead of the share's `workspace_id`. Issue #5005 is
the tracker that PR #5003's PIR referenced — sweep the **remaining** direct readers of the
legacy `users.workspace_path` / `users.workspace_status` columns and converge the genuine
latent-bug readers onto the workspace-id-keyed resolver
(`resolveActiveWorkspacePath` / `resolveActiveWorkspaceKbRoot` / `resolveWorkspacePathForUser`).

**This is a deliberately narrow refactor, not a column decommission.** The legacy columns are
NOT being removed — `workspace_status` is still the readiness source-of-truth *inside* the
resolver (step 4 of `resolveActiveWorkspaceKbRoot` reads the active-workspace OWNER's
`users.workspace_status`), and the provision/lifecycle WRITES of both columns remain. Scope is:
**read paths that resolve a path or a readiness gate from the CALLER's own legacy `users` row**,
which is stale/empty for (a) invited members viewing a non-solo active workspace and (b) any
account provisioned after the ADR-044 relocation.

No new dependencies. No schema/migration changes. No infrastructure. Test runner is `vitest`
(`apps/web-platform/test/**/*.test.ts`, node project — confirmed `apps/web-platform/vitest.config.ts:44`).

## Resolver selection per reader (verified against `workspace-resolver.ts`)

The resolver exposes three relevant primitives; choosing the right one per reader is load-bearing:

- `resolveActiveWorkspacePath(userId, supabase) → string` (`:339`) — membership-aware **active**
  workspace path. For active-workspace concerns (attachments, vision write on the active KB).
- `resolveActiveWorkspaceKbRoot(userId, supabase) → ActiveWorkspaceKbAccess` (`:350`) — full
  readiness/connectivity-gated access (404/503 discriminated). For readers that ALSO need the
  readiness + connectivity gate (kb/sync). Pair with `resolveActiveWorkspaceRepoMeta` for the
  installation id, exactly as `kb-route-helpers.ts:98-112` and `kb/upload/route.ts:77-90` do.
- `resolveWorkspacePathForUser(userId, supabase) → string` (`:653`) — the user's **default/solo**
  workspace path. For per-user solo concerns where the active-workspace claim is irrelevant (DSAR
  export of the subject's OWN data). `workspacePathForWorkspaceId(userId)` (`:668`) is the pure
  N2-equivalent when the subject's `workspace_id == user_id` is already in hand.

## User-Brand Impact

**If this lands broken, the user experiences:** for an account provisioned after the ADR-044
relocation (the expected state for any recent signup) or an invited member operating in a shared
workspace — a silent failure on a core surface: "Sync now" returns `409 Workspace not connected`
on a connected workspace; `vision.md` first-run creation 503s; chat attachments silently never
persist to disk; and (highest stakes) a **GDPR DSAR export is silently missing the subject's
workspace files** — an incomplete Art. 15/20 right-of-access response.

**If this leaks, the user's data/workflow is exposed via:** the converse risk — a resolver that
fails OPEN to a *sibling* workspace would expose cross-tenant files. The resolver's active-id
resolution fails CLOSED to the SOLO workspace (never a sibling; `workspace-resolver.ts:297-325`),
and `workspace_id` is server-stored, not attacker-controllable (security-sentinel confirmed in PR
#5003). Each converged reader inherits this IDOR-safe fail-closed behavior; the regression test
asserts it (divergent-id case must NOT resolve to the owner's stale path).

**Brand-survival threshold:** single-user incident — this is the exact latent-bug class that
produced the #5003 dogfood incident; a single post-relocation user tripping any of these 5 surfaces
is a brand-survival event. `requires_cpo_signoff: true`; CPO sign-off required at plan time;
`user-impact-reviewer` runs at review time (review skill conditional-agent block).

## Risks & Mitigations

- **Risk: behavioral drift in kb/sync response codes.** The resolver discriminates 404/503; the
  legacy route returned 409 "Workspace not connected" / "WORKSPACE_NOT_READY". *Mitigation:* read
  the `KbSyncStatus` client discrimination before changing codes; preserve the existing contract by
  message-string mapping (the `kb-route-helpers.ts:94-103` precedent), or update the client in
  lockstep with an explicit AC.
- **Risk: resolver's readiness gate is stricter/looser than the old own-row gate.** The resolver
  gates on the active-workspace OWNER's `workspace_status` (not the caller's). For solo callers
  (owner === caller, N2) this is byte-identical; for members it is the *correct* gate. *Mitigation:*
  divergent-id tests cover both solo and member cases.
- **Risk: `vision` 503 semantics change** (resolver always returns a path). *Mitigation:* gate
  "not provisioned" on a downstream FS check or drop it; document the chosen behavior in the PR.
- **Precedent diff (deepen-plan Phase 4.4 will expand):** the already-migrated
  `kb-route-helpers.ts` / `kb/upload/route.ts` / `workspace-reconcile-on-push.ts` are the canonical
  precedent for each resolver-selection choice — diff each converted reader against the closest
  precedent rather than inventing a new shape.

## Sharp Edges

- **`repo/status/route.ts` reads BOTH column families from `users`.** Beyond `workspace_path`
  (in scope), it reads `repo_url`/`repo_status`/`repo_last_synced_at` from `users` (`:25,:36-37`) —
  the ADR-044-relocated **repo** columns. Those are covered by ADR-044's own pre-decommission
  *drift gate* and are OUT OF SCOPE for #5005 (which the PIR explicitly notes covers only
  `workspace_path`/`workspace_status`). Converging only `workspace_path` here leaves an *internal
  inconsistency* (one column family resolved id-keyed, the other still own-row). Acceptable for this
  focused PR; note the adjacency in the PR body and consider a follow-up issue if the repo-column
  reads are also stale post-relocation.
- **`workspace_status` is NOT decommissionable by this PR.** The resolver itself reads the active
  workspace OWNER's `users.workspace_status` as the readiness source of truth
  (`workspace-resolver.ts:413-429`). This refactor converges *caller-keyed* reads onto the resolver;
  it does not eliminate the column. Any AC implying "no more `workspace_status` reads anywhere" is
  wrong — scope to "no caller-keyed own-row readiness/path reads in the 5 converged files."
- **Resolver-selection mismatch is the dominant correctness risk.** `resolveActiveWorkspacePath`
  (active) vs `resolveWorkspacePathForUser`/`workspacePathForWorkspaceId` (solo/default) resolve
  *different* workspaces for a member with a non-solo active claim. DSAR is per-subject (solo);
  vision/attachment/sync are active-workspace. Picking the wrong one is a silent wrong-workspace bug
  that passes a naive solo-only test — the divergent-id test MUST include a member-with-non-solo-active
  fixture for the active-workspace readers.
- **A plan whose `## User-Brand Impact` section is empty, contains only TBD/TODO/placeholder text,
  or omits the threshold will fail `deepen-plan` Phase 4.6.** This section is filled
  (threshold: single-user incident).
- **Divergent-id fixture must model the production `<WORKSPACES_ROOT>/<id>` nested layout**, not a
  flat temp dir — per the #5003 learning §Session Errors note 3, test fixtures that modeled the OLD
  flat `users.workspace_path` source must move to the id-keyed layout
  (`WORKSPACES_ROOT=dirname(tmpWorkspace)`, derive `workspace_id` from the basename).

## Test Scenarios (per converged reader)

For each of the 5 readers, the divergent-id matrix:

1. **Solo, healthy** — `workspace_id == user_id`, own `users.workspace_path` populated → byte-identical
   path; reader succeeds (backward-compat baseline).
2. **Solo, post-relocation** — `workspace_id == user_id`, own `users.workspace_path` empty/stale →
   resolver still resolves the correct id-keyed path; reader succeeds (the bug being fixed).
3. **Member, non-solo active** (active-workspace readers: kb/sync, attachment, vision) — caller is a
   member of a workspace they don't own; own `users` row is the empty solo row → reader resolves the
   ACTIVE workspace, not the empty own-row, and not a sibling (fail-closed-to-solo verified for a
   non-member claim).
4. **Regression guard** — a fixture where `workspace_id ≠ basename(legacy workspace_path)` so any
   re-read of the legacy column resolves the WRONG dir and the test fails.
