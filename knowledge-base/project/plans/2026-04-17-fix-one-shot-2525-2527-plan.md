### New Considerations Discovered

- **PreToolUse hook JSON response vs exit code.** Claude Code PreToolUse hooks have two mechanisms: (a) JSON `hookSpecificOutput.permissionDecision: "deny" | "allow" | "ask"` with reason (preferred, structured), (b) plain stderr + non-zero exit (works, but less structured). The in-repo `worktree-write-guard.sh` uses (a). The plan now prescribes (a) for consistency.
- **Python on PATH assumption.** The hook uses `#!/usr/bin/env python3`. Verify Python 3 is on PATH inside the Claude Code runtime environment. If the runtime is a minimal Docker image, this may not be true — but for local dev + CI + Codespaces, Python 3 is standard. Document the dependency in the hook header.
- **Rescue path for false negatives.** Because the hook scope is narrow (only `new_string` content), an edit that removes context around a pre-existing sink could leave the sink orphaned without firing the hook. This is acceptable — adding a sink is the dangerous action; preserving one that was already present is out of scope for a "recently introduced" advisory.
- **Cloudflare HSTS dashboard-managed state.** Historically the 1y HSTS was set via dashboard. `cloudflare_zone_settings_override` takes ownership on first apply — no import needed, but the first plan may show a diff from the current value (expected).
- **HSTS preload submission is a separate manual step.** The max-age bump alone does NOT add `app.soleur.ai` to the Chrome preload list. Once `max-age=63072000` is live, the domain becomes eligible for submission at <https://hstspreload.org/>, but submission is a one-time action the user must perform. This is explicitly scoped out; the issue's acceptance criterion is "preload-list eligible", not "preload-list submitted".

## Implementation Phases

### Phase A — `security_reminder_hook.py` (Issue #2525)

**Goal:** Add a Python PreToolUse hook that only advises when the diff introduces known workflow-injection sinks inside a `run:` block.

#### A.1 Hook implementation

**File to create:** `.claude/hooks/security_reminder_hook.py`

**Behavior contract:**

1. Read the Claude Code hook payload from stdin (JSON).
2. Extract `tool_name`, `tool_input.file_path`, `tool_input.new_string`.
3. If `tool_name` is not `Edit` → exit 0 (no-op, should not fire given matcher but defense-in-depth).
4. If `file_path` does not match `.github/workflows/*.yml` or `.github/workflows/*.yaml` → exit 0.
5. Parse `new_string` for:
   - The presence of at least one `run:` directive (YAML key at any indentation), OR the content appears to be inside a `run:` block (we scan the full `new_string`; for a precise check we require the injection sink AND the string `run:` on a line somewhere in the new_string, OR the sink appears inline in a shell-looking context — the simpler rule is: sink present in `new_string` AND `new_string` contains `run:` anywhere).
   - At least one of the known injection sinks:
     - `${{ github.event.issue.title }}`
     - `${{ github.event.issue.body }}`
     - `${{ github.event.pull_request.title }}`
     - `${{ github.event.pull_request.body }}`
     - `${{ github.event.comment.body }}`
     - `${{ github.event.review.body }}`
     - `${{ github.event.review_comment.body }}`
     - `${{ github.event.pages.*.page_name }}` (wildcard — match via regex `github\.event\.pages\.[^}]+\.page_name`)
     - `${{ github.event.commits.*.message }}` (wildcard)
     - `${{ github.event.head_commit.message }}`
     - `${{ github.event.head_commit.author.email }}`
     - `${{ github.event.head_commit.author.name }}`
     - `${{ github.event.commits.*.author.email }}` (wildcard)
     - `${{ github.event.commits.*.author.name }}` (wildcard)
     - `${{ github.event.pull_request.head.ref }}`
     - `${{ github.event.pull_request.head.label }}`
     - `${{ github.event.pull_request.head.repo.default_branch }}`
     - `${{ github.head_ref }}`
6. If BOTH conditions hold → write a Claude Code PreToolUse **JSON response** to stdout and `exit 0`. The JSON payload uses `hookSpecificOutput.permissionDecision: "deny"` with a `permissionDecisionReason` containing the advisory text. This matches the in-repo `worktree-write-guard.sh` pattern and is the structured mechanism Claude Code surfaces as a permission decision to the model.
7. Otherwise → `exit 0` silently with no stdout.

**Why JSON-over-stdout and not `exit 2`:** Claude Code PreToolUse hooks support both (a) stderr + non-zero exit, and (b) stdout JSON with `hookSpecificOutput.permissionDecision`. The in-repo convention (see `.claude/hooks/worktree-write-guard.sh`) uses (b) because it produces a structured, machine-readable decision. Staying consistent keeps hook behavior predictable across the suite.

**Advisory content (embedded in `permissionDecisionReason`):**

```text
Workflow-injection risk detected in new_string:
  - sink: <matched sink literal>
  - file: <file_path>

Untrusted ${{ github.event.* }} fields are interpolated into the shell verbatim.
Mitigation: assign the field to an env var first, then reference $VAR inside the
run: script. See:
  https://docs.github.com/en/actions/security-guides/security-hardening-for-github-actions#using-an-intermediate-environment-variable

If this is intentional and already safe (e.g., the value is pinned to a known
allow-list upstream), acknowledge and re-run the Edit to proceed.
```

**Concrete JSON output (stdout, `exit 0`):**

```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Workflow-injection risk detected in new_string:\n  - sink: ${{ github.event.issue.title }}\n  - file: .github/workflows/example.yml\n\nUntrusted ${{ github.event.* }} fields are interpolated into the shell verbatim.\nMitigation: assign the field to an env var first, then reference $VAR inside the run: script. See:\n  https://docs.github.com/en/actions/security-guides/security-hardening-for-github-actions#using-an-intermediate-environment-variable\n\nIf this is intentional and already safe, acknowledge and re-run the Edit to proceed."
  }
}
```

**Key implementation notes:**

- Use Python **stdlib only** (no `pyyaml` — existing hooks are pure bash with `jq`, and adding a Python dep would break if the runtime image strips it). Use `re` and `json`.
- Verify Python 3 is on PATH: `command -v python3`. In the worktree, `python3 --version` should return ≥3.9. The hook header will assert via shebang; if Python is missing at runtime, the hook silently fails (exit 127) which is safe because Claude Code treats unreadable hooks as no-op.
- Match sinks with a pre-compiled regex list. Literal patterns for non-wildcard sinks (ANDed with `new_string in <literal>`); regex for the `.*.` wildcards:
  - `r"github\.event\.commits\[\d+\]\.message"` — index access form
  - `r"github\.event\.commits\.[^}]+\.message"` — path-access wildcard form
  - Repeat pattern for `.author.(email|name)` and `.pages.*.page_name`
- Keep `run:` detection cheap: `re.search(r"^\s*run:", new_string, re.MULTILINE)`. This admits false positives where `run:` appears in a comment, but a comment-only `run:` together with a sink is still worth advising on (user is likely about to uncomment it).
- Header: `#!/usr/bin/env python3`. Set executable bit (`chmod +x`).
- Prose-rule cross-reference comment at top of file (matches `guardrails.sh` convention):

  ```python
  # PreToolUse hook: advise on GitHub Actions workflow-injection sinks.
  # Fires only when the Edit's new_string introduces a known untrusted
  # `${{ github.event.* }}` interpolation AND the file has a `run:` block.
  #
  # Corresponding prose rule:
  #   security_reminder_hook:workflow-injection-advisory —
  #     AGENTS.md hr-in-github-actions-run-blocks-never-use (indirect),
  #     https://docs.github.com/en/actions/security-guides/security-hardening-for-github-actions
  #
  # Response contract:
  #   - Safe edit   -> exit 0, no stdout  (allow by default)
  #   - Risky edit  -> exit 0, stdout JSON {hookSpecificOutput.permissionDecision: "deny", ...}
  #   - Not an Edit -> exit 0, no stdout  (matcher should prevent this, defense-in-depth)
  ```

**Error handling:**

- If stdin is empty or not valid JSON → `exit 0` silently. A broken hook must never block an edit it cannot analyze.
- If `tool_input.new_string` is missing → `exit 0`. Same reasoning.
- Wrap the main body in `try/except` that logs nothing on failure and exits 0. Silent fail-open is the correct posture for an **advisory** hook.

#### A.3 Unit tests

**File to create:** `.claude/hooks/security_reminder_hook.test.sh`

Use the existing repo convention: Bash-based `.test.sh` files that drive the hook via stdin-piped JSON payloads. Reference pattern: `apps/web-platform/infra/disk-monitor.test.sh` (PASS/FAIL/TOTAL counter, subshell-isolated cases, stdin JSON fixtures).

**Assertion helpers:**

```bash
assert_allow() {
  local name="$1" payload="$2"
  local out exit_code
  out=$(printf '%s' "$payload" | "$HOOK" 2>&1)
  exit_code=$?
  if [[ $exit_code -eq 0 && -z "$out" ]]; then
    echo "PASS: $name"; ((PASS++))
  else
    echo "FAIL: $name (exit=$exit_code, out=$out)"; ((FAIL++))
  fi
  ((TOTAL++))
}

assert_deny() {
  local name="$1" payload="$2" expected_sink="$3"
  local out exit_code decision reason
  out=$(printf '%s' "$payload" | "$HOOK" 2>/dev/null)
  exit_code=$?
  decision=$(printf '%s' "$out" | jq -r '.hookSpecificOutput.permissionDecision // ""')
  reason=$(printf '%s' "$out" | jq -r '.hookSpecificOutput.permissionDecisionReason // ""')
  if [[ $exit_code -eq 0 && "$decision" == "deny" && "$reason" == *"$expected_sink"* ]]; then
    echo "PASS: $name"; ((PASS++))
  else
    echo "FAIL: $name (exit=$exit_code, decision=$decision, reason=$reason)"; ((FAIL++))
  fi
  ((TOTAL++))
}
```

**Minimum cases (acceptance criteria mapping):**

1. **Benign env-only edit → allow.** Payload: Edit on `.github/workflows/web-platform-release.yml`, `new_string` contains `STATUS_POLL_MAX_ATTEMPTS: 60` and a comment line — no `${{ github.event.* }}`, no `run:` with a sink. `assert_allow`.
2. **Sink introduced in run block → deny.** Payload: `new_string` contains `run: |\n  echo "${{ github.event.issue.title }}"`. `assert_deny "github.event.issue.title"`.
3. **Whitespace-only edit → allow.** `new_string` is just spaces/newlines. `assert_allow`.
4. **Comment-only edit → allow.** `new_string` contains only `# comment lines` with no sinks. `assert_allow`.
5. **Non-workflow file → allow.** Payload: Edit on `apps/web-platform/lib/security-headers.ts` with `new_string` that (contrived) contains `${{ github.event.issue.title }}` as a string literal. Hook should not fire. `assert_allow`.
6. **Wildcard sink `commits[0].message` → deny.** Payload: `new_string` contains `${{ github.event.commits[0].message }}` and a `run:` line. `assert_deny "commits"`.
7. **Wildcard sink `pages[0].page_name` → deny.** Payload: `new_string` contains `${{ github.event.pages[0].page_name }}` and a `run:` line. `assert_deny "page_name"`.
8. **Sink present but no `run:` in new_string → allow.** Payload: `new_string` contains the sink inside `env:` (`TITLE: ${{ github.event.issue.title }}`), no `run:`. This is the **safe** pattern the advisory recommends, so the hook must NOT fire. `assert_allow`. (Documented limitation: a separate later Edit that adds `run: echo $TITLE` would still be safe as long as the sink is not re-added to `new_string`.)
9. **Non-Edit tool (matcher defense-in-depth) → allow.** Payload with `tool_name: "Write"`. `assert_allow`.
10. **`.github/actions/*` composite action edit with sink → allow (known limitation).** Payload: Edit on `.github/actions/notify-ops-email/action.yml` with a `run:` containing `${{ github.event.issue.title }}`. The hook scope (per issue #2525) is `.github/workflows/*.yml` only. Composite actions are out of scope for this iteration. Document with `assert_allow` and a comment.
11. **Malformed JSON on stdin → allow (fail-open).** Send `not-json` as stdin. `assert_allow`. Fail-open is the correct posture for an advisory hook.

**Test runner glue:** shell script runs each case, pipes fixture JSON to the hook via `printf '%s' "$payload" | "$HOOK"`, captures stdout + exit code, asserts against the `hookSpecificOutput.permissionDecision` field using `jq`. Print PASS/FAIL per case, print a `PASS=$PASS FAIL=$FAIL TOTAL=$TOTAL` summary, exit 1 iff `FAIL > 0`.

**Fixtures:** inline heredocs in the test script (not separate files) — keeps the test self-contained. Use JSON via `printf` with `\n` escapes, or here-strings with careful quoting.

**Preflight in the test script:**

- Check `command -v python3 >/dev/null` at the top; skip with a clear message + exit 0 if missing (so CI environments without Python 3 don't false-fail).
- Check `command -v jq >/dev/null`; jq is already a dep of other hooks, so this should always pass — still worth asserting.

### Phase B — Cloudflare HSTS alignment (Issue #2527)

**Goal:** Align Cloudflare edge HSTS `max-age` with origin (63072000s / 2y) via Terraform.

#### B.1 Terraform resource

**File to create:** `apps/web-platform/infra/cloudflare-settings.tf`

Using the `cloudflare` v4 provider already pinned in `main.tf`:

```hcl
# Cloudflare zone-level security settings managed via Terraform.
#
# HSTS must match the origin value defined in
# apps/web-platform/lib/security-headers.ts (max-age=63072000, 2 years).
# The edge previously rewrote origin HSTS down to 1 year via dashboard
# configuration, breaking preload-list eligibility (min 63072000).
# See #2527 for context.

resource "cloudflare_zone_settings_override" "soleur_ai" {
  zone_id = var.cf_zone_id

  settings {
    security_header {
      enabled            = true
      max_age            = 63072000
      include_subdomains = true
      preload            = true
      nosniff            = true
    }
  }
}
```

**Why `settings_override` and not a bespoke resource:**

- Provider v4 does not ship a dedicated `cloudflare_zone_hsts` resource.
- `settings_override` is the documented v4 mechanism for zone-level security_header config (Context7 docs confirmed).
- Using only the `security_header` sub-block leaves every other zone setting unmanaged — matches the narrow scope of this fix.
- v5 migration (when it happens) will split this into per-setting resources; out of scope here.

**No new variables required.** `cf_zone_id` is already declared in `variables.tf:62`.

#### B.2 Pre-apply validation gates

Before `terraform apply`, run three checks in order. Each must pass.

**Gate 1: `terraform validate` against the pinned provider version.** Per `2026-04-10-context7-terraform-provider-version-mismatch`, Context7 docs can drift ahead of the pinned provider. The project pins `cloudflare/cloudflare` at `4.52.5` (see `apps/web-platform/infra/.terraform.lock.hcl`). `cloudflare_zone_settings_override` with the `security_header` nested block is valid in v4 per the migration docs; `terraform validate` will fail fast if any attribute is v5-only.

```bash
cd apps/web-platform/infra
terraform validate
```

**Gate 2: CF API token permission probe.** The `cf_api_token` stored in Doppler `prd_terraform` has permissions granted incrementally (see `2026-03-21-cloudflare-api-token-permission-editing`). `cloudflare_zone_settings_override` requires `Zone > Zone Settings > Edit` at minimum. Probe with a read call first — this is non-destructive and confirms the token can GET zone settings. If the GET succeeds but subsequent PATCH during apply fails with 10000/Authentication Error, the token needs the Edit permission added via dashboard (Playwright MCP-automatable per the learning).

```bash
# Get token and zone_id from Doppler (two separate calls, no command substitution)
doppler secrets get CLOUDFLARE_API_TOKEN -p soleur -c prd_terraform --plain
doppler secrets get CF_ZONE_ID -p soleur -c prd_terraform --plain
# Then probe zone settings (replace placeholders with values from above):
curl -sS -H "Authorization: Bearer <TOKEN>" \
  "https://api.cloudflare.com/client/v4/zones/<ZONE_ID>/settings/security_header" \
  | jq '.success, .result.value'
```

- If `success: true` and `.result.value` includes `strict_transport_security.max_age: 31536000` → token can read; proceed to Gate 3.
- If `success: false` → token lacks read permission. Halt and update token via dashboard (see related learning). Do NOT attempt apply without read access.

**Gate 3: `terraform plan` preview.** Run with the dual-credential Doppler pattern (per AGENTS.md `cq-when-running-terraform-commands-locally` + `2026-03-21-ci-terraform-plan-workflow` §3):

export AWS_SECRET_ACCESS_KEY="$(doppler secrets get AWS_SECRET_ACCESS_KEY -p soleur -c prd_terraform --plain)"

# Use the name-transformer for TF_VAR_*. Inner `doppler run --token` avoids the
# service-token-config-scope-mismatch gotcha (2026-03-29-doppler-service-token-config-scope-mismatch).
doppler run -p soleur -c prd_terraform --name-transformer tf-var -- \

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Hook false-negative: sink inside `run:` but `new_string` does not contain the `run:` keyword (e.g., edit appends a single line inside an existing `run:` block). | Medium | Low (advisory hook; class of risk is rare, and upstream AGENTS.md rule `hr-in-github-actions-run-blocks-never-use` + SHA-pinned-action learnings already cover adjacent concerns). | Accept. The alternative (full YAML parse with Python stdlib) requires `pyyaml` which is not currently a dep. Document this limitation in the hook file's header comment with a pointer to #2525 for future enhancement. Test case 8 explicitly documents the safe-pattern scope. |
| Hook false-positive on `env:` block with sink + separate `run:` in same `new_string`. | Low | Low (pedagogically still correct — assigning `${{ github.event.issue.title }}` to an env var and then referencing `$VAR` in `run:` IS the safe pattern; the hook would then fire on a safe change). | The hook fires on sink+`run:` co-occurrence in the SAME `new_string`. When the safe pattern is introduced in one Edit call, the hook will flag it. This is a minor UX tradeoff: the advisory message explicitly calls out the env-var mitigation, so the user sees a confirming rather than contradicting signal. If this becomes a nuisance, narrow the detection to "sink within a `run:` block literal" (requires YAML parsing). |
| Cloudflare edge caches response headers and takes >60s to propagate. | Low | Low (eventual consistency — cURL re-verify will eventually succeed). | If post-apply cURL shows stale header, wait 2 min and retry. If still stale after 5 min, check Cloudflare dashboard for a stuck deployment and investigate. Bypass browser cache with `?_cf_bypass=$(date +%s)` in the URL. |
| `cloudflare_zone_settings_override` applies more fields than intended (the resource sets any settings we specify; defaults are applied for others). | Low | Low | Only include the `security_header` block inside `settings`. The resource does not touch other zone settings when they are not specified. Verified against Context7 docs for v4. `terraform plan` output (Gate 3 in B.2) will show the full diff before apply — reviewer MUST confirm only `security_header` is in the diff. |
| Existing Cloudflare HSTS value (1y, preload=true) was configured via dashboard and Terraform import fails. | Low | Low | On `terraform apply`, `cloudflare_zone_settings_override` does a PATCH-style overwrite. No import required; documented via Context7 migration guide as the intended v4 behavior. |
| CF API token lacks `Zone > Zone Settings > Edit` permission. Apply fails with Authentication Error (10000). | Medium | Low (recoverable) | Probe permissions in Gate 2 of B.2 BEFORE apply. If missing, update token via Playwright MCP per `2026-03-21-cloudflare-api-token-permission-editing` — token permission expansion does not rotate the secret, so no Doppler update is needed. This adds ~3 minutes but prevents a failed apply. |
| Context7 docs drift ahead of pinned provider 4.52.5; a recommended attribute is v5-only. | Low (verified in this plan) | Low (caught by validate) | Gate 1 in B.2 runs `terraform validate` BEFORE any API calls. A v5-only attribute fails validate immediately. Per `2026-04-10-context7-terraform-provider-version-mismatch`. |
| Test script discovery: CI does not auto-discover `.claude/hooks/*.test.sh`. | Low | Low (the test runs locally when the hook is edited; local-only coverage is still an improvement over none). | Check `scripts/test-all.sh` (A.4). If it already globs `**/*.test.sh`, no change needed. If it enumerates specific paths, add `.claude/hooks/security_reminder_hook.test.sh`. Verify by running `bash scripts/test-all.sh` locally. |
| Hook runs under Python 3 not installed in some runtime (e.g., stripped container). | Low | Low (fail-open: exit 127 → Claude Code treats as no-op, edit proceeds) | The hook is **advisory**; fail-open is correct. Document the Python 3 dependency in the hook header and the test script's preflight (skip-with-message if `python3` missing). |
| Deleting the 2nd commit reverts only the Cloudflare change — HSTS reverts to dashboard config. | Low | Low (a revert is a normal operation and re-aligns with pre-PR state) | Document in PR body that the two commits are independently revertable. R2 backend state will be rolled back on the next `terraform apply` after the revert. |

## Alternative Approaches Considered

| Alternative | Why not chosen |
|-------------|----------------|
| **#2525:** Write the hook in Bash to match the existing hook directory style (`guardrails.sh`, `worktree-write-guard.sh`). | Python is easier for the regex-heavy sink matching and JSON parsing. Bash with `jq` and `grep -P` would work but is harder to test and read. Existing Python hook `.py` files exist in other plugin directories, so the precedent is established. |
| **#2525:** Make the hook non-blocking (exit 0 with stderr warning). | Issue #2525 explicitly keeps the blocking behavior for genuine sinks — the complaint is false positives, not the blocking severity itself. An advisory-only version would weaken the signal. |
| **#2527:** Disable Cloudflare HSTS override entirely (option 2 in the issue) and let origin control. | Option 1 (align edge to 2y) is simpler, explicitly declared in Terraform, and matches the preload requirement in one step. Option 2 adds a failure mode: if Cloudflare is later re-enabled via dashboard (by a different operator), drift is silent. Option 1 fails loud at `terraform plan`. |
| **#2527:** Upgrade Cloudflare provider to v5 and use `cloudflare_zone_setting "security_header"`. | Out of scope. A provider major-version upgrade touches every Cloudflare resource in the file and changes schemas; the migration is ~15 resources. A separate PR (tracked via issue backlog when proposed) is the right home. v4 syntax is supported and documented. |
| **Bundle both fixes in one commit.** | They touch unrelated files and domains. Separate commits make `git log` cleaner and `git revert <sha>` targeted per-issue without unbundling. |
