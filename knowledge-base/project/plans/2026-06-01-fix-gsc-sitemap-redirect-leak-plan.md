## Overview

Google Search Console reports a "Page with redirect" coverage cluster for soleur.ai
(23 affected pages as of 2026-05-29). The supplied root-cause hypothesis was that
`plugins/soleur/docs/sitemap.njk`'s filter `entry.url.endsWith(".html")` was sweeping
the legacy `/pages/*.html` redirect stubs into `sitemap.xml`, so Google indexes
redirecting URLs.

**This plan VERIFIED the hypothesis against a real Eleventy build and the GSC drilldown
report, and the hypothesis does NOT reproduce on current HEAD.** The sitemap is already
clean: zero `/pages/*.html`, zero `.html` `<loc>` entries, single canonical apex host.
The `.endsWith(".html")` clause is currently **inert** because the redirect stubs are
generated by `page-redirects.njk` with `eleventyExcludeFromCollections: true`, so they
never enter `collections.all` and the sitemap loop never sees them.

The GSC cluster is therefore dominated by **Google's historical memory** of
`www.soleur.ai/*`, `/index.html`, `?ref=…`, and legacy `/pages/*.html` URLs that all
correctly 301/3xx to canonical clean apex URLs — exactly the class the task brief said
NOT to chase. There is no active sitemap leak to fix.

What IS shippable repo value: **regression-hardening**. Two latent gaps exist:

1. `sitemap.njk`'s `.html` clause is a loaded gun — if anyone later removes
   `eleventyExcludeFromCollections` from `page-redirects.njk` (or adds a new `.html`
   permalink), the stubs silently re-enter the sitemap and the GSC cluster regrows. The
   clause earns nothing today (no canonical page outputs at a bare `.html` URL) and
   should be dropped.
2. The deploy/CI sitemap gate (`validate-seo.sh`) **explicitly allows** `.html` `<loc>`
   entries (`grep -vE '(/|\.html)$'`), so it would NOT catch a `/pages/*.html`
   redirect-stub leak if one ever occurred. It needs a redirect-stub assertion to close
   the gap, mirroring the existing canonical-host (www-leak) gate.

This is a **docs/SEO change class**, single-domain, brand-survival threshold `none`
(static marketing sitemap; no user data, no auth, no money). The fix is ~3 small edits
plus one test.

## Implementation Phases

### Phase 1 — Drop the inert `.html` clause in `sitemap.njk`

**File:** `plugins/soleur/docs/sitemap.njk` (line 8)

Change the collection filter so the sitemap emits ONLY canonical trailing-slash directory
URLs. The `.html` clause earns nothing on current HEAD (verified: zero canonical pages
output at a bare `.html` URL) and is a latent re-leak vector.

```njk
{# before #}
{%- if entry.url.endsWith("/") or entry.url.endsWith(".html") %}

{# after #}
{%- if entry.url.endsWith("/") %}
```

Add a one-line comment above the filter explaining WHY `.html` is excluded (redirect
stubs at `/pages/*.html` must never enter the sitemap; canonical pages all use
trailing-slash directory URLs) so a future editor does not "helpfully" re-add it.

**Pre-edit verification (run at /work time, paste output into the commit):**
- `npx @11ty/eleventy && grep -oE '<loc>[^<]+</loc>' _site/sitemap.xml | grep -E '\.html$' || echo "no .html loc — clause is inert, safe to drop"` — must print the "safe to drop" line.

### Phase 2 — Add a redirect-stub assertion to the sitemap gate

**File:** `plugins/soleur/skills/seo-aeo/scripts/validate-seo.sh` (sitemap.xml section, ~L46-96)

The existing gate (L55) treats `.html` `<loc>` as VALID. Add a dedicated assertion AFTER
the existing non-HTML check that FAILS when any `<loc>` is a known redirecting-URL shape:
ends in `.html` OR contains `/pages/`. Place it inside the `if [[ -f sitemap.xml ]]`
block so it runs in lockstep with the canonical-host gate already there.

```bash
# Redirect-stub gate: redirect stubs live at /pages/*.html (meta-refresh +
# canonical-to-clean-URL, generated by page-redirects.njk, eleventyExcludeFromCollections).
# They MUST NEVER appear in the sitemap — Google indexes them as "Page with redirect".
# Canonical pages all use trailing-slash directory URLs; no canonical page ends in .html.
# Mirrors the canonical-host (www-leak) gate above. See the 2026-06-01 GSC plan.
REDIRECTING=$(grep -oP '(?<=<loc>)[^<]+' "$SITE_DIR/sitemap.xml" \
  | grep -E '(\.html$|/pages/)' || true)
if [[ -n "$REDIRECTING" ]]; then
  fail "sitemap.xml contains redirecting URLs (must be canonical clean URLs only): $REDIRECTING"
else
  pass "sitemap.xml contains no redirecting URLs (.html / /pages/)"
fi
```

Note the now-superseded L55 `NON_HTML` check still allows `.html`; the new gate is
additive (stricter) and does not require changing L55 — but add a one-line comment at L55
cross-referencing the new redirect-stub gate so the two are read together. (Do NOT tighten
L55 itself; its `(/|\.html)$` allowance is the "is this even a page URL" sanity check, a
different axis from "is this a redirecting URL".)

**Verification (run at /work time):**
- `bash plugins/soleur/skills/seo-aeo/scripts/validate-seo.sh _site` against the real
  build → must PASS (the live sitemap is clean).

#### Research Insights (precedent-diff)

**Precedent:** the new gate is a sibling of the **canonical-host (www-leak) gate** already
in the same `if [[ -f sitemap.xml ]]` block (`validate-seo.sh` L62-93). Both extract
`<loc>` hosts/paths via the same `grep -oP '(?<=<loc>)[^<]+'` form, both `fail`/`pass`
through the shared helpers, both guard the same GSC "Page with redirect" cluster (host
axis vs path axis). Adopt the precedent verbatim — no novel pattern.

```bash
# existing precedent (host axis, L71):
SITEMAP_HOSTS=$(grep -oP '(?<=<loc>)[^<]+' "$SITE_DIR/sitemap.xml" | sed -E '…' | sort -u || true)
# new gate (path axis) — same extraction, different predicate:
REDIRECTING=$(grep -oP '(?<=<loc>)[^<]+' "$SITE_DIR/sitemap.xml" | grep -E '(\.html$|/pages/)' || true)
```

**`set -euo pipefail` note:** the script runs under `set -euo pipefail` (L6). The trailing
`|| true` is mandatory — `grep -E` exits 1 when it finds no redirecting URLs (the PASS
case), and pipefail would abort the script before the empty-string `[[ -n … ]]` test. The
precedent gates use the same `|| true` for the same reason; mirror it exactly.

### Phase 3 — Regression test for the redirect-stub gate

**File:** `plugins/soleur/test/validate-seo.test.ts`

Add a test that writes a fixture `sitemap.xml` containing a `/pages/*.html` redirect-stub
`<loc>` and asserts `validate-seo.sh` exits non-zero with the redirect-stub failure
message. Add a positive companion: a clean trailing-slash-only sitemap fixture passes.

**Concrete test shape (verified against the existing suite, 2026-06-01):** the suite uses
`bun:test` and spawns the script via `Bun.spawn(["bash", SCRIPT, TMP_DIR], { stdout: "pipe", stderr: "pipe" })`,
then `await proc.exited` and `expect(stdout).toContain(…)`. Mirror it:

```ts
test("sitemap.xml with a /pages/*.html redirect stub fails", async () => {
  setupSite();
  // setupSite() writes a CLEAN default sitemap and has no content override —
  // overwrite it directly to inject the leak (or extend setupSite with sitemapContent?).
  writeFileSync(
    `${TMP_DIR}/sitemap.xml`,
    '<urlset><url><loc>https://soleur.ai/pages/pricing.html</loc><lastmod>2026-01-01</lastmod></url></urlset>',
  );
  const proc = Bun.spawn(["bash", SCRIPT, TMP_DIR], { stdout: "pipe", stderr: "pipe" });
  const exitCode = await proc.exited;
  const stdout = await new Response(proc.stdout).text();
  expect(exitCode).toBe(1);
  expect(stdout).toContain("contains redirecting URLs");
});

test("sitemap.xml with only clean trailing-slash URLs passes the redirect-stub gate", async () => {
  setupSite(); // default fixture sitemap is a single trailing-slash <loc>
  const proc = Bun.spawn(["bash", SCRIPT, TMP_DIR], { stdout: "pipe", stderr: "pipe" });
  const exitCode = await proc.exited;
  const stdout = await new Response(proc.stdout).text();
  expect(exitCode).toBe(0);
  expect(stdout).toContain("no redirecting URLs");
});
```

The leak `<loc>` (`https://soleur.ai/pages/pricing.html`) is synthesized from
`pageRedirects.js`'s shape (cq-test-fixtures-synthesized-only) so the test documents the
exact class it guards.

**Why this is the canonical test home:** `validate-seo.test.ts` already runs in CI and
already exercises the sitemap section of this script (lastmod, single-host). Adding the
redirect-stub case here means the gate is covered pre-merge (CI) AND post-merge
(deploy-docs.yml L75 runs the same script).
