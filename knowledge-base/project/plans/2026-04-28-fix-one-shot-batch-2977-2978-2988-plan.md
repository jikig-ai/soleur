### Key Improvements

1. **Critical reframing of #2988 thresholds.** Google restricted FAQ rich results to government/health-only sites in August 2023 and effectively discontinued them for all other sites by early 2024. Soleur is neither — so the FAQPage entity will be DETECTED and validated by the Rich Results Test, but NOT eligible for SERP rich-result rendering. Plan acceptance criteria adjusted: success = "valid + detected without errors," not "eligible for FAQ rich result." This avoids closing #2988 with a false-pass.
2. **Person + Organization cross-resolution path verified.** Confirmed via codebase: `_includes/base.njk:61` emits `Organization.founder.@id = https://soleur.ai/about/#jean-deruelle` on every page (including `/`); `pages/about.njk:146` emits matching Person `@id`. The cross-link is structurally correct — verification is mechanical string compare, not an "if it works."
3. **Playwright selector strategy locked.** Use role-based / accessible-name selectors (`getByRole('button', { name: 'TEST URL' })`, `getByText('Detected items')`) over CSS — Rich Results Test is a Google internal tool whose DOM class names may shift without notice; semantic selectors survive layout drift (per 2026 Playwright best-practice consensus).
4. **AEO framing preserved as the real win.** Even with FAQ rich results retired, FAQ schema remains a primary citation source for AI search platforms (LLM-driven crawlers parsing JSON-LD). The parity sweep #2978 protects AEO trust, not SERP rich-result eligibility — the framing in the issue body is correct, the SERP framing in the original plan was overstated.
5. **String-compare gating made explicit for cross-resolution.** Test panel may show "Person" and "Organization" as detected separately without surfacing the `@id` link — verification must extract the JSON-LD blocks from the page source (curl) and string-compare the `@id` values. Don't trust the test UI to render the cross-link.

### New Considerations Discovered

- **The Rich Results Test no longer reports "Eligible for rich result: FAQ" for non-gov/non-health sites.** It will instead report the structured data as "valid" without a rich-result-eligibility verdict for FAQPage. This is normal as of 2024 and is NOT a regression to flag.
- **`unanswered question`-class warnings can appear** when a Question lacks an `acceptedAnswer.text` — confirm none of our Questions trigger this.
- **`Schema.org Markup Validator`** (`https://validator.schema.org/`) is a complementary check that does NOT gate on rich-result eligibility — purely structural validation. Worth running alongside Rich Results Test for #2988 to disambiguate "schema valid, just not eligible for rich result" from "schema broken."
- **Playwright MCP runs `--isolated`** per `cq-playwright-mcp-uses-isolated-mode-mcp` — singleton lock failures resolved by killing Chrome processes; do NOT remove `--isolated`.

## Sharp Edges

- **`jsonLdSafe` is mandatory for any string that interpolates user/data content into a JSON-LD block.** `dump | safe` does NOT escape `</script>` (the rule from `cq-jsonld-dump-filter-not-enough-needs-jsonLdSafe`). Always use the registered `jsonLdSafe` filter.
- **The block terminator must be a literal `</script>`, never `<\/script>`.** The escape applies inside JSON string *values*, not at the block boundary (`2026-04-22-faqpage-jsonld-block-terminator-escape-placement.md`). This PR does not modify any block terminator, but reviewers should grep-confirm.
- **Diff the rendered page, not the template.** Per `2026-04-18-faq-html-jsonld-parity.md` Step 3: parity is a property of the built HTML + embedded JSON-LD, not the source `.njk`. The Phase 2 diff script reads from `_site/<page>/index.html`, not the templates.
- **HTML entities (`&mdash;`, `&rsquo;`) in HTML answers do NOT round-trip into JSON-LD.** Either normalize HTML to ASCII (chosen direction per the 2026-04-18 learning), or use literal codepoints in both surfaces. Never mix entity in HTML with ASCII in JSON.

## Research Insights

### FAQPage current state (2024-2026)

**Best Practices:**

- Google's August 2023 policy change restricted FAQ rich results to "well-known, authoritative government and health websites" only. By early 2024, FAQ rich-result rendering was effectively retired across the rest of the web.
- **The structured data is still valuable.** Per industry coverage: AI search platforms (Perplexity, Claude/ChatGPT browsing, Google AI Overviews) embraced FAQPage schema as a primary citation source — AEO use case is intact.
- **Validation tooling still validates the schema.** Both the Rich Results Test and Schema.org Markup Validator parse the FAQPage block and report it as valid; only the rich-snippet-eligibility verdict is suppressed.
- **Codepoint-exact parity between visible HTML and `acceptedAnswer.text` remains best practice.** Even though Google's developer doc no longer hard-states this requirement (recent text says "make sure each Answer includes the entire text of the answer" without "character-for-character"), the parity discipline is what AI crawlers and Schema.org validators key off. The 2026-04-18 internal learning continues to apply.

**Performance Considerations:**

- FAQ schema continues to support AEO (Answer Engine Optimization) — the LLM-driven AI search surface that's growing while traditional SERP rich snippets shrink.
- Source-of-truth interpolation via `{{ stats.X }}` in both HTML and JSON-LD eliminates the most common drift class (stat updates land in HTML, JSON-LD goes stale).

**Implementation Details:**

```nunjucks
{# Canonical pattern for stats-bearing FAQ JSON-LD: #}
"text": {{ ("Soleur deploys " + stats.agents + " AI agents across " + stats.departments + " business departments — engineering, marketing, legal, finance, operations, product, sales, and support — giving a single founder the capacity of a full organization. Every agent shares a compounding knowledge base, so your marketing agent knows what your legal agent decided, and your finance agent references your actual revenue data. Your expertise stays in the loop: agents provide starting points, you make the final call.") | jsonLdSafe | safe }}
```

The `jsonLdSafe` filter (defined `eleventy.config.js:30-35`) is `JSON.stringify(value).replace(/<\//g, "<\\/").replace(/ /g, "\\u2028").replace(/ /g, "\\u2029")` — it produces a valid JSON string literal AND escapes `</` to `<\/` *inside the string value* (NOT at the block terminator — the terminator stays a real `</script>` per `2026-04-22-faqpage-jsonld-block-terminator-escape-placement.md`).

**Edge Cases:**

- **Apostrophe encoding asymmetry.** HTML `&rsquo;` (U+2019) vs JSON-LD literal `'` (U+0027) is the second most common drift class after stats. Decision per 2026-04-18 learning: **normalize HTML to ASCII `'`** (codepoint U+0027) so both surfaces match. Typography purity loses to crawler/AI-parser parity.
- **Em-dash encoding.** HTML `&mdash;` (U+2014) is acceptable and the current JSON-LD already uses literal U+2014 (not an entity). No action needed.
- **Anchor tags inside HTML answers** (`<a href="...">see pricing</a>`) are stripped from JSON-LD `acceptedAnswer.text` by convention (JSON-LD is plain text). Strip them in the comparison script before codepoint diff. Pricing.njk Q4 ("How much does Soleur cost?") is a known case.
- **`<code>` tags inside HTML answers** (`<code>/soleur:go</code>`) are stripped in JSON-LD too. Same comparison handling.
- **`unanswered question` warning class** in RRT fires when a Question lacks `acceptedAnswer.text` — none of our questions are missing this, but verify in the diff script.

### Playwright Rich Results Test automation

**Best Practices (2026 consensus):**

- Use role-based selectors over CSS / class-name selectors. Google internal tools ship UI changes without notice; semantic selectors (`getByRole`, `getByText`) survive layout drift.
- For text-only signals (no clear role), use `browser_wait_for({ text: '...' })` with anchored substrings. Multiple wait points: "Loading…" → "Detected items" or "No items detected".
- Capture both `browser_take_screenshot` (visual artifact for the issue comment) AND `browser_snapshot` (structured DOM for parsing).
- Run with `--isolated` per Soleur's `.mcp.json` configuration. Do NOT remove — required for parallel sessions.

**Implementation Details:**

```javascript
// Pseudocode — actual MCP calls follow Playwright MCP tool surface
await navigate('https://search.google.com/test/rich-results');
await fill_form([{ name: 'URL input', ref: '<from snapshot>', value: 'https://soleur.ai/about/' }]);
await click({ name: 'TEST URL button', ref: '<from snapshot>' });
await wait_for({ text: 'Detected items', time: 30 });  // RRT fetches + analyzes; can take 15-30s

const snapshot = await browser_snapshot();
// Parse snapshot YAML for the detected items list
```

**Edge Cases:**

- **Rate limiting.** Google may rate-limit RRT after several rapid requests from the same IP. Mitigation: wait 60-90s between the two URL tests.
- **Auth wall (uncommon, but possible).** Per `hr-when-playwright-mcp-hits-an-auth-wall`: keep the tab open at the wall, ask the user to log in, resume in the same session. Never close-and-handoff-by-URL.
- **CAPTCHA wall (rare for RRT but documented for other Google services).** Same mitigation as auth wall — the only genuinely manual interaction.
- **Live-fetch caching.** RRT may serve a cached crawl of the page if recently tested; for a fresh post-deploy test, append `?_t=$(date +%s)` to the URL or use the "Live URL" toggle if available.
