## Overview

The Sentry cron monitor `scheduled-terraform-drift` (web-platform project, production) fires recurring
false-alarm "missed check-in" pages that immediately flip to "regressed" the moment the (healthy) run
finally checks in. The terraform-drift GitHub Actions workflow is **succeeding on every run** — this is
not a real-failure detector firing. The defect is config-shaped: the monitor's
`checkin_margin_minutes = 180` is undersized relative to GitHub Actions' scheduled-workflow **delivery
jitter** for this workflow.

The fix is a single-field change: raise `checkin_margin_minutes` from `180` to **`480`** on
`resource "sentry_cron_monitor" "scheduled_terraform_drift"` in
`apps/web-platform/infra/sentry/cron-monitors.tf`, and add a per-monitor rationale comment that records
the margin is sized to absorb observed GHA cron delivery jitter (citing the survey below). Sentry is
managed as IaC per ADR-031; the change auto-applies via `.github/workflows/apply-sentry-infra.yml` on
push to main (the apply workflow already `-target=`s this exact resource — see Research Reconciliation).

**The workflow itself is healthy and is NOT touched.** Only the monitor's tolerance window changes.

## Root Cause (confirmed by live jitter survey)

The workflow cron is `0 6,18 * * *` (06:00 & 18:00 UTC), defined in
`.github/workflows/scheduled-terraform-drift.yml:12`. GitHub Actions does not honor scheduled-workflow
trigger times precisely — under runner-pool load the actual dispatch is delayed, often by hours. The
Sentry monitor opens a "missed check-in" issue once `scheduled_time + checkin_margin_minutes` passes
with no check-in; when the late-but-successful run then checks in `?status=ok`, `recovery_threshold = 1`
flips the monitor straight to "regressed" — producing the false-alarm pair the operator sees.

**Live survey** (`gh run list --workflow=scheduled-terraform-drift.yml`, 115 scheduled runs over
~58 days, 2026-04-05 → 2026-06-01, all `conclusion=success` except one unrelated `failure` on 05-06):

| slot | n | min late | median late | p90 late | **max late** |
|------|---|----------|-------------|----------|--------------|
| 06:00 UTC | 57 | 64 min | 134 min | 237 min | **339 min** |
| 18:00 UTC | 58 | 49 min | 80 min | 131 min | 215 min |
| **all** | 115 | 49 min | 108 min | 215 min | **339 min** |

Lateness histogram (lower-bound bucket : count): `{0:9, 60:64, 120:29, 180:9, 240:3, 300:1}`.

- **13 of 115 deliveries (≈11%) exceeded the current 180-min margin** → ≈11% of fires are eligible to
  page a false alarm.
- 4 exceeded 240 min; 1 exceeded 300 min; **observed worst case = 339 min** (the 06:00 slot is the
  worst offender — overnight runner-pool degradation).
- The specific page that triggered this issue: the 2026-06-01 18:00 run landed at **21:34 UTC (214 min
  late)**, exceeding the 180-min margin; Sentry opened the miss at 21:00 UTC, then the successful
  check-in at 21:34 flipped the monitor to "regressed".

## Chosen value: 480 minutes (8 hours)

Sizing rule for a **scheduling-jitter** margin (distinct from a run-duration margin): cover the observed
worst-case delivery lateness with safety headroom, while staying strictly **below the interval between
consecutive fires** so a late run of one slot is never misattributed as a missed run of the next.

| candidate | covers 339-max? | headroom over max | under 720-min inter-fire gap? | verdict |
|-----------|-----------------|-------------------|-------------------------------|---------|
| 360 | yes | 21 min (6%) | yes | too tight — one bad GHA night re-fires |
| 420 | yes | 81 min (24%) | yes | acceptable |
| **480** | **yes** | **141 min (42%)** | **yes** | **chosen** |
| 540 | yes | 201 min (59%) | yes | over-padded for current data |

- **480 covers the observed 339-min worst case with 42% headroom** — absorbs further degradation
  without re-paging on a normal bad-jitter night.
- The two daily fires are **720 min apart** (06:00 → 18:00). 480 < 720, so a maximally-late 06:00 run
  (even at 480 min) checks in before the 18:00 window's miss could open. No cross-slot confusion.
- Precedent bracket: GHA-fired siblings span `scheduled_gh_pages_cert_state = 240` (daily, lower
  observed jitter) and `scheduled_realtime_probe = 1440` (daily, widened to 24h after GitHub *dropped*
  a whole run, issue #4189). 480 sits sensibly between them: larger than gh-pages because this
  workflow's observed max (339) exceeds gh-pages' tolerance; far smaller than realtime-probe because
  the failure mode here is *jitter*, not *whole-run-drop* (this workflow fires twice daily, so a single
  dropped run is recovered within 12h regardless).
- README's literal "observed + 2x" would yield ~1017 min, but that heuristic targets run **durations**;
  applied to jitter it over-pads well past the inter-fire interval and erodes real-miss sensitivity.
  480 honors the *spirit* (worst-case + meaningful safety) without the over-pad.

**Sensitivity guard:** the chosen value depends on the survey window. The 339-min max was observed over
58 days; if a future re-tune (per README's 30-day re-tune guidance) shows deliveries beyond ~400 min,
revisit toward 540. The new comment records the survey date so the next operator knows the basis.

### Research Insights — Precedent-Diff (GHA-fired margin cohort)

`git grep` of `checkin_margin_minutes` in `cron-monitors.tf` partitions the monitors into two cohorts by firing substrate:

| monitor | substrate | cadence | margin | basis |
|---|---|---|---|---|
| `scheduled_terraform_drift` (this) | **GHA** | twice daily | 180 → **480** | observed jitter max 339 min |
| `scheduled_gh_pages_cert_state` | GHA | daily | 240 | "absorbs GHA cron jitter" (cron-monitors.tf:224-234) |
| `scheduled_realtime_probe` | GHA | daily | 1440 | widened after GitHub *dropped* a whole run (#4189) |
| all Inngest-fired (e.g. `scheduled_daily_triage`, `scheduled_oauth_probe`, …) | Inngest | various | 30 | "Inngest fires deterministically with ≤2-min jitter" (header lines 28-35) |

**Precedent confirms the design pattern**: GHA-fired monitors get jitter-absorbing margins sized to their observed lateness; Inngest-fired monitors get a tight 30-min margin. `scheduled_terraform_drift` is GHA-fired, so it belongs to the jitter-margin cohort. Its observed max (339 min) is higher than `scheduled_gh_pages_cert_state`'s daily-cron tolerance (240) — explaining why 480, not 240, is the right value here. This is **not a novel pattern**; it is the established treatment for the GHA cohort, with a comment style to match (the new comment mirrors the `scheduled_gh_pages_cert_state` and `scheduled_realtime_probe` rationale blocks).
