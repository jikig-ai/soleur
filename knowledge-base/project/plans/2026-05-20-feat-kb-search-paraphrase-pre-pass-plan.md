# Plan: kb-search Stage 2 — LLM paraphrase pre-pass (#4176)

## Overview

Stage 1 of #4119 (PR #4156, merged 2026-05-20) closed the structural displacement bug in `kb-search`. Post-Stage-1 measurements: `R@5(heavy, kb-search) = 0.2947` vs grep ceiling `0.3025`, `gap_skill_roi = −0.008`. The remaining gap to the 0.4 ladder threshold is bounded by grep's own heavy-paraphrase ceiling; no keyword-strategy tuning can exceed it. Stage 1.5 (IDF/stopword) is mechanically moot. The published #4119 ladder routes `< 0.3 → Stage 2 (LLM paraphrase pre-pass)`.

This plan ships the agent-inline runtime variant (Option C from the brainstorm triad consensus) with the full operator-trust mitigation bundle: `--no-paraphrase` opt-out, adaptive routing on baseline grep hits, sensitive-query regex guard, hard caps, sha256-keyed paraphrase cache, per-bucket bench breakdown, stderr-warn fallback, and a TR2 lockstep CI assertion that is **net-new infrastructure** — Stage 1 promised operator-discipline lockstep but never landed an automated check.

The work is bench-gated. After plan + work land, the operator decides whether to spend ~$3-$14 + ~70min on the bench rerun (recomputed cost — see Risks). Ladder triage on bench result decides close-this-+-#4119 vs file-Stage-3 with mandatory ADR-trigger note.

## Research Insights

- **kb-search SKILL.md is interpreted prose, not a CLI.** Per repo-research and `scripts/learning-retrieval-bench.sh:548`: kb-search is a Markdown strategy that agents execute via grep + Read. Option C runtime is therefore *prose-based instruction to the executing agent* — the agent paraphrases inline within its own context. No separate Anthropic API call is generated by the skill itself at runtime.
- **Bench paraphrase machinery already exists** at `scripts/learning-retrieval-bench.sh:319-361` (`anthropic_paraphrase()` + `PROMPT_LIGHT`/`PROMPT_HEAVY` for corpus-side paraphrasing). Stage 2 reuses this function with a **new third prompt** (`PROMPT_QUERY_PARAPHRASE`) and extends `kbsearch_rank()` at `:494` to invoke per-variant union-rank logic.
- **Bench cache (existing) schema** is path-keyed flat: `{"path": "<rel>", "identity": "<gt>", "light": "<paraphrase>", "heavy": "<paraphrase>"}` (line 1138-1144). Runtime query-paraphrase cache is **distinct** (different key shape: sha256 of operator query, different lifecycle: 14-day TTL). Two cache files coexist; bench can warm only the corpus cache, runtime maintains its own.
- **Stage 1 added NO CI assertion** (verified via `git show 8a4f1d6d` — only added `self_test_flooding_pathology` to the bench's own `--self-test`). The "TR2 lockstep" promise was operator-discipline only. Stage 2 introduces the FIRST automated lockstep assertion (Phase 6 of this plan), landing as `plugins/soleur/test/kb-search-lockstep.test.sh` (sibling of existing `plugins/soleur/test/lint-bot-synthetic-completeness.test.sh`).
- **`.soleur/cache/` is the canonical Soleur local-state convention** per `.gitignore:52` — already gitignored. No `~/.cache/soleur/` precedent exists in the repo (brainstorm's CLO-blessed XDG path lacks Soleur precedent). See **Plan Deviations from Brainstorm** below.
- **Canonical stderr-warn idiom** is `<skill>: WARN — <message>` (em-dash separator, lowercase skill prefix), per `plugins/soleur/skills/compound/SKILL.md:245`, `plugins/soleur/skills/ship/SKILL.md:640`, `plugins/soleur/skills/code-to-prd/scripts/code-to-prd.sh:548`. Stage 2 uses this verbatim for graceful-degradation messages.
- **No community paraphrase-prepass-over-grep precedent exists.** Functional-discovery surfaced `juanre/llmemory` (Postgres hybrid + RRF fusion) and LangChain's `MultiQueryRetriever` — both adopt LLM query-expansion + fusion but assume vector-DB substrate. Soleur's grep substrate + path-union fusion is bespoke. **Record as precedent gap.** RRF (Reciprocal Rank Fusion) is cited in Risks as a deferred fusion alternative if `union-by-hit-count` proves under-discriminating in the bench rerun.
- **Description budget headroom: NEGATIVE 121 words** (cumulative 1921 / cap 1800, across 73 skills). Re-measured at plan time; brainstorm's −47 figure was 74 words off (likely measurement glitch). Plan MUST NOT grow `kb-search/SKILL.md` `description:` field. Verified per `cq-skill-description-budget-headroom`. Body sections grow freely; only the YAML frontmatter `description:` is budget-constrained.

## Risks

- **Bench cold-rerun cost recomputed: ~$6-$13, not ~$3.** Stage 1 produced ~3441 corpus paraphrases (≈ $3 cold). Stage 2 adds 3 query-paraphrase variants per query = ~10,300 extra Haiku calls × $0.0003-$0.001 → $3-$10 marginal. Total cold: $6-$13; warm rerun is sub-second. Variant count pinned at 3 to floor the cost. Operator ack at AC14 PAUSE covers the spend.
- **Paraphrase quality variance under Option C** — runtime quality depends on session model (Opus > Sonnet > Haiku). Bench measures Haiku as the **ceiling**; runtime can exceed or undershoot. Bench is the published gate; runtime variance is accepted. Revisit if 14-day actuals (AC13) show systematic underperformance.
- **Adaptive-routing edge case** — query returns 4 baseline hits and the target sits at rank 50. Adaptive gate skips paraphrase; target stays unreachable. Bounded risk: heavy-paraphrase queries typically return < 5 anyway. If 14-day actuals show this dominating, retune the constant 5 in a follow-up PR.
- **Operational tradeoffs (TR2 token uniqueness + chmod-on-shared-machines + RRF as future option)** — The `stage-2-paraphrase-union-v1` token is 27 chars and unique-by-construction; false-positive collision in unrelated prose is possible but the asymmetric-removal failure is what AC4 catches. `chmod 700` on `.soleur/cache/kb-search/` is load-bearing on multi-user dev machines, symbolic on single-user laptops. Reciprocal Rank Fusion (RRF) per `juanre/llmemory` is the natural Stage 2.5 alternative if `union-by-hit-count` proves under-discriminating in the bench rerun; deferred, not in scope.
