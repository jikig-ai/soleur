# KB Manifest (INDEX.md) and Cross-Domain Search Skill

## Overview

Add two components to improve knowledge base discoverability: (1) an auto-generated `knowledge-base/INDEX.md` manifest listing every KB file with a one-line description, and (2) a `soleur:kb-search` skill that greps across all KB domains. This addresses the discovery gap where agents don't know what files exist and cross-category searches fail.

## Problem Statement

Soleur's knowledge base has ~2,375 markdown files with no discovery mechanism. The learnings-researcher agent searches only `knowledge-base/project/learnings/` via keyword grep on YAML frontmatter. Files outside that path, or files with different terminology, are invisible. The KB viewer UI has no functional search. Context compaction already causes problems at current scale.

## Proposed Solution

### Component 1: INDEX.md Generator (`scripts/generate-kb-index.sh`)

A bash script that:

1. Walks all `.md` files in `knowledge-base/` recursively (skips non-markdown: `.png`, `.pen`, `.gitkeep`)
2. Excludes `archive/` directories and `INDEX.md` itself
3. Extracts `title:` from YAML frontmatter using `awk '/^---$/{c++; next} c==1'` pattern (per learning: 2026-03-12-directory-driven-content-discovery-frontmatter-parsing)
4. Falls back to first `# heading` when `title:` is absent, then to kebab-to-title-case filename
5. Outputs a deterministic, sorted flat markdown list grouped by top-level domain
6. Completes in <5 seconds for the current corpus

Output format — flat list, grepable by agents, scannable by humans:

```markdown
# Knowledge Base Index

> Auto-generated by `scripts/generate-kb-index.sh`. Do not edit manually.
> Total files: 1842

## engineering

- [Non-Functional Requirements Register](engineering/architecture/nfr-register.md)
- [Architecture Decision Records](engineering/architecture/decisions/README.md)

## marketing

- [Brand Guide](marketing/brand-guide.md)
- [Soleur vs Anthropic Cowork](marketing/distribution-content/2026-03-16-soleur-vs-anthropic-cowork.md)

## project

- [Review Compound Before Commit](project/learnings/2026-02-12-review-compound-before-commit-workflow.md)
- [Doppler Service Token Config Scope Mismatch](project/learnings/2026-03-29-doppler-service-token-config-scope-mismatch.md)
```

Paths are relative to `knowledge-base/`. No tags column (82% of files lack tags). No subsection counts. No tables (flat lists are diffable and don't break on special characters).

### Component 2: kb-search Skill (`plugins/soleur/skills/kb-search/`)

A skill that accepts keyword arguments and searches across all KB domains:

```text
/soleur:kb-search authentication middleware
```

The skill:

1. Greps `knowledge-base/INDEX.md` for title matches (tier 1 — high relevance)
2. Runs parallel Grep across `knowledge-base/` file contents for body matches (tier 2 — lower relevance)
3. Returns tier 1 results first, then tier 2, capped at 20 results via `head`

Two-tier ranking only. No weighted scoring, no recency bonus. Agents consuming the output apply their own judgment. If ranking proves insufficient, add complexity later with real data.

### Component 3: Lefthook Integration

Add INDEX.md regeneration as a pre-commit hook:

```yaml
# lefthook.yml — under pre-commit: commands:
generate-kb-index:
  priority: 10
  glob: "knowledge-base/**/*.md"
  run: bash scripts/generate-kb-index.sh && git add knowledge-base/INDEX.md
```

Add `knowledge-base/INDEX.md` to `.markdownlintignore` so the generated file does not trip the `markdown-lint` hook (priority 4).

Known constraint: lefthook hangs in worktrees >60s — the script must complete in <5 seconds.

## Technical Considerations

**Frontmatter extraction in bash:** The `awk` pattern from the directory-driven discovery learning handles `---`-delimited YAML blocks and files without frontmatter. Pure bash/awk for speed — no Python dependency in the hook path.

**Deterministic output:** Sort files lexicographically within each domain section using `LC_ALL=C sort`. No timestamp in the output (avoids unnecessary diffs on every run).

**INDEX.md size:** At ~2,375 files with one line per file, expect ~60-80 KB. Fits easily in Claude's context window.

**learnings-researcher integration:** Update the agent's instructions to check INDEX.md first for file discovery, then fall back to direct grep for content matching. This is an instruction update to `plugins/soleur/agents/engineering/research/learnings-researcher.md`.

**Merge conflicts:** INDEX.md is a generated file. Do not use `.gitattributes` merge strategies. After any merge that conflicts on INDEX.md, regenerate by running `bash scripts/generate-kb-index.sh`. Document this in the script's `--help` output.

## Test Scenarios

- Given no INDEX.md exists, when `bash scripts/generate-kb-index.sh` runs, then INDEX.md is created listing all non-archived KB files
- Given INDEX.md exists, when the script runs again with no file changes, then output is identical (diff returns 0)
- Given a new file is added to `knowledge-base/project/learnings/`, when the script runs, then the new file appears in the index
- Given a file has no `title:` frontmatter but has a `# Heading`, when the script runs, then the heading is used as the title
- Given a file has no frontmatter and no heading, when the script runs, then the filename (kebab-to-title-case) is used as the title
- Given files exist under `knowledge-base/project/brainstorms/archive/`, when the script runs, then archived files are excluded from INDEX.md
- Given `soleur:kb-search authentication`, when the skill executes, then results include files from multiple domains (not just learnings)
- Given the lefthook hook is configured, when committing a change to a KB file, then INDEX.md is regenerated and staged

## Domain Review

**Domains relevant:** Engineering, Product, Marketing, Operations

Domain assessments carried forward from brainstorm (2026-04-07). All four leaders recommended improved file-based retrieval over RAG/PageIndex. No new domain implications from the implementation plan — purely internal tooling with no user-facing, cost, or brand impact.

## Dependencies and Risks

**Dependencies:**

- `awk` and `sort` available in the shell environment (standard on Linux/macOS)
- lefthook installed and configured (already in place)

**Risks:**

- Lefthook hang in worktrees: mitigated by keeping script under 5 seconds and documenting the `pkill -f lefthook` workaround
- INDEX.md merge conflicts: resolved by regenerating post-merge (`bash scripts/generate-kb-index.sh`)
- Title quality: ~1,700 files lack `title:` frontmatter, so titles fall back to headings or filenames. Quality improves over time as new files get proper frontmatter via compound/brainstorm/plan skills (no separate enforcement needed — those skills already write frontmatter)

## Alternative Approaches Considered

| Approach | Verdict | Reason |
|----------|---------|--------|
| RAG with vector embeddings | Deferred (#1740) | 4+ new components, contradicts brand, infrastructure cost |
| PageIndex (LLM reasoning) | Deferred (#1740) | LLM-in-LLM anti-pattern, $50-200+/month at scale |
| Weighted scoring heuristic | Rejected | Overengineered for v1. Two-tier ranking (title then body) is sufficient. Add complexity when simple proves insufficient. |
| Table format for INDEX.md | Rejected | Tables break on special characters, aren't diffable, and Tags column is empty for 82% of files. Flat list is simpler. |
| Frontmatter enforcement (Component 3) | Deferred | Convention change that touches 4+ skill directories. Separate concern from index generation. Existing skills already write frontmatter on new files. |
| CI-only index generation | Rejected | INDEX.md should be current locally, not just in CI. |
