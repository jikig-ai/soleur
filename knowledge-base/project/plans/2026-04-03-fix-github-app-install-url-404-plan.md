# fix: GitHub App install URL returns 404 for non-owner users

## Problem Statement

During new account setup via GitHub OAuth, the "connect repo" onboarding step redirects users to `https://github.com/apps/soleur-ai/installations/new`. This URL returns a 404 page.

**Root cause (two issues):**

1. **GitHub App visibility is set to private.** Private GitHub Apps can only be installed by the account that owns the app. Any other user visiting the install URL sees a 404 page. For Soleur to work for any user, the app must be set to **public**.

2. **The app slug `soleur-ai` is hardcoded as a fallback.** The frontend uses `process.env.NEXT_PUBLIC_GITHUB_APP_SLUG ?? "soleur-ai"` (line 54 of `connect-repo/page.tsx`). The env var `NEXT_PUBLIC_GITHUB_APP_SLUG` is not set in any Doppler config or GitHub Actions secret. If the actual GitHub App was registered with a different slug, the hardcoded fallback is wrong. Additionally, hardcoding the slug is fragile -- if the app is ever renamed, the URL breaks silently.

**Evidence:**

- `gh api /apps/soleur-ai` returns 404 (app either doesn't exist or is private)
- `GITHUB_APP_ID` and `GITHUB_APP_PRIVATE_KEY` are not in any Doppler config (`dev`, `prd`, `ci`, `prd_terraform`)
- `NEXT_PUBLIC_GITHUB_APP_SLUG` is not set anywhere
- GitHub docs confirm: private apps show a 404 to non-owners at the `/installations/new` URL

## Proposed Solution

### Phase 1: Verify/Create the GitHub App (configuration)

1. **Check if the GitHub App exists.** Use Playwright MCP to navigate to `https://github.com/organizations/jikig-ai/settings/apps` (or the personal account equivalent) to verify whether a GitHub App named "soleur-ai" (or similar) has been registered.

2. **If the app does not exist:** Create it via GitHub's app registration page (`https://github.com/settings/apps/new`) with:
   - Name: `Soleur` (slug will be auto-generated by GitHub)
   - Permissions: `contents:read+write`, `metadata:read`, `members:read` (for org install verification)
   - Setup URL: `https://soleur.ai/connect-repo` (the callback page)
   - Webhook: disabled (not needed for this flow)
   - Visibility: **Public** (critical for multi-user access)

3. **If the app exists but is private:** Change visibility to public via GitHub App settings.

4. **Store credentials in Doppler `prd` config:**
   - `GITHUB_APP_ID` -- the numeric App ID from the app settings page
   - `GITHUB_APP_PRIVATE_KEY` -- generate a new private key and store it
   - `NEXT_PUBLIC_GITHUB_APP_SLUG` -- the app slug from the settings page

### Phase 2: Add server-side slug resolution (defense-in-depth)

Instead of relying solely on a hardcoded env var for the slug, add a server-side API route that fetches the app slug dynamically from GitHub's `GET /app` API endpoint (authenticated with the App JWT). This provides a robust fallback if the slug ever changes.

**Files to create/modify:**

- `apps/web-platform/server/github-app.ts` -- add `getAppSlug()` function that calls `GET /app` with the App JWT and returns the `slug` field. Cache the result in memory (the slug never changes at runtime).
- `apps/web-platform/app/api/repo/app-info/route.ts` -- new API route `GET /api/repo/app-info` that returns `{ slug: string }` for authenticated users. Calls `getAppSlug()` server-side.
- `apps/web-platform/app/(auth)/connect-repo/page.tsx` -- replace hardcoded `GITHUB_APP_SLUG` constant with a fetch to `/api/repo/app-info` on mount. Fall back to the `NEXT_PUBLIC_GITHUB_APP_SLUG` env var if the API call fails.

### Phase 3: Update deployment configuration

- Add `GITHUB_APP_ID`, `GITHUB_APP_PRIVATE_KEY`, and `NEXT_PUBLIC_GITHUB_APP_SLUG` to the Docker environment / deployment pipeline so the production server has access to these values.
- Verify `GITHUB_APP_ID` and `GITHUB_APP_PRIVATE_KEY` are accessible to the server-side code at runtime (these are already referenced in `server/github-app.ts` but were never provisioned).

## Acceptance Criteria

- [ ] GitHub App exists and is set to **public** visibility
- [ ] `GITHUB_APP_ID`, `GITHUB_APP_PRIVATE_KEY`, and `NEXT_PUBLIC_GITHUB_APP_SLUG` are stored in Doppler `prd` config
- [ ] Visiting `https://github.com/apps/<actual-slug>/installations/new` as a non-owner GitHub user shows the install page (not 404)
- [ ] The connect-repo page correctly redirects to the working GitHub App install URL
- [ ] Server-side `getAppSlug()` function fetches and caches the slug from GitHub API
- [ ] New `/api/repo/app-info` route returns the correct slug for authenticated users
- [ ] Frontend falls back gracefully if `/api/repo/app-info` fails (uses env var)

## Test Scenarios

- Given a new user completes GitHub OAuth, when they reach the connect-repo page and click "Connect", then they are redirected to a working GitHub App install URL (no 404)
- Given the `NEXT_PUBLIC_GITHUB_APP_SLUG` env var is set, when `/api/repo/app-info` fails, then the frontend falls back to the env var value
- Given the GitHub App JWT credentials are configured, when `getAppSlug()` is called, then it returns the correct slug and caches it
- Given a user is not authenticated, when they call `GET /api/repo/app-info`, then they receive a 401 response
- **Browser:** Navigate to `https://github.com/apps/<slug>/installations/new` in an incognito window (not logged in as app owner) -- verify the page loads with an "Install" button, not a 404

## Context

- The connect-repo feature was implemented in the feat-repo-connection session (2026-03-29) but the GitHub App credentials were never provisioned to production
- The `server/github-app.ts` file has working JWT auth, token exchange, and repo operations -- but they all throw on startup because `GITHUB_APP_ID` is not set
- Three places in `connect-repo/page.tsx` redirect to the install URL (lines 1185, 1202, 1210)
