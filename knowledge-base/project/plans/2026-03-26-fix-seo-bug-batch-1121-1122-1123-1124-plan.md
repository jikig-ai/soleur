## Overview

Four SEO issues flagged by the 2026-03-23 and 2026-03-25 growth audits. Local research reveals that two are real bugs requiring code changes, one is already resolved (false positive from audit methodology), and one is a false positive caused by the SEO audit not following HTTP redirects.

## Issue Triage

### #1121 -- Meta tags (OG, canonical, Twitter cards) not rendering in production HTML

**Verdict: FALSE POSITIVE -- audit methodology bug**

The SEO audit agent uses `curl -s https://soleur.ai` without the `-L` flag. Cloudflare's Bot Fight Mode returns a 301 redirect before serving the actual page. Without `-L`, `curl` receives only the 301 response (a bare `<html><head><title>301 Moved Permanently</title></head>` page), which naturally lacks meta tags.

**Evidence:**

- `curl -s https://soleur.ai | head -5` returns: `<html><head><title>301 Moved Permanently</title></head>`
- `curl -sL https://soleur.ai | grep -E 'og:|twitter:|canonical'` returns all 15+ meta tags correctly
- Local `npx @11ty/eleventy` build produces correct meta tags in `_site/index.html`
- The `deploy-docs.yml` CI workflow includes a `validate-seo.sh` step that verifies meta tags in the build output (and passes)

**Root cause:** The `scheduled-seo-aeo-audit.yml` workflow's Claude agent fetches live pages with `curl` without `-L`, gets the 301 redirect page, and reports all tags as missing. This has been a recurring false positive in both the 2026-03-23 and 2026-03-25 audits.

**Action:** Close #1121 as not-a-bug. File a separate issue to fix the SEO audit agent's `curl` usage (add `-L` flag) to prevent future false positives.

#### Research Insights

**Agent prompt gap:** The `seo-aeo-analyst.md` agent (Step 2, Meta Tags) says to "Check for `<link rel="canonical">`" but does not instruct how to fetch live pages. The agent independently decides to use `curl -s` without `-L`. The fix should add a guidance line to the agent prompt: "When fetching live pages for verification, always use `curl -sL` (follow redirects) -- Cloudflare Bot Fight Mode returns 301 redirects that strip all page content."

**Scope of follow-up issue:** The follow-up issue should update:

1. `plugins/soleur/agents/marketing/seo-aeo-analyst.md` -- add curl `-L` guidance to the Meta Tags audit step
2. Consider adding a generic "Important Guidelines" bullet: "When fetching live URLs with curl, always use `-L` to follow redirects"

### #1122 -- feed.xml incorrectly included in sitemap.xml

**Verdict: REAL BUG**

The `sitemap.njk` template iterates `collections.all` (line 7), which includes the Atom feed at `/blog/feed.xml`. The Eleventy RSS plugin (v3) sets `eleventyExcludeFromCollections: ["blog"]` on the feed template -- this excludes it from the `blog` collection (avoiding circular reference) but NOT from `collections.all`.

**Impact:** One non-indexable XML entry in sitemap. Search engines waste crawl budget and may log warnings.

**Fix:** Add a URL extension filter in `sitemap.njk` to exclude non-HTML entries. Filter out entries whose URL ends with `.xml`:

File: `plugins/soleur/docs/sitemap.njk`

```njk
{%- for entry in collections.all %}
{%- if not entry.url.endsWith(".xml") %}
  <url>
    <loc>{{ site.url }}{{ entry.url }}</loc>
    <lastmod>{{ entry.date | dateToShort }}</lastmod>
  </url>
{%- endif %}
{%- endfor %}
```

**Alternative considered:** Setting `eleventyExcludeFromCollections: true` in the feed plugin config. Rejected because the plugin generates a virtual template and the config option is not exposed -- the exclusion logic is internal to the plugin (see `node_modules/@11ty/eleventy-plugin-rss/src/virtualTemplate.js` lines 141-151).

#### Research Insights

**Eleventy docs confirmation (Context7):** The Eleventy v3 documentation recommends `eleventyExcludeFromCollections: true` for feeds and sitemaps. The RSS plugin uses the array form `["blog"]` which is a valid Eleventy feature (exclude from specific collections only). This is intentional -- the feed needs to be in `collections.all` for other purposes (like the Eleventy build pipeline), but the sitemap should filter it out. Template-level filtering is the correct approach.

**Nunjucks compatibility verified:** `entry.url.endsWith(".xml")` works because Nunjucks passes through JavaScript string methods on string values. Verified locally with `node -e "..."` against the installed Nunjucks version.

**Edge cases:**

- `llms.txt` is already excluded via `eleventyExcludeFromCollections: true` in its frontmatter -- it does not appear in `collections.all`
- No other non-HTML entries exist in `collections.all` besides `feed.xml`
- Future non-HTML outputs (e.g., JSON API endpoints) would also be caught by this filter, which is the desired behavior
- The filter uses `entry.url` (the output URL path) not `entry.outputPath` (the filesystem path), matching the sitemap's URL context

**Regression guard enhancement:** Add a sitemap validation check to `validate-seo.sh` to catch non-HTML entries in CI:

```bash
# ── sitemap.xml: no non-HTML entries ──────────────────────────────────────────
if [[ -f "$SITE_DIR/sitemap.xml" ]]; then
  NON_HTML=$(grep -oP '(?<=<loc>)[^<]+' "$SITE_DIR/sitemap.xml" | grep -vE '(/|\.html)$' || true)
  if [[ -n "$NON_HTML" ]]; then
    fail "sitemap.xml contains non-HTML entries: $NON_HTML"
  else
    pass "sitemap.xml contains only HTML entries"
  fi
fi
```

This catches future regressions where non-HTML content (XML feeds, JSON endpoints, txt files) leaks into the sitemap.

### #1123 -- Case studies missing from Atom feed entries

**Verdict: FALSE POSITIVE -- already resolved**

All 5 case studies are in the `plugins/soleur/docs/blog/` directory and inherit the `blog` tag from `blog/blog.json` via Eleventy's directory data cascade. The local build produces a feed with 12 entries including all case studies. The live production feed at `https://soleur.ai/blog/feed.xml` also includes all 5 case studies.

**Evidence:**

- `blog/blog.json` sets `"tags": "blog"` which applies to all files in the directory
- Local build: `grep '<entry>' _site/blog/feed.xml | wc -l` returns 12
- Live production: `curl -sL https://soleur.ai/blog/feed.xml | grep 'case-study'` returns all 5 case study URLs
- The 2026-03-25 audit reports "6 of 12 blog posts" in the feed, but the live feed contradicts this

**Root cause of false positive:** Likely the same `curl -s` without `-L` issue as #1121, or the audit was comparing against a cached/stale version of the feed.

**Action:** Close #1123 as not-a-bug. The audit methodology fix from #1121 will prevent future false positives.

#### Research Insights

**Eleventy directory data cascade (Context7):** Confirmed that `blog.json` directory data files cascade to all files in the directory. The `"tags": "blog"` setting merges with per-file frontmatter tags. Case studies that define `tags: [case-study, marketing]` in their frontmatter will end up with BOTH their custom tags AND the directory-level `blog` tag. This is correct Eleventy behavior and is why the case studies appear in `collections.blog`.

**Relevant learning:** `knowledge-base/project/learnings/2026-03-05-eleventy-blog-post-frontmatter-pattern.md` documents this exact pattern -- blog posts inherit `layout` and `ogType` from `blog.json`, and individual frontmatter should not duplicate these fields. The case studies follow this pattern correctly.

### #1124 -- Author URL points to homepage instead of About page

**Verdict: REAL BUG -- but blocked by dependency**

`site.json` line 28 sets `site.author.url` to `"https://soleur.ai"` (the homepage). This URL is used in:

1. `blog-post.njk` line 40: `<a href="{{ site.author.url }}" rel="author">` (visible byline)
2. `blog-post.njk` line 23: JSON-LD `BlogPosting.author.url` (structured data)
3. `base.njk` line 56: JSON-LD `SoftwareApplication.author.url` on homepage (Organization type)

The issue explicitly states this is blocked by the About page (scheduled Apr 10-16).

**Partial fix (implementable now):** Update `site.author.url` to `"https://soleur.ai/about/"` in `site.json`. The link will be a dead link until the About page is created, but:

- Search engines will see the author URL pointing to the expected location
- When the About page is created, everything will work automatically without a second SEO fix
- A 404 for `/about/` is arguably better for SEO than linking author to the homepage (it signals intent)

**Alternative: Do nothing until About page exists.** This avoids dead links but delays the SEO improvement further. The issue says Apr 10-16, but that could slip.

**Recommended approach:** Implement the partial fix now. Update `site.author.url` to `/about/` and ensure the 404 page provides a reasonable experience. Add a code comment noting the dependency.

#### Research Insights

**E-E-A-T impact:** Google's E-E-A-T (Experience, Expertise, Authoritativeness, Trustworthiness) framework evaluates author attribution as a quality signal. The JSON-LD `BlogPosting.author.url` pointing to an author profile page (even a 404 that will become a real page) is structurally more correct than pointing to the homepage. Search engines can distinguish between "author has a profile page at this URL" and "author URL is the site homepage."

**Scope of change:** The `site.author.url` is used in 3 places (not 2 as originally noted):

- `blog-post.njk` line 23: `BlogPosting.author.url` in JSON-LD (Person type)
- `blog-post.njk` line 40: visible byline `<a href="{{ site.author.url }}" rel="author">`
- `base.njk` line 56: `SoftwareApplication.author.url` in JSON-LD (Organization type, homepage only)

The third usage is for the Organization (not Person) author of the SoftwareApplication. Pointing an Organization's URL to `/about/` is a standard pattern. No template changes needed -- the single `site.json` change propagates to all three locations.

**Relevant learning:** `knowledge-base/project/learnings/2026-03-24-agent-hallucinated-author-name-from-org-context.md` documents how the author name was previously hallucinated. The author name is now correct ("Jean Deruelle"), but the URL still points to the homepage. The same principle applies: author identity data should point to a canonical source.

**404 page experience:** The existing 404 page at `plugins/soleur/docs/404.njk` provides a reasonable experience with a "Back to Home" button. Users who click the author byline before the About page exists will see a clean 404, not a broken page.

## Regression Guard: validate-seo.sh Enhancement

Add a new check to `plugins/soleur/skills/seo-aeo/scripts/validate-seo.sh` within the sitemap section to catch non-HTML entries:

```bash
# Check for non-HTML entries in sitemap
NON_HTML=$(grep -oP '(?<=<loc>)[^<]+' "$SITE_DIR/sitemap.xml" | grep -vE '(/|\.html)$' || true)
if [[ -n "$NON_HTML" ]]; then
  fail "sitemap.xml contains non-HTML entries: $NON_HTML"
else
  pass "sitemap.xml contains only HTML entries"
fi
```

This ensures that if future Eleventy plugins or templates add non-HTML outputs to `collections.all`, the CI build will catch the sitemap pollution before deployment.
