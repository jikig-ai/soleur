# Brainstorm: Design taste-learning (multi-variant + committed taste-profile)

## What We're Building

Wave 3 · FR7 of the gstack-adoption epic (#5983). Adapts gstack's `design-shotgun`
into Soleur's committed-knowledge, all-Claude frame. Two coupled capabilities added
to Soleur's design surfaces:

1. **Multi-variant design fan-out** — when generating UI (coded interfaces via the
   `frontend-design` skill, `.pen` wireframes via the `ux-design-lead` agent), fan
   out N variants in parallel, each seeded a distinct aesthetic direction, so the
   operator chooses from a slate instead of a single take.
2. **A committed `taste-profile`** artifact in `knowledge-base/` (NOT a home-dir
   `~/.gstack`-style store) that learns the operator's design preferences from their
   variant selections, with **decaying confidence** and **contradiction-flagging**.
   It is **loaded via FR6** (#5989 declarative context-injection) so future design
   sessions are primed with learned taste.

**Rides #5989 (FR6) — no bespoke loader.** FR6 landed today (PR #6035, ADR-086):
a PostToolUse `Skill`-matcher hook reads a skill's `context_queries:` frontmatter
and injects a *pointer* (read-directive) to committed `knowledge-base/` artifacts;
the agent Reads them through its normal channel. This feature adds the taste-profile
path to the consumer's `context_queries` rather than building any new loader.

## Why This Approach

Operator directives locked three core mechanics (Phase 2 dialogue):

- **Write-time recompute for decay.** FR6 injects a *static pointer*, so nothing
  computes decay at load time. Confidence is therefore stored as explicit values
  with a `last_reinforced` date, decayed-and-rewritten at the start of each design
  session (self-contained; no cron, no DB). Rejected: scheduled decay cron (infra +
  recurring job for a single-operator artifact — heavier than the AC needs).
- **Parallel sub-agents for fan-out.** gstack's design-shotgun used multiple *models*;
  Soleur is all-Claude (ADR-053). Variants are generated by N parallel `Agent`-tool
  sub-agents, each seeded a distinct aesthetic direction from the list
  `frontend-design` already enumerates (brutalist, maximalist, editorial, refined-
  minimal, …). Real parallelism + genuinely distinct variants + agent-native.
- **Auto-supersede on contradiction.** When a new selection contradicts a recorded
  preference on the same axis, the contradiction is **detected and logged (the flag
  fires — satisfies the AC)**, then resolved by the newest selection superseding the
  old entry. Rejected: inline operator prompt (adds a mid-flow gate) and keep-both-
  let-decay-resolve (holds conflicting entries too long).

**Surface asymmetry (load-bearing).** The FR6 hook matches the `Skill` tool only, so:
- `frontend-design` (a **skill**) loads the taste-profile via FR6 — add its path to
  `context_queries`. Literally "loaded via FR6" per the AC.
- `ux-design-lead` (an **agent**, invoked via the Agent tool) is *not* reachable by
  the Skill-matcher hook — it gets an explicit "Read the taste-profile before
  designing" directive in its agent body. This is a direct Read, **not** a bespoke
  loader mechanism (the artifact + write path are shared; only load-injection differs
  by surface type).

## Key Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | Taste-profile lives at `knowledge-base/product/design/taste-profile.md` | Product/design domain; `ux-design-lead` already writes design artifacts under `knowledge-base/product/design/{domain}/` |
| D2 | Decay = write-time recompute (explicit `confidence` + `last_reinforced`, rewritten at session start) | FR6 is pointer-only; no runtime to compute decay at load — must be baked into committed values |
| D3 | Fan-out = N parallel `Agent`-tool sub-agents, each seeded a distinct aesthetic direction | All-Claude (ADR-053); real parallelism + distinct variants |
| D4 | Default N = 3 variants (parameterizable) | Balances diversity vs. cost; design-shotgun-typical |
| D5 | Contradiction resolution = auto-supersede; the flag still fires (detect + log the contradiction event) | Operator choice; satisfies AC "contradiction flag fires" without a mid-flow gate |
| D6 | `frontend-design` (skill) loads via FR6 `context_queries`; `ux-design-lead` (agent) loads via a direct-Read directive | FR6 Skill-matcher hook does not fire for agents — but both share one artifact + write path |
| D7 | Taste entries keyed by an **axis** (e.g. density, ornamentation, palette-temperature); contradiction = same axis, conflicting value | Gives contradiction detection a well-defined trigger |
| D8 | Write trigger = operator selecting a variant records it as evidence + recomputes confidence | The "learning" signal is the selection, per design-shotgun |
| D9 | No wireframes (Phase 3.55 skip) — internal tooling that *generates* UI, does not *add* a product UI surface | Consistent with epic's internal-plugin-tooling classification |

## Open Questions

- **OQ1 (shared write helper):** Should the taste-profile write (append evidence,
  recompute decay, detect+log contradiction, supersede) be a small shared jq+bash
  helper both surfaces call, or a documented schema each surface edits directly?
  Decide at plan time — a shared helper keeps the two write paths consistent but adds
  a script; direct markdown editing is lighter. Lean: shared helper if the decay math
  is non-trivial.
- **OQ2 (decay function):** exact decay curve + half-life (e.g. linear vs exponential,
  reinforce-on-reselect bump). Plan-time; must be deterministic (no `Date.now()` in
  workflow scripts — pass the date in).
- **OQ3 (contradiction axis vocabulary):** fixed enum of design axes vs. free-form
  tags. Fixed enum makes contradiction detection reliable; free-form is more
  expressive. Lean: small fixed enum seeded from `frontend-design`'s aesthetic list.
- **OQ4 (variant persistence):** do rejected variants get recorded as negative
  evidence, or only the selection as positive? Affects how fast taste converges.
- **OQ5 (base rebase):** the worktree branched from a `main` HEAD that predates
  PR #6035 — rebase onto current `origin/main` before implementation so the FR6 hook
  + `context_queries` idiom are present.

### Engineering (CTO — carry-forward + FR7 delta)

**Summary:** Epic assessment recommended taste-learning ride T2-6/FR6 (now landed,
PR #6035) rather than a bespoke loader. FR7 delta surfaced this session: FR6's
`Skill`-matcher hook does not reach the `ux-design-lead` **agent**, so the agent needs
a direct-Read directive while the `frontend-design` **skill** uses `context_queries`.
Decay must be write-time (pointer-only injection). Fan-out via parallel sub-agents fits
the all-Claude frame. No new shared-hook edits (unlike T2-8/T3-11), so blast radius is
low — additive frontmatter + agent-body + one committed artifact.

### Legal (CLO — carry-forward)

**Summary:** Epic CLO gate: redaction hardening (#5987) must precede any feature that
increases **data egress**. Taste-profile is committed to the repo (design-preference
metadata: aesthetic axes + selection evidence), not egressed to any external
sub-processor — so it does not trip the egress gate. No PII, no new sub-processor. Keep
selection-evidence entries free of operator-customer data (record the aesthetic choice,
not the content being designed).

## User-Brand Impact

- **Artifact:** the `taste-profile.md` artifact + its load-injection into the
  `frontend-design` skill and `ux-design-lead` agent.
- **Vector:** a malformed taste-profile write (bad decay math or a supersede that drops
  a valid entry) silently biases every future design session toward a wrong learned
  preference for the single operator — or a parse issue in the added `context_queries`
  path. FR6 is fail-open by construction (ADR-086: exit 0 every path, PostToolUse
  cannot fail-closed skills), which caps the loader-side blast radius.
- **Threshold:** single-user incident.
