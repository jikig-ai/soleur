# A watchdog-authored issue's root-cause hypothesis is a CLAIM, not a fact — verify via observability first

## Problem

`/soleur:go 6407` routed a P1 `ci/inngest-down` watchdog issue (the 6th recurrence in ~2 days) to `one-shot`.
The issue body — auto-generated by the watchdog — asserted a specific root cause and remediation:

> the durable backend likely lost its Redis (`inngest-redis.service` missing → crash-loop): the deploy must
> re-stage the redis assets, OR roll the `ExecStart` back to SQLite-only.

Building the fix on that hypothesis would have chased a **non-existent subsystem**.

## Investigation (self-pull, not operator-fetch — `hr-no-dashboard-eyeball-pull-data-yourself`)

Pulling the failed auto-restart's own diagnostic payload (`gh run view <id> --log-failed`) and Better Stack
(`scripts/betterstack-query.sh`) showed the issue body was stale:

- The restart died with `reason=lock_contention` — it could not acquire the `ci-deploy.sh` flock. **Not** redis.
- The same payload reported the host **healthy**: `inngest_server=active`, `restart_count:0`, `oom_killed:false`,
  `sandbox_canary=pass`, and the journal showed inngest **live-processing events** (`triage.p0p1_issue`,
  `heartbeat-recovery-verify`) *during* the "outage".
- The live `ExecStart` is `inngest start --sqlite-dir /var/lib/inngest --postgres-max-open-conns 5 …` — the durable
  backend is **SQLite + Postgres. There is no Redis in the current architecture** (the issue's redis framing was
  copied from an older incident era).

## Root cause (the real one)

A false-positive watchdog + a restart-lock bug, in a feedback loop:
1. The watchdog's cheap `/v0/gql functions` liveness read returned a transient `__FETCH_FAILED__`.
2. The liveness classifier declared full `inngest_down` **even though the server was active and serving** → filed a P1.
3. That dispatched a restart, which RED-failed on `lock_contention` (a concurrent deploy held the flock).

## Solution

Fixed the two real defects (not redis): (A) corroborate a failed functions-query against a loopback `/health` probe
before declaring down — 200 → new soft mode `functions_query_degraded` (no restart), routed through every watchdog
consumer + a persistence-escalation ceiling; (B) make the restart verify-poll treat `lock_contention` as non-terminal
(ADR-079 amendment #5960) with a final STATE re-read; (C) `SOLEUR_*` markers so both self-report. The PR body corrects
the issue's misframing.

## Key Insight

An issue body's stated root cause — **especially an auto-generated one from a monitor/bot** — is a starting
hypothesis to falsify against telemetry, exactly like `hr-verify-repo-capability-claim-before-assert` applied to
routing. The cheapest disproof is often the *failed recovery action's own structured payload*: it already contains
the real failure reason (`reason=lock_contention`) and a live service snapshot that contradicts the narrative. A
recurring monitor alert whose auto-remediation keeps failing is signal that the **classifier** is wrong, not just the
target service. Verify the claim before spending a plan→work→ship cycle on it.

## Session Errors

- **Draft-PR push failed on a transient DNS error** (`Could not resolve hostname github.com`) while `gh` API calls
  worked before and after. Recovery: continued (the skill tolerates this; `/ship` creates the PR later). Prevention:
  one-off transient network — no rule change; the draft-pr step already warns-and-continues.
- **First planning subagent died mid-`Write`** on a transient API "Server error mid-response", producing zero
  artifacts. Recovery: ran the one-shot partial-artifact check (found nothing), wrote the fallback session-state, and
  re-spawned a fresh planning subagent with the corrected root cause baked in. Prevention: already covered by
  one-shot's fallback path; the re-spawn (fresh, not resume) avoids inheriting broken partial state.
- **Implementation subagent stalled** (600s stream-watchdog, "no progress") at the `test-all.sh` full-suite gate,
  before committing its work. Recovery: verified the subagent's uncommitted working-tree changes against the
  correctness gates myself, then committed. Prevention: **do not run the full `test-all.sh` suite inside a subagent** —
  its long silent stretches trip the stream watchdog; run long suites as a detached parent `run_in_background` Bash
  command (which re-invokes on exit regardless of duration) and have subagents run only the fast, directly-affected
  suites.
- **Issue-body root cause was stale/wrong** (redis vs the real lock_contention + false-positive). Recovery: self-pulled
  the failed restart payload + Better Stack and re-scoped the plan before any code was written. Prevention: this
  learning + the existing `hr-verify-repo-capability-claim-before-assert` / self-pull-observability rules.
- **Orphan-suite gap**: the inngest `scripts/*.test.sh` (liveness-classify, restart-age-gate) were never registered in
  `test-all.sh` or any CI workflow — they gated nothing. Recovery: registered all three (incl. the new
  restart-poll-classify) in the scripts shard; full suite went 168→171. Prevention: when adding a sourceable
  `scripts/*.sh` helper with a `.test.sh`, register it in `test-all.sh`'s explicit `run_suite` list in the same commit
  (the runner is not a glob) — the #3533 orphan-suite class.
