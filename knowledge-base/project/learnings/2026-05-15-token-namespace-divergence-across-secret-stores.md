# Learning: token namespace divergence across secret stores

## Problem

While probing the Sentry residency contradiction (#3861), the Doppler `prd` `SENTRY_AUTH_TOKEN` returned HTTP 401 on `sentry.io` and HTTP 403 on `de.sentry.io` — consistent with a DE-region-bound token (auth recognized only on DE; scope insufficient for `org:read`). But the AC14 migration-audit artifact at `knowledge-base/legal/audits/sentry-migration-audit-2026-05-15.md`, generated by `.github/workflows/reusable-release.yml` using the *GitHub repo secret* `SENTRY_AUTH_TOKEN`, successfully enumerated 8 cron monitors on `sentry.io` (US).

A DE-region-bound token cannot return 200 on US `/organizations/jikigai/monitors/`. The only way both observations can be true is if the Doppler `prd` `SENTRY_AUTH_TOKEN` and the GitHub repo secret `SENTRY_AUTH_TOKEN` are **distinct token values** with distinct cluster scopes, despite sharing a name.

Additionally, `plugins/soleur/skills/postmerge/SKILL.md:188` references a *third* name: `SENTRY_API_TOKEN` (Doppler `prd`, used for debug-read of the Sentry API by the postmerge skill). The audit script at `apps/web-platform/scripts/audit-sentry-extra-text-references.sh:35-37` even falls back from `SENTRY_API_TOKEN` to `SENTRY_AUTH_TOKEN`, treating them as interchangeable when they may not be.

## Solution

Never assume same-name → same-value across secret-storage backends. Three principles:

1. **Inventory secrets across all backends before residency / scope reasoning.** For Sentry specifically: `gh secret list` for GitHub repo secrets + `doppler secrets --project soleur --config prd | grep SENTRY` for Doppler. Compare by name AND by token-prefix (`sntrys_` org-auth releases-only, `sntryu_` user-auth full-scope, `sntryi_` internal-integration token).

2. **Where the secret originates IS the cluster.** Sentry tokens are minted from one specific cluster's UI and only authenticate on that cluster. A token created at `de.sentry.io/settings/account/api/auth-tokens/` will 401 on `sentry.io` and vice versa. The cluster of origin is recoverable from the token-creation page that mints it; once it's stored as a plain secret value, the origin is unrecoverable from the token bytes alone.

3. **Rotation must be cluster-aware.** When rotating a Sentry token after a regional migration, the new token MUST be minted from the *new* cluster's UI. Re-using the same token name across Doppler `prd`, the GitHub repo secret, and any script-fallback (`SENTRY_API_TOKEN` ⇄ `SENTRY_AUTH_TOKEN`) silently masks cluster mismatches until a probe like #3861 surfaces them.

## Key Insight

Token-by-name comparison across secret-storage backends is **structurally unsafe**. The #3861 bug's root cause was that `reusable-release.yml:308` defaulted `SENTRY_API_HOST: ${{ secrets.SENTRY_API_HOST || 'sentry.io' }}` AND the GitHub repo secret `SENTRY_AUTH_TOKEN` happened to have access on US — the two values together silently routed IaC monitor provisioning to a parallel US-side `jikigai` shadow org while the public-facing Sentry SDK kept ingesting events to DE via the DSN. No single token rotation surfaces this; the divergence is invisible until both stores are inventoried and compared.

For the Sentry domain specifically:

- `SENTRY_AUTH_TOKEN` (Doppler `prd`): consumed by web-platform runtime + dev scripts; should be DE-region-bound.
- `SENTRY_AUTH_TOKEN` (GitHub repo secret): consumed by `apply-sentry-infra.yml` + `reusable-release.yml`; MUST be DE-region-bound after #3861.
- `SENTRY_API_TOKEN` (Doppler `prd`): consumed by postmerge skill + audit scripts as a fallback; cluster-bind unverified pre-#3861.

After #3861's Phase A2, all three should be the same DE-cluster token value (or distinct DE-cluster values, never US). The third (`SENTRY_API_TOKEN`) cleanup is housekeeping for a separate PR — not strictly required to close #3861.
