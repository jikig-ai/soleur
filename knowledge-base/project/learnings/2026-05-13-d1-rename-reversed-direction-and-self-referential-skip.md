# Learning: Plan-time research can reverse a brainstorm-time rename direction; mechanical-rename plans must anticipate planner-emitted artifacts in AC filters

## Problem

When #2725 (incident-commander skill) brainstorm proposed a D1 prerequisite to standardize the `brand_threshold` / `brand_survival` / `brand_survival_threshold` / `threshold` frontmatter-key vocabulary, the brainstorm's recommendation framed the rename direction as TOWARD `brand_threshold:` (singular). The rationale was that the dashboard-error-postmortem.md (the only worked PIR in repo) used that form.

The operator endorsed the brainstorm decision, and an issue (#3724) was filed prescribing the TOWARD-singular rename direction.

## What plan-time research revealed

At plan-time, a 30-second `git grep` count revealed the direction was wrong:

- `brand_survival_threshold:` (plural, snake-case) frontmatter appears in **97 files**.
- `brand_threshold:` (singular) appears in **1 file** — the canonical PIR file (`dashboard-error-postmortem.md`).
- `brand_survival:` appears in 5 files.
- Semantic `threshold:` in ~7 files (some load-bearing, must NOT rename).

Moreover, EXISTING TOOLING already canonicalized the long form:

- `plugins/soleur/skills/brainstorm/SKILL.md:398` MANDATES `brand_survival_threshold:` in spec frontmatter.
- `plugins/soleur/skills/preflight/SKILL.md:463` greps the prose form `**Brand-survival threshold:**`.
- `plugins/soleur/agents/engineering/review/user-impact-reviewer.md` triggers on `single-user incident` (space form).

Renaming TOWARD `brand_threshold:` would have touched ~110 files AND broken existing tooling conventions. Renaming AWAY from it (the dominant form was the target) touches ~13 files AND preserves the tooling integration.

The plan-time AskUserQuestion surfaced the conflict, operator chose REVERSE, and the spec / brainstorm / D1 issue body got reciprocal `[Updated 2026-05-13]` markers.

## Key insight

**Plan-time `git grep` of the rename-target form (count of occurrences) is the cheapest verification step that catches direction-wrong rename decisions made at brainstorm time.** The brainstorm operates from a specific artifact (the PIR) and infers canonical form from local context; the codebase corpus shows the dominant form is somewhere else. The brainstorm cannot easily verify "how often does each form appear?" without running the grep itself — and brainstorm-time research typically doesn't run that.

The fix is mechanical: when a brainstorm proposes "rename X to Y", the plan phase runs `git grep -l "^X:" | wc -l` AND `git grep -l "^Y:" | wc -l` AND grep-counts of any related forms. If the brainstorm-target form is the MINORITY form by ≥3x AND existing tooling references the MAJORITY form, the direction should be reversed at plan time with operator confirmation.

## Secondary insight: planner-emitted artifacts must be pre-listed in AC exclusions

For mechanical-rename PRs, the AC verification grep MUST exclude:

1. **Tag-array slug uses** (KB tag convention is hyphen-slugified by design) — well-known, plan AC3 anticipated this.
2. **Auto-generated artifacts** (`INDEX.md`, `kb-tags.txt`, etc. generated by `scripts/generate-kb-index.sh`) — these get refreshed post-rename and will contain the slug form for KB tag-array preservation. Predictable.
3. **Self-referential planner-emitted files** — the plan, tasks.md, and session-state.md that DESCRIBE the rename. These literally contain `single-user-incident` tokens in disposition tables, AC commands, and code-block examples; renaming them would corrupt their documentation of what was done.

The third class is non-obvious — it requires the plan author to anticipate "I'm writing a plan about renaming a token; this plan ITSELF contains the token in its examples." Without the pre-list, /work-time has to add the exclusion inline (~2 minutes), and code-simplicity reviewer flags it as a P3 plan-time-tightening miss.

## Recommendation for future rename-class plans

When drafting a plan that renames a token `X → Y` across the repo:

1. Run `git grep -l "^X" | wc -l` AND `git grep -l "^Y" | wc -l` AND any inflected forms (with/without trailing colon, with/without quoted form, with/without specific value-form prefix). If the brainstorm-time target is the MINORITY by ≥3x, surface to operator before drafting AC.
2. Anticipate AC filter exclusions for the three classes above (tag-arrays, auto-gen, self-ref planner-emitted). Pre-list each in plan AC3 (or equivalent).
3. The plan body's own description of the rename will contain literal token references — explicitly call out the self-ref skip in §FR2 disposition table.

## Tertiary insight: `worktree-manager.sh create` blocks when another worktree has `main` checked out

During one-shot Step 0b, `worktree-manager.sh create feat-one-shot-3724` failed with `fatal: 'main' is already used by worktree at '.worktrees/feat-cla-legal-rigor'`. The script's `git fetch origin main:main` step requires local main NOT to be checked out elsewhere; when another active session has main checked out (release-cut work, post-merge verification, etc.), the script aborts.

**Recovery:** Bypass with `git worktree add .worktrees/feat-one-shot-3724 -b feat-one-shot-3724 origin/main` — creates worktree from `origin/main` directly, never touches local main.

**Prevention:** Filed #3741 with proposed fix: `worktree-manager.sh create` should default to `origin/main` (not local main) so concurrent-session conflicts cannot block worktree creation.

## Session Errors

1. **worktree-manager.sh blocked by sibling session** — `feat-cla-legal-rigor` worktree was actively on `main` branch (1.2G of uncommitted work, HEAD ahead of dc7c8b71). Recovery: direct `git worktree add origin/main` bypass. **Prevention:** #3741 filed.
2. **Plan AC3 filter incomplete at draft time** — needed 2 new exclusions added at /work time (INDEX.md auto-gen, self-ref files). Recovery: applied inline; updated plan AC3 with corrected filter. **Prevention:** this learning's "Recommendation for future rename-class plans" section.
