# Learning: Plans Prescribing Test Frameworks Not Installed in the Repo

## Problem

During implementation of #2211 (kb-search frontmatter faceting), the plan specified:

- Files to Create: `tests/scripts/test-generate-kb-index.bats`
- Test Strategy table: all tests labeled as `bats`

But `bats` is not installed on this system and the repo convention is bash `.test.sh` files under `plugins/soleur/test/` using a shared `test-helpers.sh` — wired into `scripts/test-all.sh` via a `for f in plugins/soleur/test/*.test.sh` loop.

The plan also said "no new dependencies" in its Overview, creating an internal contradiction: bats would be a new dependency.

The implementer (work skill, Phase 2) caught this by running `which bats` and `bats --version`, both failed. Adapted by using the existing `.test.sh` convention. The test suite runs cleanly against the existing runner.

## Root Cause

The plan author prescribed a test framework without verifying:

1. Whether the tool was installed in the repo (`command -v bats`)
2. Whether existing test files use a different convention (`ls plugins/soleur/test/`)

Both checks take one bash command each. Neither is in the plan skill's Test Strategy instructions.

## Solution

Adapted at work-skill time:

- Created `plugins/soleur/test/generate-kb-index.test.sh` using `test-helpers.sh`
- Placed fixtures at `plugins/soleur/test/fixtures/kb-facets/`
- Test runs via `bash plugins/soleur/test/generate-kb-index.test.sh` and is picked up automatically by `scripts/test-all.sh`

The deviation was documented in a commit message and in the plan review summary.

## Prevention

Route-to-definition proposal: **plan skill Test Strategy section** gains a sharp edge:

> Before specifying a test framework (bats, pytest, vitest, etc.), verify it is installed in the repo:
>
> 1. Run `command -v <tool>` to confirm the binary exists
> 2. Grep `package.json`, `Gemfile`, or equivalent for the dependency
> 3. List existing test files (`ls test/`, `ls plugins/*/test/`) to match convention
>
> If the framework is absent but adjacent conventions exist, default to the existing convention. Never prescribe a new test framework without an explicit "Add <framework> dependency" task in the plan.

Route-to-definition proposal: **work skill Phase 1** already instructs "verify the plan does not contradict conventions in AGENTS.md and constitution.md" — extend this to also verify test-tooling availability when the plan's Test Strategy names a specific framework.

## Session Errors

- **Plan prescribed `bats` framework not installed in repo** — Recovery: adapted to `.test.sh` convention with fixtures under `plugins/soleur/test/fixtures/`. Prevention: plan skill's Test Strategy instructions should require `command -v <tool>` check before specifying a framework.

- **Committed twice without running compound between them** — Recovery: running compound now at Phase 4 handoff. Prevention: work skill Phase 2 "Incremental Commits" section should clarify that compound runs once at Phase 4 handoff; incremental commits during implementation are exempt from the compound-before-commit gate. The rule is otherwise recursive (compound creates commits).

## Key Insight

Plans that prescribe tooling without verifying availability create silent contradictions ("no new dependencies" + "use bats"). The implementer catches it but pays an attention cost. The cost is small per-instance but compounds across plans — and plans generated by agents without repo context are especially prone.

The fix is not "trust implementers to catch it." The fix is a 2-line check in the plan skill's Test Strategy section: verify availability before naming.
