# Troubleshooting: Plan Prescribed Wrong Test Runner Command

## Problem

The deepened plan for #1927 specified `bun test ci-tools github-api tool-tiers canusertool-tiered-gating` but the web-platform project uses vitest (configured in package.json `scripts.test`). The bun test command failed silently with "no test files found" even though all test files existed.

## Environment

- Module: System (cross-cutting workflow issue)
- Affected Component: web-platform test suite
- Date: 2026-04-11

## Symptoms

- `bun test ci-tools github-api` returns "The following filters did not match any test files"
- `bun test ./test/ci-tools.test.ts` also fails with same message
- Test files exist and are valid vitest test files

## What Didn't Work

**Attempted Solution 1:** Using bun test with filter names (as plan prescribed)
- **Why it failed:** bun test uses its own test runner, not vitest. The project's vitest config defines test file patterns and workspace settings that bun test does not read.

**Attempted Solution 2:** Using bun test with explicit file paths (`./test/ci-tools.test.ts`)
- **Why it failed:** Same reason — bun's built-in test runner has different file discovery than vitest.

## Session Errors

**Wrong test runner command in plan** — Plan specified `bun test` but project uses `vitest` (per package.json `scripts.test: "vitest"`). Recovery: Checked package.json, switched to `npx vitest run`. **Prevention:** Always check `package.json scripts.test` before running tests; never trust plan-prescribed test commands verbatim.

**Shell CWD confusion after `cd apps/web-platform`** — Subsequent commands using relative paths like `apps/web-platform/test/` failed because we were already inside that directory. Recovery: Used absolute paths from worktree root. **Prevention:** Use absolute paths in the Bash tool instead of `cd` into subdirectories; if `cd` is needed, always use absolute paths for the return.

## Solution

Check `package.json` for the actual test script before running tests:

```bash
# Check what test runner the project uses
cat apps/web-platform/package.json | grep -A1 '"test"'
# Output: "test": "vitest"

# Run with the correct runner
cd apps/web-platform && npx vitest run test/ci-tools.test.ts test/github-api.test.ts
```

## Why This Works

The plan was generated by an agent that used `bun test` (the project's runtime) instead of `vitest` (the project's test framework). While bun has a vitest compatibility layer, `bun test` uses bun's built-in test runner by default, not vitest. The `scripts.test` field in package.json is the source of truth for which test runner to use.

## Prevention

- Always read `package.json` `scripts.test` before running tests in a new project context
- Plans should reference `npm test` or `package.json scripts.test` rather than hardcoding a specific runner
- Use absolute paths in Bash tool calls to avoid CWD drift
