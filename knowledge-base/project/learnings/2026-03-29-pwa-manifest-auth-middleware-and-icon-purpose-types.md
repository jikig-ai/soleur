# Learning: PWA manifest route requires PUBLIC_PATHS entry and Next.js icon purpose types are strict unions

## Problem

Two issues discovered during PWA installability implementation:

1. **Manifest blocked by auth middleware**: The plan asserted that `/manifest.webmanifest` (generated by Next.js App Router from `app/manifest.ts`) "bypasses the matcher" because "the route handler runs without middleware intervention since it returns JSON, not HTML." This was wrong. The middleware matcher regex matched the path, and since `/manifest.webmanifest` was not in `PUBLIC_PATHS`, unauthenticated requests (Lighthouse audits, Android install checks, expired sessions) would redirect to `/login` instead of returning JSON.

2. **`"any maskable"` not a valid TypeScript type**: The W3C Web App Manifest spec allows space-separated `purpose` values like `"any maskable"`, but Next.js `MetadataRoute.Manifest` types define `purpose` as a strict union: `"any" | "maskable" | "monochrome" | undefined`. The plan prescribed `"any maskable"` which compiled with a TS error.

## Solution

1. Added `"/manifest.webmanifest"` to the `PUBLIC_PATHS` array in `middleware.ts`. This is preferred over adding to the matcher exclusion regex because PUBLIC_PATHS still applies CSP headers to the response, maintaining consistent security posture.

2. Split the 512x512 icon into two manifest entries: one with no `purpose` (defaults to `"any"`) and one with `purpose: "maskable"`. This gives browsers a large icon for both maskable contexts (Android adaptive icons) and non-maskable contexts (desktop shortcuts, splash screens).

## Key Insight

**Don't trust plan assertions about middleware behavior without verifying the regex.** The middleware matcher is a negative lookahead regex — it's easy to incorrectly reason about what it excludes. The plan's reasoning sounded plausible ("route handler returns JSON, not HTML") but conflated Next.js route handling with middleware execution order. In Next.js App Router, middleware runs before route handlers for all matched paths, regardless of response content type.

**W3C spec values ≠ framework type definitions.** When a plan references a spec value (like `"any maskable"`), verify it against the framework's TypeScript types before implementing. Next.js wraps the spec with stricter types.
