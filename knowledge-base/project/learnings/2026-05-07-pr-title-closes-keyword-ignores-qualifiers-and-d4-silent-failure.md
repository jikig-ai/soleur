# Learning: `Closes #N <qualifier>` in PR title auto-closes pre-merge, AND scheduled-workflow D4 abort-path neutralization silently fails

Two coupled bugs surfaced together in the post-merge dogfood for `--once` schedule neutralization (#3185, verifying #3155's D4 mechanism). One bug pre-empted the dogfood; the other invalidated the dogfood's success criteria.

## Problem 1 — GitHub auto-close keyword ignores qualifiers in PR titles

PR #3200 used the title:

```
chore(schedule): dogfood D4 neutralization for #3155 (Closes #3185 after fire)
```

The intent was "this PR doesn't actually close #3185 — the workflow will, after it fires." GitHub's keyword parser does not read English. It saw `Closes #3185`, ignored the parenthesized "after fire", and auto-closed #3185 the moment the PR merged (`2026-05-04T15:20:31Z`) — **17 hours before the workflow's first fire window**.

This is the same class of trap as `Closes #3185 partially` (already documented in AGENTS.md rule `wg-use-closes-n-in-pr-body-not-title-to`), generalized: any qualifying text *after* a `Closes #N` keyword in a PR title is ignored, including parenthetical conditions like "(after X)", "(only if Y)", or "(post-fire)".

### Cascade

When the scheduled workflow fired (`2026-05-05T10:42:24Z`), its pre-flight check #4 (`Issue OPEN + same repo`) found #3185 already CLOSED and routed to the abort path — which then triggered Problem 2.

## Problem 2 — D4 abort-path neutralization silently fails inside `claude-code-action@v1`

The pre-flight abort spec (line 171–173 of the workflow YAML) requires the agent to:
1. Post a single observation comment to the issue naming the failed check.
2. Invoke the Neutralization primitive (Read+Edit YAML, configure git identity, commit, `git push origin HEAD:main`, with PR-fallback on push failure, and a fallback comment if both legs fail).

**Observed end state of run `25371675500`:**

| Expected side-effect | Result |
|---|---|
| Observation comment on #3185 | ❌ Zero comments from agent |
| `schedule:` removed from YAML on main | ❌ Still present |
| `chore/neutralize-scheduled-dogfood-3155.yml-*` PR | ❌ Not created |
| Fallback "auto-cleanup failed" comment on #3185 | ❌ Not posted |
| Run conclusion | `success` (false success) |
| Telemetry | 4 turns, 15.4s, **`permission_denials_count: 1`** |
| Agent output | Hidden (`Running Claude Code via SDK (full output hidden for security)`) |

The agent ran. The agent exited cleanly. Nothing happened. We cannot see *which* tool call was denied because `claude-code-action` defaults to hiding the SDK transcript.

### Why this is brand-survival level

- **The cron is annual.** `0 9 5 5 *` fires every May 5th at 09:00 UTC. Without manual neutralization the workflow re-fires `2027-05-05`, `2028-05-05`, ... D3 (the date guard hardcoded to `FIRE_DATE: "2026-05-05"`) catches the date mismatch — but routes to the *same* neutralization primitive that just demonstrated silent failure. Annual silent recurrence.
- **The dogfood was supposed to validate D4 end-to-end.** D4 was introduced in #3155 to replace `gh workflow disable` (broken since #3153 — App-token doesn't honor `actions: write`). Every future `--once` schedule generated by `/soleur:schedule` relies on D4 self-cleaning. **D4 is now unverified, with one observed silent-failure data point.** Treat all `--once` schedules as not-self-cleaning until D4 is fixed.
- **Pre-flight aborts share the broken primitive with the success path.** If neutralization can silently fail, the abort path inherits that failure mode. A run can fail every pre-flight check and produce zero diagnostic output.

## Prevention strategies

### For the auto-close trap (Problem 1)

The existing rule `wg-use-closes-n-in-pr-body-not-title-to` covers this in spirit but only mentions `Closes #N partially` as the example. Adding to the schedule skill (and PR-creation guidance):

- **Never put `Closes #N` in a PR title with a qualifier.** GitHub ignores `(after fire)`, `(if foo)`, `(post-merge)`, `partially`, etc. If the issue should *not* close at merge, use `Ref #N` in the title and document the closing condition in the body.
- **For `--once` schedule PRs specifically**: the issue MUST stay open until the agent posts the result comment. PR title pattern: `chore(schedule): dogfood ... (Ref #N, agent will close on success)`.

### For the D4 silent-failure trap (Problem 2)

Domain-scoped to `plugins/soleur/skills/schedule/SKILL.md`, not AGENTS.md:

- **Pre-flight abort path must not depend on the same write primitive as the success path.** If the success path requires `git push HEAD:main`, the abort path needs a strictly weaker side-effect requirement (e.g., posting an observation comment via `gh api` is sufficient and uses a different token scope than a git push).
- **`claude-code-action` runs with hidden output by default** — for dogfood/verification workflows, opt into `show_full_output: true` or pipe the SDK output to a forensic artifact upload step. A success exit with hidden output and zero side-effects is the worst possible debug surface.
- **Dogfood-class workflows must verify side-effects, not exit codes.** "Run conclusion: success" is necessary but not sufficient — the verifier must read the actual issue/repo state for evidence the work happened.
