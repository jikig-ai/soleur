# Learning: GSC "Page with redirect" is usually Google's historical memory, not a live sitemap leak — verify the root-cause hypothesis against a real build before fixing

## Problem

A GSC Coverage drilldown reported a "Page with redirect" cluster for soleur.ai
(23 affected URLs). The intake brief carried a confident, code-read root-cause
hypothesis: the Eleventy sitemap template (`plugins/soleur/docs/sitemap.njk`)
filtered collection entries with `entry.url.endsWith("/") or entry.url.endsWith(".html")`,
and the `.html` clause was sweeping the legacy `/pages/*.html` redirect stubs
(generated by `page-redirects.njk` over `_data/pageRedirects.js`) into
`sitemap.xml`, so Google was indexing redirecting URLs.

The hypothesis was plausible from reading the template in isolation — but it was
never checked against the rendered output.

## Solution

**Build the site and inspect the actual artifact before touching code.**
`npx @11ty/eleventy` then `grep -oE '<loc>[^<]+</loc>' _site/sitemap.xml`:

- 48 `<loc>` entries, **all** trailing-slash apex (`https://soleur.ai/…`)
- **0** `.html` `<loc>`, **0** `/pages/` `<loc>`

The hypothesized leak did NOT reproduce. The redirect stubs carry
`eleventyExcludeFromCollections: true` (added in #1851), so they never enter
`collections.all` and the sitemap loop never sees them — the `.endsWith(".html")`
clause was **inert**. Cross-checking the GSC `Table.csv`: 14/23 affected URLs
were `www.soleur.ai/*` clean paths, 1 `/index.html`, 1 `?ref=peerlist`, 6
`/pages/*.html` stubs — all 301/3xx **correctly** to canonical apex URLs.

The work reframed from "fix the leak" to **regression-hardening**: (1) drop the
inert `.html` clause as defense-in-depth against a future re-leak (if someone
removes the exclude flag), (2) add a redirect-stub gate to the deploy/CI
validator `validate-seo.sh` (which previously *allowed* `.html` `<loc>`),
mirroring the existing canonical-host gate, (3) regression tests.

## Key Insight

1. **A GSC "Page with redirect" cluster is dominated by Google's historical
   memory** of URLs it learned before a canonicalization (www→apex, `.html`→clean,
   `?ref=` query forms). Those URLs 3xx correctly and the cluster self-heals
   server-side as Google re-crawls — there is usually no repo defect to chase.
   The actionable repo defect (if any) is narrower: a *redirecting* URL present
   in the *sitemap* itself. Verify that specific condition against the built
   `sitemap.xml`; do not treat the GSC URL list as a bug list.

2. **A confident, code-read root-cause brief is a hypothesis, not a fact.**
   When a "fix" brief names an exact mechanism, the cheapest first move is to
   reproduce it against the live artifact (build the site, grep the output;
   run the failing query; reproduce on a real row). Here the premise reversed
   on the first build. Reframing fix→regression-hardening when the premise
   doesn't reproduce is a legitimate, value-preserving outcome — the gate that
   would have caught a future regression is the shippable value.

3. **An OR-predicate gate needs each arm independently covered.** The redirect-
   stub gate is `(\.html$|/pages/)`. A single `/pages/pricing.html` negative
   fixture satisfies BOTH arms at once, so a regex-narrowing regression (drop
   the `\.html$` arm) would stay green. Add a root-level `about.html` (no
   `/pages/`) case so each arm is guarded — same defect class as the
   "single-literal gate over a multi-member union" review pattern.

## Session Errors

1. **CWD/worktree resolution trap** (plan phase, recovered) — Bash per-call CWD
   resolved to the bare-root mirror, not the worktree; first `Write` was blocked
   by the worktree-write guard, re-issued to the explicit worktree path.
   **Prevention:** already hook-enforced; chain `cd <worktree-abs> && <cmd>` in
   every bash call (subagents do not inherit the parent's `cd`).
2. **Content-guard false trigger** (plan phase, recovered) — a literal
   `doppler secrets set` phrase in prose asserting the change does NOT mutate
   secrets tripped the IaC-routing content guard. **Prevention:** describe the
   absence of secret mutation without quoting the mutating command literal.
3. **Plan snippet vs disk drift** (work phase, recovered) — the plan's
   illustrative "after" code block for `sitemap.njk` showed the clause already
   dropped; the first `Edit` matched the "after" form and failed against the
   real "before" on disk. **Prevention:** match `old_string` to current disk
   content (which I had Read), never to a plan's target-state snippet.
4. **`&&`-chain early-exit on `grep -c` zero-match** (work phase, expected) — a
   verification chain stopped when `grep -c` returned 0 (exit 1). **Prevention:**
   append `|| true` or split count-checks into separate commands.
