# Learning: SDK v0.2.80 canUseTool / PreToolUse allow-branch ZodError

## Problem

Users updating `knowledge-base/overview/vision.md` from a KB Tree PDF upload via the web UI saw the dashboard agent refuse with a surfaced `ZodError: invalid_union — permission request validation is failing internally` on both Write and Edit, plus a secondary `Read-only file system` error when the agent fell back to Bash `echo >`. The agent's self-explanation labelled it a platform bug and suggested the user use `update-config` to widen permissions (which cannot work because the SDK sandbox is configured with `settingSources: []`).

Screenshots showed the exact ZodError text inside the chat transcript — the Zod issue path was leaking to the user.

## Root Cause

Two converging SDK-level issues in `@anthropic-ai/claude-agent-sdk@0.2.80`:

1. **PreToolUse hook allow-path schema.** Our `sandbox-hook.ts` returned a bare `{}` on allow paths. The SDK's runtime Zod discriminated-union for `SyncHookJSONOutput` produced `invalid_union` on that shape in some Write/Edit call paths — public docs show `{}` as valid, but the runtime schema in 0.2.80 was stricter than the published `.d.ts`.
2. **`canUseTool` PermissionResult allow-branch.** Our callback returned `{ behavior: "allow" as const }` without `updatedInput`. Public docs at `code.claude.com/agent-sdk/permissions` declare `updatedInput` as required on allow; our pinned `.d.ts@0.2.80` declared it optional. The runtime schema followed the docs, not the `.d.ts`, so every file-tool call produced a schema failure.

SDK changelog between 0.2.80 and 0.2.85 documents two upstream fixes directly in this surface: v0.2.81 fixed a `canUseTool` bug; v0.2.85 fixed "PreToolUse hooks with `permissionDecision: 'ask'` being ignored in SDK mode". Our pinned version sat exactly in the broken window.

Two orthogonal contributors amplified the user-visible symptom:

- The workspace provisioner created `knowledge-base/project/*` but not `knowledge-base/overview/`. When the agent tried to Write vision.md, `isPathInWorkspace` had to walk up from a non-existent parent — the walk worked correctly, but any intermediate `ELOOP`/`EACCES` would have returned `null` and amplified the deny.
- `buildVisionEnhancementPrompt` emitted a relative path `knowledge-base/overview/vision.md`, which resolves against the agent's CWD at tool-call time — brittle inside an SDK-managed sandbox.

## Solution

Four layered fixes, each independently valid but bundled because each contract between SDK and platform needed sealing:

1. **Upgrade `@anthropic-ai/claude-agent-sdk` from 0.2.80 to 0.2.85 (exact pin).** This alone likely fixes the ZodError.
2. **sandbox-hook.ts: explicit PreToolUse allow.** Replace `return {}` with `return { hookSpecificOutput: { hookEventName: "PreToolUse", permissionDecision: "allow" } }`. Matches both permissive and strict schemas.
3. **agent-runner.ts: echo `updatedInput: toolInput` on every `canUseTool` allow branch.** Extracted into an `allow(toolInput)` helper at module scope so future branches cannot silently regress. Behaviorally a no-op — the SDK already uses the original `toolInput` — but satisfies the `PermissionResult` discriminated union.
4. **workspace.ts: pre-create `knowledge-base/overview/`** in both `provisionWorkspace` and `provisionWorkspaceWithRepo` so path validation never hits a missing ancestor.
5. **vision-helpers.ts: `buildVisionEnhancementPrompt` emits the absolute workspace path** so Write/Edit always receive a fully-qualified `file_path`.

## Key Insight

**When an SDK exposes both a TypeScript `.d.ts` and a runtime Zod schema, the runtime schema is authoritative and may disagree with the `.d.ts`.** Public docs typically mirror the runtime schema, not the `.d.ts`. When a schema field is marked optional in the `.d.ts` but required in the docs, treat it as required in production code — echoing input back on allow is a cheap defensive posture that satisfies both contracts.

**Corollary: always extract a helper when a discriminated-union return value is constructed at 3+ call sites.** The `allow(toolInput)` helper added in this PR prevents the single-branch-forgets-updatedInput regression class entirely.

## Session Errors

1. **Planning subagent hit usage limits (104 tool uses, 16 min)** before emitting the Session Summary return-contract. **Recovery:** plan + deepen-plan artifacts had persisted to disk; continuation read from disk instead of from subagent stdout. **Prevention:** the one-shot and work skills should capture the plan file path into `session-state.md` *before* the subagent's final write, so a mid-run cutoff still leaves recoverable state. Propose a skill-instruction edit to `one-shot/SKILL.md` Phase 1-2: log the plan-file path to `session-state.md` *as soon as* the `soleur:plan` sub-step completes, not only after `soleur:deepen-plan` finishes.
2. **bun and npm resolved `^0.2.85` to different versions** (bun → 0.2.104, npm → 0.2.85). **Recovery:** pinned with `npm install --save-exact 0.2.85` and re-ran both installers. **Prevention:** AGENTS.md rule `cq-before-pushing-package-json-changes` already says regenerate both lockfiles. Extend it: when pinning an exact version, use `--save-exact` so the caret is absent and both resolvers agree.
3. **`cd apps/web-platform && <cmd>` failed with "No such file or directory"** because my CWD was already inside that directory. **Recovery:** ran `pwd` and issued the command from the actual cwd. **Prevention:** prefer absolute paths in Bash commands when working in a worktree, or check `pwd` before chained `cd`. This is a subset of `hr-mcp-tools-playwright-etc-resolve-paths` — generalize to "always use absolute paths in worktrees, including for Bash `cd`".
4. **`gh issue create --milestone 6` rejected with `'6' not found`**. **Recovery:** switched to `--milestone "Post-MVP / Later"` (title string). **Prevention:** `gh issue create --milestone` expects the milestone *title*, not the number returned by `gh api .../milestones`. Add a note to the issue-creation path of any skill that currently calls `gh issue create` with numeric milestones.
