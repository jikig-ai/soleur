# GDPR Art. 22 — PA-22 (Anthropic SDK leader-prompt runtime)

Per the PR-B (#4379) brainstorm, the spec Non-Goal #3 deferred a "full
Art. 22 automated-decisioning analysis". PR-B reviewer feedback redirected:
ship the brief assessment inline; defer only the depth-analysis that
requires post-merge dogfood data or cohort-onboarding risk profiles.

This is the brief assessment. It produces a conclusion + safeguards
inventory; it does NOT produce the full 7-step worksheet that
`knowledge-base/legal/dpia/PA-22-anthropic-leader-loop.md` (Non-Goal #1,
filed as a follow-up issue) is scoped to deliver.

## Q1. Is the leader-loop output a "decision"?

**Yes.** Per-class artifacts materially influence downstream outcomes:

- `engineering.pr_review_pending` — code-line review comments shape PR
  merge outcomes.
- `engineering.ci_failed` — triage comments shape downstream debugging.
- `triage.p0p1_issue` — label + comment route issue triage assignment.
- `security.cve_alert` — branch + draft PR opens a code change for
  operator review.
- `knowledge.kb_drift` — branch + commit fixes a manifest the operator
  hosts.

## Q2. Is the decision "solely automated"?

**No — meaningful human involvement at click-time + post-artifact review.**

The triggering input is the operator's click on the Today card. The
operator chose the action class, sees the source object, and could
discard the card instead of clicking Spawn. After the artifact lands,
the operator (and any GitHub reviewer downstream) reviews it on GitHub
before any merge / triage / patch is accepted. The autonomous portion
is the *drafting* of the artifact, not its *acceptance*.

Per Art. 22 doctrine (and CJEU C-634/21 SCHUFA Holding), a "solely
automated" decision requires the absence of meaningful human review
between the algorithmic output and the legal effect on the data subject.
PR-B's loop does NOT meet that threshold under the operator-as-controller
shape.

## Q3. Does it produce "legal effects" or "similarly significant" effects?

**For the operator-themselves (data subject under (c) of PA-22):** no.
Cost is borne under their BYOK + their own contract; the operator IS
the controller of their own repo.

**For third-party committers (commit authors whose content is piped to
Anthropic under (g)6 PII-scrub):** unlikely to rise to Art. 22's bar:

- A code-line review comment is reputational at most; the contributor
  is identifiable on GitHub already (the commit is public).
- An issue label / triage routing is internal-repo metadata that does
  not legally or significantly affect the contributor.
- A draft PR or branch is operator-controllable (Undo / Stop) and
  cannot land without operator-or-reviewer approval.

The borderline case is the `security.cve_alert` flow which opens a
draft PR against the operator's own repo with a manifest bump —
contributor-authored code is referenced but not modified; the PR
remains in draft until operator review.

## Q4. Safeguards (relied-on TOMs)

PR-B ships these load-bearing safeguards already:

1. **Operator Stop button** (AC13) — operator can cancel the loop
   within one turn boundary.
2. **Operator Undo button** (AC14) — reverses every reversal handle
   with per-element ledger + 207 Multi-Status partial-failure
   transparency.
3. **Per-class tool allowlist** (AC8) — model cannot invoke a tool
   outside the enumerated per-class set without short-circuiting with
   `failure_reason = "leader_tool_invalid"`.
4. **Per-spawn $2.00 cost ceiling** (AC15) — caps the maximum work an
   autonomous loop can perform per click.
5. **8-turn ceiling** (LEADER_MAX_TURNS) — additional backstop.
6. **PII-scrub at prompt assembly** (AC16) — third-party commit-author
   emails redacted before Anthropic call.
7. **LEADER_CLASSES_DISABLED kill switch** — Doppler-config short-
   circuit for any class surfacing quality or safety issues.
8. **Operator audit surface** — Today card surfaces failure reasons
   with operator-readable copy (no raw `failure_reason` leaks per CPO-2).

## Conclusion

Under the operator-as-controller-on-own-repo shape that PR-B ships,
**Art. 22 does NOT apply** because the decision is not solely automated
(operator clicks → review on GitHub) and the effects on third-party
data subjects (commit authors) are not "legal" or "similarly
significant" given the reversible, draft, operator-reviewed artefact
shape.
