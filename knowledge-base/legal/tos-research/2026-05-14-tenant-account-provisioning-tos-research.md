# ToS research — Tenant account provisioning

This artifact is the Phase 0 gate for plan `2026-05-14-feat-soleur-managed-deploy-substrate-v1-scaffolding-plan.md`. It records whether each upstream provider's Terms of Service permits Soleur (acting as agent for an authorized operator-tenant) to provision a tenant-isolated account or sub-resource. For v1, the only contracting party is **Jikigai SARL** (Soleur-as-tenant-zero); the analysis below is therefore conservatively framed against the v1 single-tenant scope. Future non-Soleur tenants will each carry their own Tenant DPA (per Phase 2 runbook Step 0) under which the tenant — not Jikigai — is the contracting party with the upstream provider.

Verdict sentinels are greppable:

```text

Verdict: ALLOWED_WITH_CONDITIONS
Verdict: NOT_ALLOWED
```

Plan acceptance criterion: at least four `Verdict: ALLOWED*` lines (one per provider).

## 1. Hetzner Cloud — sub-project provisioning

**Provider**: Hetzner Online GmbH (DE).
**Provisioning shape**: Create a Hetzner Cloud sub-project under the operator-tenant's master account; mint a project-scoped API token; embed the token (or `hetznercloud/tps-action`-minted short-lived token) in the operator-tenant's GitHub repository secrets.

**Cited documents**:

- Hetzner Cloud — Terms and Conditions: `https://www.hetzner.com/legal/cloud-server/`
- Hetzner — General Terms and Conditions: `https://www.hetzner.com/legal/general-terms-and-conditions/`
- Hetzner Cloud — System Policies (acceptable use): `https://www.hetzner.com/legal/system-policies/`
- Hetzner — Data Privacy: `https://www.hetzner.com/legal/privacy-policy/`

**Analysis**: Hetzner Cloud terms define a binary contracting relationship between Hetzner and the Customer; sub-projects are an account-internal organizational unit and not a separate ToS-level entity. For v1 (Soleur-as-tenant-zero), Jikigai is the Customer and the sub-project sits inside Jikigai's own account — wholly permitted by the standard Customer terms.

For future non-Soleur tenants, the contracting party shifts to the tenant. Hetzner does not appear to prohibit a tenant from authorizing a third-party agent (Soleur) to create sub-projects on their behalf, provided the tenant has signed the Hetzner Cloud Terms and the tenant remains the Customer of record. The Hetzner cloud-server terms do not contain an explicit prohibition on agent-assisted account administration; absence of prohibition is conservatively read as ALLOWED_WITH_CONDITIONS pending counsel confirmation.

**Conditions** (Soleur-side operational, not ToS-imposed):

1. Tenant must independently accept Hetzner's Customer-facing terms; agent (Soleur) does not "accept on behalf of" the tenant. Phase 2 runbook Step 1 documents this.
2. The Hetzner Cloud Terms govern the tenant's master account; Soleur as agent operates strictly within the project-scoped API token's blast radius (single sub-project, no cross-project privileges).
3. Token lifecycle: Soleur as agent must not persist long-lived Hetzner tokens on Soleur infrastructure (the substrate's hard credential-aggregation constraint — see brainstorm Phase 0.1). Plan's Phase 2 Step 6 uses `hetznercloud/tps-action` short-lived tokens; the long-lived `HCLOUD_TOKEN` repo secret lives in the **tenant's** GitHub repo, never in Soleur's Doppler.

**Operator action at activation**: First non-Soleur tenant onboarding triggers a counsel-led re-read of the then-current Hetzner Cloud Terms specifically against the agent-provisioning shape.

**Verdict: ALLOWED_WITH_CONDITIONS** — v1 (Soleur-as-tenant-zero) is in-scope of the standard Hetzner Customer terms. Counsel re-verify before first non-Soleur tenant. Compliance-posture.md Active Items records the re-evaluation trigger.

## 2. Cloudflare — account / scoped sub-account provisioning

**Provider**: Cloudflare Inc (US).
**Provisioning shape**: For v1 (Soleur-as-tenant-zero), one Cloudflare account under Jikigai's name with a scoped account-API token (Workers Deploy + Pages Deploy + the specific zone only). Cloudflare's Tenant Platform and Enterprise resale tiers offer a richer multi-tenant model, but those tiers are out of scope at v1 (no contract).

**Cited documents**:

- Cloudflare — Self-Serve Subscription Agreement: `https://www.cloudflare.com/terms/`
- Cloudflare — Website Terms of Use: `https://www.cloudflare.com/website-terms/`
- Cloudflare — Tenant Platform documentation (not contracted at v1, listed for completeness): `https://developers.cloudflare.com/cloudflare-for-platforms/`
- Cloudflare — Acceptable Use Policy: `https://www.cloudflare.com/acceptable-use-policy/`
- Cloudflare — DPA (Data Processing Addendum): `https://www.cloudflare.com/cloudflare-customer-dpa/`

**Analysis**: At v1 (Soleur-as-tenant-zero) the account belongs to Jikigai under Cloudflare's Self-Serve Subscription Agreement. Scoped account-API tokens are a first-class Cloudflare feature; their use by automation (including GitHub Actions deploy workflows) is the documented and supported pattern (`developers.cloudflare.com/workers/ci-cd/external-cicd/github-actions/`). Wholly permitted.

For future non-Soleur tenants, two shapes are available:

- (a) Each tenant signs their own Self-Serve Subscription Agreement and Soleur as agent helps the tenant mint a scoped account-API token. The token lives in the tenant's GitHub repo secrets, never in Soleur's Doppler. This shape inherits the same v1 ToS posture and is the substrate's default per plan Phase 2 Step 2.
- (b) Cloudflare Tenant Platform — Soleur could in principle become a Cloudflare Tenant Platform partner, holding a parent account under which tenant accounts are provisioned via Cloudflare's Tenant API. This shape changes Soleur's role from agent to partner/reseller and triggers an entirely different ToS regime (Tenant Platform Terms, separate from Self-Serve). **Explicitly out of scope at v1**; captured as an open escape hatch in ADR-030 with re-evaluation trigger "Soleur exceeds 5 active non-Soleur tenants under shape (a)".

**Conditions**:

1. Shape (a) only at v1. Shape (b) requires a separate Cloudflare partner agreement and is not pursued.
2. Scoped account-API token must follow least-privilege (Workers Deploy + Pages Deploy + zone-specific); Phase 2 runbook Step 2 documents the exact scope.
3. Rotation cadence quarterly (documented in offboarding runbook).

**Verdict: ALLOWED_WITH_CONDITIONS** — v1 shape (a) is in-scope of the standard Self-Serve Agreement; shape (b) Tenant Platform is deferred to ADR-030's open escape hatches.

## 3. Doppler — project + OIDC service-account identity provisioning

**Provider**: Doppler Inc (US).
**Provisioning shape**: Create a Doppler project (`prd_tenant_<id>` config) under the operator-tenant's master Doppler account; configure a Doppler Service Account Identity for OIDC trust against the tenant's GitHub repository.

**Cited documents**:

- Doppler — Terms of Service: `https://www.doppler.com/legal/terms`
- Doppler — Data Processing Addendum: `https://www.doppler.com/legal/dpa`
- Doppler — Acceptable Use Policy: `https://www.doppler.com/legal/aup`
- Doppler — OIDC service-account identities documentation: `https://docs.doppler.com/docs/service-account-identities`
- Doppler — GitHub OIDC examples: `https://docs.doppler.com/docs/github-oidc-examples`

**Analysis**: Doppler's documented OIDC service-account-identity feature is purpose-built for trust-based access from CI systems (including GitHub Actions). For v1 (Soleur-as-tenant-zero) Jikigai is the Doppler Customer; the project sits inside Jikigai's account. For future non-Soleur tenants, the tenant becomes the Customer; Soleur as agent walks the tenant through OIDC identity setup in the runbook. Doppler's ToS does not appear to restrict third-party agent assistance in account setup, and the OIDC pattern is the documented best-practice (the very pattern Doppler markets via its GitHub OIDC examples).

**Conditions**:

1. Doppler DPA must be in force for any non-Soleur tenant whose data flows through Doppler secret storage. v1 (Soleur-as-tenant-zero) is covered by Jikigai's existing Doppler DPA (verified 2026-05-05, see `compliance-posture.md`).
2. The tenant must independently accept Doppler's Customer terms; agent (Soleur) does not accept on behalf of the tenant.
3. Service-account-identity rotation per Doppler's documented cadence.

**Verdict: ALLOWED_WITH_CONDITIONS** — v1 fully in-scope; non-Soleur tenant DPA-verification gate at first onboarding (runbook Step 0).

## 4. GitHub — repository creation + GitHub App installation

**Provider**: GitHub Inc (US).
**Provisioning shape**: Create a GitHub repository under the operator-tenant's organization; install the Soleur GitHub App (slug `app/soleur`) with narrow scope (`actions: write` + `metadata: read`, repo-pinned not org-wide); configure GitHub Environments + required reviewers + branch policy.

**Cited documents**:

- GitHub — Terms of Service: `https://docs.github.com/en/site-policy/github-terms/github-terms-of-service`
- GitHub — Customer Agreement (Enterprise): `https://docs.github.com/en/site-policy/github-terms/github-customer-agreement`
- GitHub — Acceptable Use Policies: `https://docs.github.com/en/site-policy/acceptable-use-policies`
- GitHub — Customer Data Protection Agreement: `https://github.com/customer-terms/customer-data-protection-agreement`
- GitHub Apps — installation permissions: `https://docs.github.com/en/apps/creating-github-apps/registering-a-github-app/choosing-permissions-for-a-github-app`
- GitHub Environments — deployment protection: `https://docs.github.com/actions/managing-workflow-runs/reviewing-deployments`

**Analysis**: GitHub ToS §B.3 ("Account Requirements") prohibits accounts "registered by 'bots' or other automated methods" but explicitly permits **machine accounts** "set up by an individual human who accepts the Terms on behalf of the Account, provides a valid email address, and is responsible for its actions." GitHub Apps installed on a tenant's repository are the canonical pattern for agent-assisted automation: the installation is performed by a human acting under the tenant org's authority, after which the App acts on behalf of the installation per the App's declared permissions.

Soleur's design (the `app/soleur` GitHub App with narrow `actions: write` + `metadata: read` scope, repo-pinned, installed by the tenant org owner) fits this pattern exactly. For v1 (Soleur-as-tenant-zero), Jikigai performs the install on its own org. For future non-Soleur tenants, the tenant org owner performs the install — Soleur as agent walks them through it via the runbook but does **not** install on their behalf.

**Conditions**:

1. Installation must be performed by the tenant org owner (human), not by Soleur automation. Runbook Step 4 documents this gate.
2. App slug appears as `app/soleur` in PR author logs (NOT `*[bot]`); any tenant-side allowlists matching `*[bot]` will silently exclude Soleur — runbook calls this out explicitly.
3. Customer DPA in force per `compliance-posture.md` Vendor DPA Status (GitHub Customer DPA at `github.com/customer-terms/customer-data-protection-agreement`).
4. Repo-pinned (not org-wide) installation enforced as a hard ceiling on blast radius.

**Verdict: ALLOWED_WITH_CONDITIONS** — GitHub Apps + machine-account pattern is the canonical agent-assisted-automation shape; tenant org owner must perform the install (cannot be automated).
