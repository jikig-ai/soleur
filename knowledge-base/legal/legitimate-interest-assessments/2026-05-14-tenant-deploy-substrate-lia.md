# Legitimate Interest Assessment — Tenant deploy substrate (orchestration-plane meta-audit log)

This LIA documents the Art. 6(1)(f) three-part test for the **orchestration-plane meta-audit log** that anchors the v1 multi-tenant deploy substrate (plan `2026-05-14-feat-soleur-managed-deploy-substrate-v1-scaffolding-plan.md`, issue #3723). The processing activity is the act of recording — into the `public.tenant_deploy_audit` table — that a Soleur agent triggered a `workflow_dispatch` on a tenant's GitHub repository on behalf of an authorized operator.

Per ICO and CNIL guidance on Art. 6(1)(f), a legitimate interest assessment must address three limbs: **purpose**, **necessity**, and **balancing**. Each is addressed below in a dedicated `## Purpose` / `## Necessity` / `## Balancing` H2 section (greppable for AC).

## Purpose

The processing serves **two cumulative legitimate purposes**:

1. **Operate the orchestration plane.** Soleur agents act on behalf of operator-tenants to dispatch GitHub Actions workflows that deploy each tenant's application stack (per ADR-030 and plan §1). The orchestration plane requires a record of "Soleur agent triggered deploy for founder X on tenant repo Y at time T" so that:
   - Operator-tenants can audit which deploys were triggered on their behalf (Art. 28(3) sub-processor instruction evidence).
   - Soleur's CLO/CTO rotation can reconstruct dispatch history during incident response.
   - The substrate provides operator-side visibility into the same data that GitHub already records server-side on each tenant repo (mirroring the controller's view of what the tenant's processor recorded).

2. **Evidence Art. 28(3) sub-processor instructions and Art. 32 TOMs.** Where Soleur is acting as a processor or joint-controller for tenant deploys (analysis at first non-Soleur tenant DPA), Art. 28(3)(a) requires the controller's processing instructions to be evidenced. The audit log is the primary evidence that each `workflow_dispatch` action was traceable to an authorized operator decision rather than an autonomous agent action. Art. 32 requires "ability to demonstrate" appropriate TOMs; the WORM-protected audit log evidences both access-control (RLS zero policies, service_role-only writes) and integrity (BEFORE UPDATE/DELETE triggers).

The purpose is **specific, explicit, and legitimate**:

- **Specific**: limited to recording the dispatch event and its outcome — not the workflow's payload, not the deployed application's data, not end-user data flowing through the tenant's app.
- **Explicit**: stated in advance via this LIA + ADR-030 + the Article 30 register row.
- **Legitimate**: necessary for the substrate to function as a multi-tenant orchestration plane; without an audit log, Soleur could not evidence its sub-processor obligations and the tenant could not exercise its Art. 28(3)(a) right to documented processing instructions.

**Not pursued under this LIA**: any processing of the deployed application's end-user data. End-user data does **not** flow through the substrate at v1 — only operator-side metadata (founder_id, target_repo, target_workflow, gh_run_id, oidc_jti, trigger_outcome).

## Necessity

The processing is **necessary** — meaning a less intrusive alternative cannot achieve the same legitimate purpose — for the following reasons:

1. **Alternative A — no audit log**: Rejected. Without a Soleur-side audit log, the controller has no way to evidence Art. 28(3)(a) processing instructions or Art. 32 TOMs. The tenant would have to rely solely on GitHub's audit log on the tenant repo — which captures **GitHub's view** of the dispatch (an API call from the Soleur App's installation token) but not Soleur's intent-side context (which Soleur agent decision led to the dispatch, which conversation context, which authorized-operator session). Incident reconstruction would degrade from "minutes" to "hours of cross-system correlation."

2. **Alternative B — audit log on tenant infrastructure**: Rejected as a *substitute*. Logging only on the tenant's side would force every controller-side audit and incident-response query to enumerate every tenant — N queries instead of 1 — which (i) creates a cross-tenant data-access pattern that re-introduces the credential-aggregation risk the substrate exists to prevent and (ii) makes orchestration-plane health monitoring (e.g., dispatch-failure rate across tenants) operationally intractable. The Soleur-side log is **additive** to the GitHub-side log on each tenant repo; both are necessary for their respective audit purposes.

3. **Alternative C — anonymous / aggregated logs only**: Rejected. The audit purpose requires per-event attribution to a specific founder_id (the data subject who authorized the dispatch). An aggregate-only log cannot evidence Art. 28(3)(a) at the granularity the regulation requires ("on documented instructions from the controller" — instructions are per-event).

4. **Data minimisation within the necessary processing**: Applied. The schema records only:
   - `founder_id` (UUID, pseudonymous controller-side identifier)
   - `event_type` (3 enum values only: `workflow_dispatch_triggered` | `workflow_run_completed` | `workflow_run_failed`)
   - `target_repo` (length- and charset-CHECK-constrained text; minimisation against log injection)
   - `target_workflow` (same)
   - `gh_run_id` (non-PII bigint)
   - `oidc_jti` (non-PII text; RFC 7519 §4.1.7 case-sensitive string)
   - `trigger_outcome` (4 enum values)
   - `event_at` + `retention_until` (timestamps; retention basis is the column not a proximity sweep)
   No request bodies. No workflow payloads. No IP addresses. No end-user data. The schema is the minimum that satisfies the audit purpose.

5. **Retention proportional to purpose**: 12 months (per legal-compliance reviewer SHOULD #2 — down from the original draft's 24 months). Aligned with Processing Activity 8's P0-mirror retention target. Orchestration-plane events do not require longer retention; statutory limitation periods for sub-processor evidence are addressed by aggregate operational logs (Hetzner stdout) which are out of scope of this LIA.

**Conclusion**: the processing is necessary for the legitimate purpose; minimisation has been applied at schema, retention, and access-control layers; less intrusive alternatives were considered and rejected for the reasons above.

## Balancing

The balancing test weighs the controller's legitimate interest (above) against the rights and freedoms of the data subjects (founder-operators). Per WP29 Opinion 06/2014 §III.3, the balancing test considers: (i) nature of the interests, (ii) nature of the data, (iii) reasonable expectations of the data subject, (iv) impact on the data subject, (v) safeguards.

### (i) Nature of the controller's interests

- **Legitimate operational interest** in operating a multi-tenant orchestration plane safely.
- **Statutory interest** in evidencing Art. 28(3)(a) sub-processor instructions and Art. 32 TOMs.

### (ii) Nature of the data subject's interests

- The data subjects are **founder-operators** — adult professionals actively using the Soleur platform under a contractual relationship governed by `docs/legal/terms-and-conditions.md`. They are not vulnerable data subjects; the data subjects are themselves the operators who *cause* the processing by triggering deploys.

### (iii) Nature of the data

- Pseudonymous controller-side identifier (`founder_id` UUID) + operational metadata (repo name, workflow name, run ID, outcome).
- **No special-category data (Art. 9)**. **No criminal-offence data (Art. 10)**. **No financial data**. **No location data**. **No biometric or health data**.
- Risk profile: low — disclosure of the audit log would reveal **which** operator deployed **what** to **which** tenant repo at **what time**, but would not reveal application contents, secrets, or end-user data.

### (iv) Reasonable expectations of the data subject

- A founder-operator actively triggering a deploy via Soleur agents has a **strong reasonable expectation** that the action will be recorded server-side — this is the canonical pattern for any operator-facing CI/CD platform (cf. GitHub Actions audit log, GitLab pipeline log, CircleCI audit log).
- The audit log records the **operator's own action** (triggering a deploy). The data subject is also the operator; the processing is not surveillance of the data subject by a third party but record-keeping of the data subject's own authorized actions.
- Operator transparency: `docs/legal/privacy-policy.md` will be updated at first non-Soleur tenant onboarding to disclose the orchestration-plane audit log (Art. 13 transparency). At v1 (Soleur-as-tenant-zero), the controller and data subject are the same legal entity (Jikigai's operator is Jikigai's founder), so the Art. 13 obligation is trivially satisfied.

### (v) Impact on the data subject

- **Negligible.** The data subject (founder-operator) is the one triggering the recorded action; the record is operationally beneficial to the data subject (their own audit trail). No automated decision-making in the sense of Art. 22 — the record is purely evidentiary and not used to make decisions about the data subject.

### (vi) Safeguards (TOMs — Art. 32 cross-reference)

Substantial safeguards reduce residual risk to negligible:

- **RLS with zero policies** on `public.tenant_deploy_audit` — service_role-only access; no broad-read surface from operator-facing roles.
- **WORM trigger** — BEFORE UPDATE/DELETE triggers raise `P0001` except during the Art. 17 anonymise flow (GUC-gated + role-gated).
- **SECURITY DEFINER writer RPC** with `SET search_path = public, pg_temp` (injection defense).
- **REVOKE-from-named-roles + GRANT-only-to-service_role** per `2026-05-06-supabase-default-privileges-defeat-revoke-from-public.md` (defeating Supabase's `ALTER DEFAULT PRIVILEGES` auto-grant).
- **CHECK constraints** on `target_repo` + `target_workflow` for both minimisation and log-injection defense (legal-compliance SHOULD #9).
- **`oidc_jti` is text + length-bounded** (RFC 7519 §4.1.7 compliance — Kieran P1-4).
- **`ON DELETE RESTRICT` on `founder_id` FK** forces the Art. 17 anonymise RPC to run BEFORE auth.users deletion — preserves audit-row integrity (spec-flow P0 #8).
- **12-month retention** via `tenant-deploy-audit-retention` pg_cron job (Art. 5(1)(e) storage limitation).
- **Art. 17 cascade** via `anonymise_tenant_deploy_audit(p_founder_id uuid)` — UPDATEs `founder_id = NULL` (preserves row count and audit-trail integrity; does not DELETE).
- **GitHub Environments + required reviewers** on tenant repos — organizational TOM that caps `workflow_dispatch` blast radius (load-bearing per external research and Kieran P2).
- **`::add-mask::` on Doppler/OIDC secrets** in workflows (confidentiality at log layer).
- **Short-lived tokens** via `hetznercloud/tps-action` + Doppler Service Account Identity OIDC (pseudonymisation/minimisation of long-lived secrets).

### Conclusion

The controller's legitimate interest (operational and statutory) **substantially outweighs** the data subject's minor interest in not having their own authorized dispatch actions recorded server-side, given:

- The data subject is the actor triggering the recorded event.
- The data is pseudonymous, low-sensitivity operational metadata.
- The reasonable expectation of a record is overwhelming for any CI/CD platform.
- The impact is operationally beneficial to the data subject.
- The safeguards stack is extensive and load-bearing.

**Balancing outcome: legitimate interest prevails.** Art. 6(1)(f) is the appropriate lawful basis for the orchestration-plane meta-audit log at v1.
