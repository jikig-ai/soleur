**Sub-clause (c) — workspace-scoped repo-connection state + cross-member processing (ADR-044, migrations 079/080/081, #4543 / #4558).** As of ADR-044, the GitHub repo-connection state that drives sub-purposes (i) and (ii) above — `repo_url`, `repo_provider`, `github_installation_id`, `repo_status`, `repo_last_synced_at` — is relocated from the per-user `users` table to the `workspaces` table (additive migration 079; idempotent solo-only backfill 080; application-layer TS read-cutover in `current-repo-url.ts` / `resolve-installation-id.ts` — workspaces-only reads, no migration; Art. 17 erasure cascade migration 081, which re-CREATEs `anonymise_organization_membership` to null `github_installation_id` + `repo_last_synced_at` and set `repo_status='not_connected'` on a departing owner's workspaces, retaining `repo_url` so a promoted replacement owner can re-authorize the same repo). This changes the **grain** of the processing, not the recipients: **(1) New cross-member data flow.** Previously repo state was per-user, so a natural person who JOINED another user's workspace as a Co-Member could not reach the workspace owner's connected repository. Post-ADR-044, repo-connection state is workspace-scoped, so a Co-Member of a workspace can sync and access THAT WORKSPACE's connected repository AND the knowledge-base derived from it. The consequence is that a Workspace Owner's repository content and the repo-derived knowledge-base become accessible to, and processed for, the Co-Members of that workspace — at the workspace-ownership grain, not per-user. No new third-party recipient is introduced: GitHub Inc. (Microsoft Corporation) remains the **source** (not a recipient) as in (d)/(e) above; the new recipients are the workspace's own Co-Members, already enumerated as a data-recipient category under PA-2 §(d) (workspace co-member). **(2) Lawful basis for the Co-Member's processing of the owner's repo/KB data.** The Co-Member's ACCESS to the Workspace Owner's connected repository and derived knowledge-base rests on the Workspace Owner's Art. 6(1)(a) consent, captured in the migration-058 `workspace_member_attestations` row (the inviter attestation, PA-2 substrate): the Owner attests at invite-time that the invited member will access this workspace's connected repository and knowledge-base. The existing Art. 6(1)(f) legitimate-interest basis recorded under "Lawful basis" above continues to govern the repo-DERIVED signals only. This consent-for-access + legitimate-interest-for-derived-signals split is the same composition disclosed in Privacy Policy §4.11, DPD §2.3(u), and GDPR Policy §5.3 (no doc carries a different primary basis for the cross-member access). Backfill scope is consent-preserving by construction: migration 080 copies repo-connection state to **solo workspaces only** (the `w.id = u.id` join plus a member-count = 1 canary guard); any workspace that already had a Co-Member at backfill time is SKIPPED, so no pre-existing connection becomes Co-Member-accessible without a fresh Art. 6(1)(a) invite attestation post-flag-flip. The Art. 6(1)(a) basis therefore never operates retroactively on a connection established before the Owner's consent of record. **(3) Credential gate (Art. 32 TOM).** The `github_installation_id` is a GitHub App token grant (a CREDENTIAL). It is removed from the `authenticated` role's column grant — implemented as a table-level `REVOKE SELECT ON public.workspaces FROM authenticated` followed by a `GRANT SELECT` on the explicit non-credential column list that omits `github_installation_id` (a bare column-level `REVOKE SELECT (github_installation_id)` is a no-op while Supabase's default table-level grant exists, so the table-revoke-then-column-regrant is the only effective form) — so the row-level `workspaces_select_for_members` policy cannot expose it; the value is readable ONLY via the membership-checked SECURITY DEFINER RPC `resolve_workspace_installation_id(p_workspace_id)`, which returns NULL to non-members. **(4) Single source of truth.** Repo reads come from `workspaces` ONLY — no per-user (`users`-table) read-time fallback — so a Co-Member's sync resolves the workspace owner's repo deterministically rather than via the deleted installation-id heuristics (#4546/#4557). See ADR-044 (`knowledge-base/engineering/architecture/decisions/ADR-044-workspace-repo-ownership.md`), brand-survival threshold `single-user incident`; cross-references PA-2 §(d) (workspace co-member recipient category) and DPD §2.3(u).
