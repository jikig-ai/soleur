# ADR-060: Outbound-email sending domain + single compliance chokepoint, audited via a dedicated `outbound_sends` WORM table

- **Status:** Accepted
- **Date:** 2026-06-15
- **Issue:** #5325 (pilot; generic subsystem deferred to #5331)
- **Decider:** `soleur:engineering:cto` agent (binding architecture call at /work; the substrate fork was routed to the CTO, not the operator)

## Decision

Agent-callable cold outbound email sends from a **dedicated `mail.soleur.ai` subdomain** (Resend, eu-west-1; SPF/DKIM/DMARC `p=quarantine`→`reject`) — a separate sender-reputation stream isolating cold-outreach reputation from the `soleur.ai` apex transactional sender (`notifications@soleur.ai`); same registrable domain, distinct subdomain — through a **single compliance chokepoint** (`apps/web-platform/server/email-triage/outbound.ts` `sendCompliantOutbound`) that is the only cold-send path and the only holder of the `mail.soleur.ai` FROM literal (enforced by a source sentinel). Every CLO condition is refuse-to-send (throws before Resend), in fixed order: domain-verified precondition → C1–C5 (C3 = six discrete Art. 14 element predicates; mechanical presence, human approver owns semantics) → RFC-5322 header-injection guard → recipient allow-list (reject internal/own-domain/role) → **body-hash approval match** → **in-transaction suppression recheck** → Resend → record audit. The send is recorded into a **dedicated `outbound_sends` WORM table** (migration 104): append-only pure-reject trigger, owner-RLS, `record_outbound_send` the only write path, `anonymise_outbound_sends` (service-role-only) for Art. 17 — mirroring the proven `action_sends` posture (mig 051) but **NOT** foreign-keyed to `messages`. The three agent tools (`email_send`/`email_reply`/`email_suppress`) are `gated`-tier: the human review gate is the trust boundary (the operator sees the exact recipient + body), and the chokepoint binds the send to the approved body by recomputing `sha256(body)` against the gate-approved hash. Recipients are pseudonymised at rest via `recipient_hash = HMAC-SHA-256(EMAIL_HASH_PEPPER, normalize(email))` (deterministic-by-design, so cross-campaign suppression matching works); plaintext recipients and bodies are never persisted.

## Rejected alternatives

- **Reuse `action_sends` (migration 051) for the send-audit + approval binding** (the plan's deepen-P0-1, and the reason the plan forbade a new table). **Rejected:** `action_sends.message_id` is a `NOT NULL` FK to `public.messages` with `UNIQUE(message_id)` — built for the founder-clicks-Send-on-a-draft-*message* path. Agent tool handlers have **no `messages.id`** at tool-execution time (the tool runs inside the SDK iterator; the assistant message is persisted only at the `result` event with a fresh UUID; the `buildEmailTriageTools` closure carries `userId` only), and `scope_grants` creation is UI-only. Reusing it would require threading a synthesized message id + an autonomous grant path through the load-bearing agent runtime (`agent-runner.ts`/`cc-dispatcher.ts`) — a multi-user-blast-radius change to satisfy an audit shape. The plan's "do NOT add `outbound_sends`" rule rested on the (code-falsified) premise that this reuse was clean.
- **Gate-only approval with no durable WORM send-record** (the in-memory `gated` approval + a Sentry breadcrumb + the persisted assistant message). **Rejected:** for cold outbound to non-consenting third parties, GDPR Art. 5(2) accountability (and a recipient's Art. 15/17/21 request) requires a durable, tamper-evident, recipient-bound, body-bound send record. A PII-free breadcrumb cannot demonstrate recipient binding by construction, and a free-text conversation message is mutable and not a hash-bound proof.
- **A forgeable bare `approved_at` column** (the original pre-deepen design). **Rejected** as in the plan: self-writable by the same send RPC, body-unbound, replayable across recipients. The body-hash-bound, single-use approval supersedes it.
- **Per-row random salt for `recipient_hash`** (stronger unlinkability). **Rejected:** would foreclose cross-campaign suppression matching (the C5 opt-out guarantee), which is *weaker* protection for the data subject. The keyed HMAC is linkable by design and confined to the suppression/audit purpose; the pepper is segregated in Doppler.
