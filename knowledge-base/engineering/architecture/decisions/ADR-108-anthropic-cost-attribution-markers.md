- **Status:** adopting for the markers (Phases 1–4). The daily Admin cost-report half is **BLOCKED on an account-tier decision, not merely sequenced** — the Admin API is unavailable to individual accounts and this org is one, so `ANTHROPIC_ADMIN_KEY` is un-mintable until the operator chooses to convert to a team organization; it may never be provisioned. ## Context

Anthropic spend (~$430/mo on the shared `github-claude-code-key`) could not be attributed per-cron or per-model. The key is byte-identical across Doppler configs and powers the production web-platform Inngest cron fleet plus the agent-runner / leader-loop web sessions. No structured cost marker was logged anywhere self-servable. The four downstream optimizations (Opus→Sonnet tier audit, prompt caching, spawn-only-when-work-exists, per-surface key split) are unmeasurable without per-model / per-cron cost telemetry.

Better Stack ingests the app container's pino stdout via the Vector `app_container_journald` source, **filtered to pino WARN+ (level ≥ 40)** by `app_container_warn_filter`; an `info`-level line does not ship. `scripts/betterstack-query.sh --grep` already matches `raw LIKE '%…%'`.

## Decision

**Emit a `SOLEUR_CLAUDE_COST` structured marker at pino WARN, via a dedicated pino instance that does NOT install the `mirrorToSentry` logMethod hook, from all three Anthropic-spend choke points, plus a distinct `SOLEUR_CLAUDE_COST_DAILY` marker from a new daily Admin cost-report cron.**

1. **WARN level (not INFO).** Required so the existing `app_container_warn_filter` ships the marker without a `vector.toml` change and without relaxing the global INFO filter (which would blow the 3 GB/mo Better Stack quota).
2. **Dedicated Sentry-mirror-bypassing logger.** `logger.ts` auto-mirrors every WARN+ line to a Sentry breadcrumb on the active scope. A steady per-turn cost-marker stream would evict genuine diagnostics from the shared-scope ring buffer, so `claude-cost-marker.ts` uses its own `pino()` instance with no `hooks.logMethod`. This is the sanctioned observability-of-observability exemption to `cq-silent-fallback-must-mirror-to-sentry`: `emitClaudeCostMarker` / `emitClaudeCostDailyMarker` catch-and-swallow, because a marker-emit failure must never red a cron or fail a session, and mirroring the failure would re-enter the same broken path.
3. **Three choke points.** (a) `cost-writer.persistTurnCost` / `persistTurnCostAwaitable` — the single funnel for all three session paths (surface the already-accumulated cost). (b) `spawnClaudeEval` — the 15 spawn-based eval crons (NEW capture via `--output-format json`; see ADR-033 I9). (c) `postAnthropicMessage` — the HTTP-transport crons `spawnClaudeEval` misses (`cron-compound-promote` real spend, `credit-probe` canary; tokens-only, `cost_usd:null`, since `/v1/messages` returns no `total_cost_usd`).
4. **Positive `capture_status` on every substrate exit.** Row-absence is not a probe; the substrate ships `capture_status ∈ {ok, no-result-event, parse-error, timeout}` + `cost_usd:null` on a capture failure, so "capture broke" is a shipped, queryable event distinct from "genuinely $0" and "cron never ran".
5. **Daily Admin cost-report cron.** `cron-anthropic-cost-report` pulls `GET /v1/organizations/cost_report` (authoritative org-total $) + `/usage_report/messages` (`group_by[]=model`, per-model tokens) once a day and emits `SOLEUR_CLAUDE_COST_DAILY`. The per-model entry is a **field-allowlist (named picks)** — never a `...row` spread — so `api_key_id` / `workspace_id` never reach Better Stack. Reads `ANTHROPIC_ADMIN_KEY`; self-reports `{status:"key-missing"}` benignly (positive dark marker, no page) while unprovisioned. (This cron only GETs, but the credential is **not** scope-limited — see Rejected alternative (c).)
6. **`key-missing` mirrors to Sentry at `warning`, never `error` (#6297).** Item 5's "no page" is a *severity* requirement, not just a marker requirement, and the original implementation did not honour it: it called `reportSilentFallback`, which captures at `level:"error"`. Sentry derives issue **priority** from level, and a high-priority notification rule fires on the derived priority — so the benign dark state paged the operator daily from 2026-07-10 until the fix. The branch now calls `warnSilentFallback`. The mirror itself is **not** optional (`cq-silent-fallback-must-mirror-to-sentry`; item 2's exemption covers only the marker emitter), and the level must stay **≥ pino 40** so Vector's `app_container_warn_filter` keeps shipping *this diagnostic line*. The genuinely-fatal 401/403 arm stays at `error`.

   > **Precision, because an earlier draft of this bullet conflated two emissions:** the `SOLEUR_CLAUDE_COST_DAILY` marker ships from `claude-cost-marker.ts`'s own dedicated pino instance at a hard-coded `log.warn`, so its delivery does **not** depend on the level chosen in the cron branch. Dropping the cron branch to `info` would silence the *diagnostic line* (in both Better Stack and Sentry) while the marker row kept shipping unchanged — bad, but not "the surface goes dark entirely". Keep the two independent when reasoning about either.
7. **`days_since_first_dark` on the key-missing marker (#6297).** The key-missing row carries whole UTC days since the **first observed** dark fire, so a stalled mint is visible in the data rather than inferred from the absence of an `ok` row. It is a frozen-origin count, **not** the age of the current dark window: it does not reset across a mint-then-rotate cycle (see Consequences). Inert reporting data — nothing branches on it, so a stale reading can never page. Omitted entirely on `status:"ok"` rows. Complies with item 2's ‼️ BOUNDARY: an integer day count carries no user id, email, secret, or regulated data.

## Rejected alternatives

- **(a) Relax the global Vector INFO filter** — rejected: ships the request-log firehose, blows the 3 GB/mo quota.
- **(b) A targeted Vector transform matching `SOLEUR_CLAUDE_COST` at INFO** — rejected: still a `vector.toml` deploy for no gain over emitting at WARN.
- **(c) Inngest→GHA dispatch-hybrid for the Admin key** (per the 2026-06-02 credential-heavy-cron learning) — rejected: execution is two trivial HTTP GETs and the `credit-probe` precedent already calls Anthropic direct from the app container. The dispatch-hybrid learning's own scope-note warns against mis-citing it for this trivial-execution workload.

  > **⚠️ One leg of this rejection was falsified (#6297, verified 2026-07-20).** It originally read *"the admin key is **read-only**"* and leaned on that for blast radius. Console Admin keys carry **no selectable scopes** — every key grants full Admin-API access (member/invite/workspace mutation included). Read-only describes how *this cron* uses it, not what the credential permits. The rejection now rests only on the trivial-execution + precedent legs; **if the key is ever actually minted, re-evaluate (c) on the corrected blast radius** before storing it in the app container's environment.

## Consequences

**Easier:** the four optimizations become measurable from Better Stack alone (per-cron `GROUP BY source`, per-model `JSONExtractString(raw,'model')`, daily org total). No Anthropic Console needed.

**Trade-offs:** the markers ride the same WARN lane + 3 GB/mo quota that preserves cron-failure diagnostics — session markers are per-turn, so post-deploy quota re-check is a hard gate (AC15); gate behind sampling if the measured volume is non-trivial. `tfstate` carries the admin key in cleartext (Approach B) — the state backend must stay encrypted + access-restricted; treat exposure as a rotation trigger (plan R-G).

**`days_since_first_dark` is measured from a frozen date, so it does NOT reset across a mint-then-rotate cycle.** If the key is minted and later unset — and exposure is a rotation trigger per the paragraph above, so this is an anticipated state, not a hypothetical — the counter will read the full elapsed span on day one of a fresh, entirely benign gap. This is why nothing branches on the value. A day-31 severity escalation keyed on it was designed and **cut** for exactly this reason: it would have paged at `error` on the first day of a benign rotation window, re-arming the #6297 defect inside the branch that fixes it. Escalation for a stalled mint belongs in the tracker/backlog channel, not in application severity.

**Item 5's premise depends on an account tier this ADR did not record, and the assumption was wrong (verified 2026-07-20).** The Admin API — and therefore `/v1/organizations/cost_report` — is **unavailable to individual accounts**. The operator's org is an individual org, for which `platform.claude.com/settings/admin-keys` returns **"Page not found"** and no "Admin keys" item appears in Console navigation. So `ANTHROPIC_ADMIN_KEY` is not merely un-minted; it is **un-mintable** until the org is converted to a team organization (Console → Settings → Organization → *Convert to team*). That conversion is an account-structure change with seat/billing implications and is an operator decision, not an engineering one. Until then the cron stays permanently on its (now non-paging) `key-missing` branch, and `#6297`'s follow-through probe correctly reports TRANSIENT rather than closing. Recorded here because the daily-cost-report half of this ADR is **blocked on an account-tier decision**, which no amount of IaC or key handling resolves.

**Non-goals:** this does NOT change any model tier, cadence, or key assignment, and does NOT add spend-vs-budget alerting (the other, still-deferred half of `Ref #5674`).
